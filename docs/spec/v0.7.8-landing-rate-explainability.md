## 1. Hintergrund & Problem

Wiederkehrendes Pilot-Feedback (zuletzt GSG-218-Fall):
> *"Mein Volanta zeigte ~232 fpm aber AeroACARS sagt -357. Wer hat Recht?"*

Beide Werte sind physikalisch korrekt — sie messen unterschiedliche Sachen:
- **Volanta / DLHv / Cockpit-VSI**: Mittel ueber 1.0-1.5 s, Spikes ausgefiltert (= "Pilot-Feeling").
- **AeroACARS-Score**: Sinkrate exakt am `on_ground`-Edge (= Engineering-Standard nach FAR 25.473, misst Belastung der Fahrwerksstruktur).

Der Client zeigt **bisher nur den Edge-Wert** in der Touchdown-Card. Die anderen 4 Mittelwerte + die Position-Trace + die Flare-Forensik liegen im `LandingRecord` aber werden im UI nicht erklaert. Resultat: Pilot interpretiert den Score-Wert isoliert ohne Kontext und versteht ihn nicht.

Auf dem VPS-Web (Admin-Tool, v0.7.7) wurde die Aufklaerungs-UI bereits umgesetzt — siehe `webapp/src/components/LandingAnalysis.tsx`. Diese Spec uebertraegt die gleiche Aufklaerung auf den **Pilot-Client** in AeroACARS-Design-Sprache und mit i18n.

## 2. Scope

### In Scope (v0.7.8)
- **F1**: Sinkraten-Vergleichs-Sektion mit 5 Werten (1.5 s / 1.0 s / 0.5 s / 0.25 s / Edge) im Pilot-Landing-Detail-View
- **F2**: Score-Basis-Hervorhebung — welcher Wert wird gescort, mit Engineering-Standard-Begruendung

### 3.2 Bucket-Aufschluesselung — komplette Client-Berechnung

Die Phase-Aufschluesselung wird **lokal im Client** aus den 4 kumulativen Mittelwerten + dem Edge-Wert berechnet (genau wie auf VPS-Seite in `_ApproachChart.tsx` / `VsBucketBreakdown`):

```
mean[-1500..-1000 ms] = (1500 * mean1500 - 1000 * mean1000) / 500
mean[-1000..-500 ms]  = (1000 * mean1000 - 500 * mean500)   / 500
mean[-500..-250 ms]   = (500 * mean500 - 250 * mean250)     / 250
mean[-250..0 ms]      = vs_at_edge_fpm  (fallback: vs_smoothed_250ms_fpm)
```

**Trend-Diagnose (v1.1-Korrektur: ueber Betrag, nicht Vorzeichen):**

Sinkraten sind negativ. "Sinkrate nimmt zu" = `|VS|` waechst = `VS` wird negativer. Die Vergleichs-Bedingung deshalb explizit ueber `Math.abs`:

```ts
const deltas = [
  Math.abs(bucket1000) - Math.abs(bucket1500),  // Phase 2 vs Phase 1
  Math.abs(bucket500)  - Math.abs(bucket1000),  // Phase 3 vs Phase 2
  Math.abs(bucketEdge) - Math.abs(bucket500),   // Phase 4 vs Phase 3
];
const isMonotonAccelerating = deltas.every(d => d > 20);
```

Wenn `isMonotonAccelerating === true`: Auto-Tipp **"Sinkrate hat zum Touchdown hin staerker zugenommen — Flare nicht gehalten / durchgesackt."** (Wortlaut explizit, ohne mathematisches "monoton steigend" das bei negativen Werten missverstaendlich ist.)

### 3.3 Position-Trace-Snippet — aus `approach_samples`

Filter: `is_scored_gate == true OR is_flare == true` mit `t_ms >= -3500 ms` → ca. 4-5 Samples. Zeigen als Tabelle:

```
t = -3.1 s   vs = -110 fpm   AGL 18 ft
t = -2.1 s   vs = -144 fpm   AGL 12 ft
t = -1.1 s   vs = -171 fpm   AGL  8 ft
t = -0.6 s   vs = -249 fpm   AGL  3 ft
t = -0.1 s   vs = -348 fpm   AGL  1 ft
```

Falls `approach_samples.len() < 3` (sehr alte PIREPs ohne Buffer-Dump): Sektion still ausblenden, kein Error.

## 5. Coaching-Tipp-Generator (Auto-Text)

Im **Aufklaerungs-Block** generieren wir einen kontextspezifischen Tipp aus den Daten:

### 5.1 Trigger-Regeln

| Trigger | Tipp (DE) |
|---|---|
| `bucket trend monoton steigend (3x +20 fpm)` | "Flare zu frueh ausgelaufen — Pitch-Attitude bis zum Aufsetzen halten, Throttle erst NACH Touchdown raus." |
| `peak_g_post_500ms >= 1.70` | "1.82 G beim Aufprall — Pull-up frueher beginnen (30 ft AGL), nicht erst im 10-ft-Bereich." |
| `flare_reduction_fpm < 50` | "Kein Flare detektiert — bei 30 ft Nase 2-3° anheben und halten." |
| `vs_at_edge - vs_smoothed_1500ms > 100` | "Deutlicher Drop im Final — Sinkrate war im Mittel weicher als am Aufsetz-Moment. Schau in die Bucket-Aufschluesselung wo das Sacken begann." |
| **Kein Trigger** | "Sauberer Touchdown — Sinkrate gleichmaessig, kein Drop im Final." |

Pro Touchdown wird **EIN** Tipp gezeigt (Prioritaet von oben nach unten — erster Match gewinnt). Mehrere Tipps lassen die Sektion zu lang werden.
