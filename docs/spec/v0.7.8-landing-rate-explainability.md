## 1. Hintergrund & Problem

Wiederkehrendes Pilot-Feedback (zuletzt GSG-218-Fall):
> *"Mein Volanta zeigte ~232 fpm aber AeroACARS sagt -357. Wer hat Recht?"*

Beide Werte sind physikalisch korrekt — sie messen unterschiedliche Sachen:
- **Volanta / DLHv / Cockpit-VSI**: Mittel ueber 1.0-1.5 s, Spikes ausgefiltert (= "Pilot-Feeling").
- **AeroACARS-Score**: scort den Cascade-Wert **`landing_peak_vs_fpm ?? landing_rate_fpm`** — also den Output des Touchdown-v2-Detection-Cascade (z.B. `vs_at_impact_frame` / `sampler_gear_force` / `agl_estimate_xp` je nach Sim-Adapter, sichtbar in `landing_source`). Das ist die Zahl die die Belastung am Fahrwerk beschreibt (FAR 25.473-naehe).

Der Client zeigt aktuell **nur den finalen Score-V/S-Wert** in der Touchdown-Card. Die anderen Mess-Sichten (4 Smoothed-Mittel, `vs_at_edge_fpm`, Pre-Flare-Peak), die Position-Trace und die Flare-Forensik liegen im `LandingRecord` aber werden im UI nicht erklaert. Resultat: Pilot interpretiert den Score-Wert isoliert ohne Kontext und versteht ihn nicht.

Auf dem VPS-Web (Admin-Tool, v0.7.7) wurde die Aufklaerungs-UI bereits umgesetzt — siehe `webapp/src/components/LandingAnalysis.tsx`. Diese Spec uebertraegt die gleiche Aufklaerung auf den **Pilot-Client** in AeroACARS-Design-Sprache und mit i18n.

function hasForensics(record: LandingRecord): boolean {

      || record.vs_smoothed_250ms_fpm != null

Weitere Bedingungen INNERHALB der Sektion (wenn `hasForensics === true`):
- **Bucket-Aufschluesselung (Block [3]):** rendert nur wenn ALLE 4 `vs_smoothed_*` UND `vs_at_edge_fpm` gesetzt sind (die Bucket-Math braucht alle 5 Werte). Sonst Bucket-Sub-Sektion ausblenden, kein Error.
- **Position-Trace (Block [6.1] in `<details>`):** rendert wenn `touchdown_profile.length >= 3` UND mindestens ein Sample in `t_ms ∈ [-3500, 0]`. Sonst nur Aufprall-Belastung im `<details>`.

Alle relevanten Felder sind `Option<...>` mit `#[serde(default)]` in der Rust-Storage — alte `landing_history.json`-Eintraege parsen weiter ohne Migration.

### 3.2 Bucket-Aufschluesselung — komplette Client-Berechnung

Die Phase-Aufschluesselung wird **lokal im Client** aus den 4 kumulativen Mittelwerten + dem Edge-Wert berechnet (genau wie auf VPS-Seite in `_ApproachChart.tsx` / `VsBucketBreakdown`):

```
mean[-1500..-1000 ms] = (1500 * mean1500 - 1000 * mean1000) / 500
mean[-1000..-500 ms]  = (1000 * mean1000 - 500 * mean500)   / 500
mean[-500..-250 ms]   = (500 * mean500 - 250 * mean250)     / 250
mean[-250..0 ms]      = vs_at_edge_fpm  (fallback: vs_smoothed_250ms_fpm)
```

**Trend-Diagnose (v1.1-Korrektur: ueber Betrag, nicht Vorzeichen):**

Sinkraten sind negativ. "Sinkrate nimmt zu" = `|VS|` waechst = `VS` wird negativer. Die Vergleichs-Bedingung deshalb explizit ueber `Math.abs`:

```ts
const deltas = [
  Math.abs(bucket1000) - Math.abs(bucket1500),  // Phase 2 vs Phase 1
  Math.abs(bucket500)  - Math.abs(bucket1000),  // Phase 3 vs Phase 2
  Math.abs(bucketEdge) - Math.abs(bucket500),   // Phase 4 vs Phase 3
];
const isMonotonAccelerating = deltas.every(d => d > 20);
```

Wenn `isMonotonAccelerating === true`: Auto-Tipp **"Sinkrate hat zum Touchdown hin staerker zugenommen — Flare nicht gehalten / durchgesackt."** (Wortlaut explizit, ohne mathematisches "monoton steigend" das bei negativen Werten missverstaendlich ist.)

### 3.3 Position-Trace-Snippet — aus `touchdown_profile`

v1.3-Korrektur: Position-Trace nimmt `touchdown_profile: Vec<LandingProfilePoint>` (`storage/lib.rs:266`), NICHT `approach_samples`. Begruendung siehe §3.1 (`ApproachSample` hat real nur `vs_fpm` + `bank_deg`).

Filter: `t_ms ∈ [-3500, 0]`. Da `touchdown_profile` ~150 Samples bei ~20-30 ms Cadence hat, ergeben 3.5 s ca. 100-170 Samples — wir reduzieren auf **5 repraesentative Zeitpunkte** im UI: ca. -3 s, -2 s, -1 s, -0.5 s, -0.1 s (nearest-neighbor aus dem gefilterten Slice).

Beispiel-Output:

```

t = -2.1 s   vs = -144 fpm   AGL 12 ft
t = -1.1 s   vs = -171 fpm   AGL  8 ft
t = -0.6 s   vs = -249 fpm   AGL  3 ft
t = -0.1 s   vs = -348 fpm   AGL  1 ft
```

Falls `touchdown_profile.length < 3` ODER kein Sample in `[-3500, 0]`: Position-Trace-Sub-Sektion still ausblenden, kein Error. Die `<details>`-Wrapper bleibt nur sichtbar wenn Position-Trace ODER Aufprall-Belastung Inhalt hat.
