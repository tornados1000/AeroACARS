# v0.12.3 — Landing-G: FOQA-konforme Messung (EMA-max statt Roh-Peak)

**Status:** SPEC ACCEPTED (R5) — von Thomas Kant freigegeben
**Autor:** v0.1x-Session-Owner
**QS-Reviewer:** Thomas Kant (VA-Owner)
**Trigger:** Flug TAP533 (EDDB → LPPT). Die G-Force-Forensik scort den **rohen
50-Hz-Einzelframe-Peak** (1.95 g). Volanta zeigt für dieselbe Landung 1.14 g.
Per X-Plane-DataRefTool verifiziert: **beide** X-Plane-G-DataRefs
(`sim/flightmodel2/misc/gforce_normal` und `sim/flightmodel/forces/g_nrml`)
liefern im Touchdown ~1.9 g — es ist **kein DataRef-Fehler**. Auch die G-Spur

Mess-**Methode**: ein roher Einzelframe-Peak aus 50-Hz-Abtastung ist schärfer,
als ihn ein echter Flugschreiber je aufzeichnen würde.

---

## Kontext: Was AeroACARS schon hat (Scoping)

AeroACARS ist **bereits** ein multi-parametrischer, gewichteter Landing-Scorer:

- Sub-Scores: `sub_g_force`, `sub_landing_rate` (Sinkrate), `sub_bounces`,
  `sub_stability` (Bank/Pitch/Approach-Stabilität), `sub_rollout`, `sub_fuel`,
  `sub_loadsheet`.
- Touchdown-Forensik v2: Edge-Detection (`contact_at` / `impact_at`),
  50-Hz-Sample-Fenster, Bounce-Count, Flare-Erkennung.
- Die G-Force-Forensics-Card zeigt schon EDGE / 250-ms-Mittel / 500-ms-Peak /
  1-s-Peak / Median-500 ms / P95-500 ms.

Eine komplette „FOQA-Scoring-Engine" neu zu bauen hieße, das alles neu zu
erfinden und mit dem bestehenden Score zu kollidieren. **Diese Spec macht das
nicht.** Sie fixt genau die eine Lücke, die TAP533 aufdeckt: der gescorte
G-Wert ist der rohe Einzelframe-Peak — er muss ein **EMA-geglätteter
Fenster-Peak** werden, so wie FOQA/FDM messen.

Der Fix ist **sim-agnostisch**: er wirkt auf die Touchdown-Forensik, nicht auf
einen X-Plane-Pfad. Für MSFS (SimConnect `G FORCE`, intern bereits leicht
geglättet) ist der Effekt klein; für X-Plane größer. Eine einheitliche Methode
über beide Sims ist gewollt.

## Problem

Der gescorte G-Wert ist `peak_g` = `max()` über die rohen 50-Hz-Samples im
Touchdown-Fenster. Bei TAP533: 1.95 g.

FOQA/FDM-Praxis (FAA: freiwilliges Programm, Best-Practice-Guidance, **keine**
feste Grenzwerttabelle; Airbus FDM: Vergleich gegen vordefinierte Hüllkurven):
Die Normalbeschleunigung wird vor dem Sampling **anti-aliasing-gefiltert** (ein
leichter Tiefpass) und mit ~8 Hz aufgezeichnet; bewertet wird der **Peak des
gefilterten Signals**. Kein Langzeit-Mittel (das würde eine harte Landung
weichrechnen), aber auch kein roher Einzelframe-Peak.

AeroACARS' roher 50-Hz-Peak ist *schärfer* als jede FDR-Aufzeichnung. Das
überzeichnet — besonders bei Study-/GA-Add-ons mit harten Fahrwerks-Modellen,
die echte Einzelframe-Spikes erzeugen.

## LE1 — Gescorter G-Wert = EMA-geglätteter Fenster-Peak

Der Wert, der in `sub_g_force` fließt **und** in der Card als „Scored G-peak"
prominent steht, ändert sich:

- **Bisher:** `peak_g` = roher `max()` über die 50-Hz-Samples.
- **Neu:** `scored_g` = `max()` über das **EMA-geglättete** G-Signal im
  Touchdown-Fenster.

Begründung = FOQA/FDM-Methode: leichter Tiefpass (≙ Anti-Aliasing-Filter),
dann Peak. Das killt einzelne Sim-Physik-Überschwinger, lässt eine echte,
*anhaltende* Spitze aber stehen — eine harte Landung wird **nicht**
weichgerechnet.

## LE2 — Framerate-unabhängiger EMA

Ein fixes per-Frame-`alpha` (z. B. `0.7·prev + 0.3·g`) ist **framerate-
abhängig**: bei 50 fps ≈ 62 ms Zeitkonstante, bei 30 fps ≈ 92 ms — dieselbe
Landung würde auf unterschiedlichen PCs anders bewertet.

Verbindlich: `alpha` aus dem **gemessenen `dt`** und einer festen
Ziel-Zeitkonstante `tau` ableiten:

```
alpha     = 1 - exp(-dt / tau)
smoothed  = smoothed + alpha · (g - smoothed)
```

Jedes Forensik-Sample hat einen Zeitstempel → `dt` ist vorhanden. Der EMA wird
über das **gesamte** Forensik-Fenster gerechnet, **inklusive der Samples vor
dem Edge** (das Fenster puffert ohnehin Pre-Edge-Daten) — so ist der EMA am
Edge bereits eingeschwungen (Pre-Buffer; deckt auch eine minimal verzögerte
`on_ground`-Meldung ab).

**`tau` = 100 ms** (QS-R0 von Thomas bestätigt — 80 ms noch nah am Roh-Spike,
120 ms dämpft schon recht stark). Verifiziert am echten TAP533-Fenster (EMA,
echtes `dt`):

| `tau` | `max(smoothed_g)` |
|---|---|
| 60 ms | 1.87 g |
| 80 ms | 1.83 g |
| **100 ms** | **1.78 g** |
| 120 ms | 1.74 g |
| 150 ms | 1.68 g |

(Roh-Peak zum Vergleich: 1.95 g · EDGE: 1.46 g)

**Exakte EMA-Regeln (QS-R0/P2 — verbindlich, damit R1 reproduzierbar dieselben
Zahlen liefert):**

1. **Sortierung:** Samples werden **nach Timestamp aufsteigend** verarbeitet.
2. **Initialisierung:** Die EMA startet mit dem **ersten finiten Sample**
   (`smoothed = erstes gültiges g`), **nicht** mit einer Konstante wie `1.0`.
3. **Kein Pre-Edge:** Gibt es keine Pre-Edge-Samples, startet die EMA beim
   ersten Sample am/nach dem Edge. Der `max()` (LE3) greift unverändert nur
   in `[edge, edge + 1.0 s]`.
4. **Nicht-finite Samples:** `NaN` / `Inf` werden **übersprungen** — sie
   aktualisieren die EMA nicht und zählen nicht in den `max()`. (Defensiv; der
   Forensik-Buffer sollte keine liefern, aber LE2 ist damit deterministisch.)
5. **`dt`:** Differenz der Timestamps zweier aufeinanderfolgender Samples. Bei
   nicht-positivem oder fehlendem `dt` → nominaler Fallback `dt = 20 ms`.

## LE3 — Messfenster

`scored_g = max(smoothed_g)` über `[edge_at, edge_at + 1.0 s]`. Der EMA selbst
läuft ab den Pre-Edge-Samples (LE2), das `max()` wird nur über das
Post-Edge-Fenster genommen. Das bestehende Touchdown-Fenster der Forensik v2
wird wiederverwendet — kein neues Fenster, keine neue Edge-Detection.

**QS-R0 bestätigt:** Fenster `[edge, edge + 1.0 s]`. Bounce-Impulse innerhalb
dieser Sekunde dürfen bewusst drinbleiben — sie sind Teil des realen
Touchdown-Lastverlaufs.

## LE5 — `sub_g_force`-Schwellen bleiben

Die Schwellen in `sub_g_force` (1.20 / 1.40 / 1.70 / 2.10 g) bleiben
**unverändert**. Sie bekommen künftig den EMA-Wert statt des Roh-Peaks.

Netto-Effekt: das G-Scoring wird minimal **fairer** — die 50-Hz-Überzeichnung,
die Grenzfälle bisher in ein härteres Band schob, fällt weg. Für TAP533 ändert
sich das Band nicht (1.78 g und 1.95 g liegen beide im Band 1.70–2.10 →
„firm", 30 Punkte); bei Landungen mit einem echten Einzelframe-Spike ist der
Effekt größer und gewollt.

**QS-R0 bestätigt (Thomas):** keine Neukalibrierung der Schwellen für v0.12.3.
Die Änderung ist eine beabsichtigte kleine Korrektur einer zu scharfen Messung,
kein Score-Umbau. Erst echte Vergleichsdaten sammeln, dann ggf. kalibrieren.

## LE6 — Sim-agnostisch: gilt einheitlich für MSFS und X-Plane

Die Touchdown-Forensik ist **ein** Code-Pfad. Das 50-Hz-Fenster speichert
`g_force`, unabhängig davon, ob der Wert von X-Planes RREF
(`sim/flightmodel2/misc/gforce_normal`) oder von MSFS' SimConnect (`G FORCE`)
stammt. Der EMA-max-Umbau (LE1–LE3) sitzt **in der Forensik**, nicht im
X-Plane- oder MSFS-Adapter — er greift damit für **beide** Sims automatisch.

**Verbindlich: kein sim-spezifisches G-Handling.** Dieselbe physische Landung
muss in MSFS und X-Plane gleich bewertet werden; ein eigener Pfad (oder ein
eigenes `tau`) pro Sim würde genau diese Cross-Sim-Konsistenz brechen.

Nuance: SimConnects `G FORCE` ist intern bereits glatter / hat weniger Jitter
als X-Planes RREF. Da der EMA framerate-unabhängig ist (`tau` aus echtem `dt`,
LE2), passt er sich der jeweiligen Datenrate an — der Effekt bei MSFS ist
klein, `scored_g` liegt dort nahe am Roh-Peak. Das ist gewollt: dieselbe
Methode, keine Übersmoothing-Gefahr.

**MSFS-Klarstellung — Fahrwerks-Lastspitze (R3).** `max_gear_force_n` kommt bei
X-Plane aus dem DataRef `gear_normal_force`; **MSFS' SimConnect liefert keine
saubere Fahrwerks-Normalkraft in Newton** → bei MSFS-Landungen ist das Feld
`None`. Die Card-Zeile „Fahrwerks-Lastspitze" (Block [4], LE4) ist im Code
bereits **konditional** gerendert (`gearForceN != null && > 0`) — sie wird bei
MSFS schlicht **nicht angezeigt**, ohne leere Zeile / „N/A". Kein Extra-Aufwand,
die Card funktioniert ansonsten identisch. Gleiches gilt für die VPS-Webapp-Card
(LE9).

## LE7 — Feldsemantik: roher Peak bleibt roh, EMA-Wert ist ein neues Feld (QS-R0/P1)

**QS-Befund:** `landing_peak_g_force` (intern, `lib.rs`) bzw. `peak_g_load`
(Wire-Feld, `aeroacars-mqtt`) ist heute gleichzeitig Score-Input,
Activity-Text-Quelle **und** Wire-Feld. Würde man `peak_g_load` einfach auf den
EMA-Wert umwidmen, lesen alte Consumer (Monitor, Webapp, externe Tools) statt
des Roh-Peaks plötzlich den EMA-Wert — eine **stille semantische Änderung**.
Das widerspricht der R0-Aussage „backward-kompatibel".

**Verbindlich R1:**

- `landing_peak_g_force` (intern) / `peak_g_load` (Wire) **behalten ihre
  Semantik: roher 50-Hz-Peak.** Unverändert. Backward-kompatibel.
- **Neues Feld:** intern `landing_scored_g_force`, Wire `scored_g_load`. Enthält
  den EMA-max-Wert (LE1–LE3).
- **`sub_g_force`-Score, Card-Headline und Activity-Text** lesen ab v0.12.3
  `landing_scored_g_force` / `scored_g_load`.
- Der rohe Peak bleibt als **Forensik-Detail** in der Card und im Wire-Payload
  erhalten (LE4).
- Wire-Format **additiv**: `scored_g_load` kommt hinzu, `peak_g_load` bleibt.
  Payloads ohne `scored_g_load` (von älteren Clients) müssen weiter
  deserialisieren — Feld optional.
- **Persistiertes `LandingScored`-Event (QS-R3/P2):** das Event hat heute nur
  `peak_g_force`. Es bekommt **additiv** `scored_g_force` + `scored_g_method`;
  `peak_g_force` bleibt der **rohe** Wert (unverändert). So bleibt das
  persistierte Event mit Activity-Text und Card-Anzeige (beide EMA) konsistent,
  ohne ein bestehendes Feld umzuwidmen.
- **Event-Backward-Compat (QS-R4/P2):** `FlightLogEvent` ist `Serialize +
  Deserialize` — alte JSONL-Logs mit `landing_scored` müssen weiter laden. Die
  neuen Felder sind daher verbindlich **`scored_g_force: Option<f32>`** und
  **`scored_g_method: Option<String>`**, jeweils mit **`#[serde(default)]`**.
  Ein Alt-Event ohne die Felder deserialisiert zu `None`. Test:
  `old_landing_scored_event_deserializes`. Ein zweiter Test sichert, dass
  `peak_g_force` roh bleibt.

## LE8 — Ein zentraler Helper, alle Score-Einstiegspunkte (QS-R0/P1)

**QS-Befund:** mehrere Code-Stellen setzen oder lesen `landing_peak_g_force`:
Analyse-Übernahme (`lib.rs` ~9090), Score-Klassifizierung (`lib.rs` ~9105),
Fallback-/Touchdown-Pfade (`lib.rs` ~11220), spätes Peak-Tracking (`lib.rs`
~11633). LE1–LE3 beschreiben nur den Forensik-Pfad — die anderen wären nicht
abgedeckt.

**Verbindlich R1:**

- Die EMA-max-Logik (LE1–LE3) existiert **genau einmal** in einem zentralen
  Helper, der `scored_g` aus dem Touchdown-Fenster berechnet.
- `landing_scored_g_force` wird **einmal** befüllt (in der Forensik, wo das
  Fenster existiert).
- **Jeder** Score-Pfad, der Activity-Text und die Wire-Serialisierung lesen
  `landing_scored_g_force` — **kein** Pfad re-derived den Wert aus dem Roh-Peak.
- **Definierter Fallback:** ein Score-Pfad ohne Touchdown-Fenster (z. B. spätes
  Peak-Tracking ohne Forensik-Buffer) setzt `scored_g = roher G-Wert`, markiert
  mit `method = "raw_fallback"` statt `"ema_max"`. **Kein** Pfad darf
  `landing_scored_g_force` unbefüllt (`0` / `None`) an den Score geben.
- Der Helper liefert `{ scored_g, raw_peak, method }`; die `method`-Kennung
  kann die Card nutzen („EMA-geglättet" vs „Roh-Fallback").
- **Landing-Scoring-Crate + TS-Mirror (QS-R3/P1):** der `LandingScoringInput`
  des `landing-scoring`-Crates **und** das TS-`computeSubScores`
  (Client-Legacy-Pfad, `LandingPanel.tsx`) bekommen ein **optionales**
  `scored_g_load`-Feld. `sub_g_force` scort `scored_g_load ?? peak_g_load`.
  Ohne diesen Schritt bliebe der Crate-/Mirror-Score auf dem Roh-Peak, während
  der Rust-Hauptpfad bereits EMA nutzt — eine stille Inkonsistenz. `peak_g_load`
  bleibt im Input erhalten (Roh-Forensik + Fallback).

## LE9 — Anzeige-Flächen: In-App-Card UND VPS-Webapp (cross-repo) (QS-R1)

**QS-Hinweis (Thomas):** die G-Force-Forensik wird an **zwei** Stellen
gerendert:

  landing_peak_g_force`; QuickFlags + RunwayDiagram nutzen ihn. **Ausnahme:**
  die klassische Keyval-Zeile `landing.peak_g` darf den **rohen** Wert weiter
  zeigen — wird dafür aber als „Roh-Peak G" (i18n-Label, Forensik-Detail)
  gekennzeichnet, damit klar ist, dass das nicht der gescorte Wert ist.
- **Webapp — G-Force-Forensik-Card:** spiegelt die `GForceForensik`-5-Block-
  Struktur und bekommt **dieselben 3 Block-Änderungen wie LE4**: Block [1]
  Intro-Text neu, Block [2] Peak-Tiles-Hint „· Roh 50 Hz", Block [4] Score-Tile
  zeigt `scoreG`. Layout/Struktur unverändert. Fahrwerks-Lastspitze konditional
  (LE6) — bei MSFS-PIREPs ausgeblendet.
- **CSS (QS-R3/P3):** kein hartes `gforce-*`-Erfordernis. Die In-App-Card nutzt
  die geteilten `sinkrate-*`-Layout-Klassen (bewusst, Optik-Konsistenz mit der
  Sinkrate-Card); die Webapp ist ein eigenes Repo mit eigenem Styling und darf
  vorhandene Klassen oder Inline-Styles verwenden. Maßgeblich ist die
  **5-Block-Struktur**, nicht ein Klassenname.
- **Backward-Fallback:** PIREPs/Payloads von pre-0.12.3-Clients haben **kein**
  `scored_g_load`. Die Webapp fällt dann auf `peak_g_load` zurück
  (`display_g = scored_g_load ?? peak_g_load`) — kein Bruch für Alt-PIREPs in
  der Historie.

**Release-Kopplung.** v0.12.3 ist das **AeroACARS-Client-Release**. Die
`aeroacars-live`-Änderung (Recorder + Webapp) ist ein **eigener PR im
`aeroacars-live`-Repo** plus VPS-Deploy. Wegen des additiven Wire-Formats
(LE7) gibt es **keine harte Reihenfolge**: ein v0.12.3-Client an einer noch
nicht aktualisierten Webapp ist unkritisch — das neue Feld wird ignoriert, die
Webapp zeigt weiter `peak_g_load`. Für konsistente Zahlen (App = Web) sollten
Client-Release und VPS-Deploy aber **zeitnah** erfolgen.

## LE10 — Accident-/Crash-Detection bleibt auf Roh-Peak (QS-R3/P2)

**QS-Befund:** die Accident-Heuristik (`accident.rs`) nutzt `peak_g_load` als
strukturellen Marker für die Crash-/Over-G-Erkennung. R3 ließ offen, ob sie auf
`scored_g` mitwandert.

**Verbindliche Entscheidung R4: Accident-Detection bleibt bewusst auf dem
*rohen* `peak_g_load`.** Begründung: Crash-/Over-G-Erkennung soll den
**physischen Extremwert** sehen — einen echten Struktur-Überlast-Spike. Die
EMA-Glättung existiert, damit eine *Pilot-Bewertung* nicht durch die
50-Hz-Schärfe unfair wird (LE1). Bei der Detektion eines Crashs ist das
**Gegenteil** gewollt: der schärfste Wert ist der relevante. Accident-Detection
ist Extremwert-Detektion, **kein** faires Scoring — `scored_g` (geglättet)
könnte einen Grenzfall-Crash unter die Schwelle drücken.

Im Code wird das mit einem Kommentar abgesichert, damit ein späterer Refactor
es nicht versehentlich auf `scored_g` umstellt.

## Bewusst NICHT in v0.12.3 (Scope-Abgrenzung)

Aus dem FOQA-Designvorschlag bewusst **verschoben** — gute Ideen, aber separat:

- **Multi-Parameter-Score-Umbau** (Penalty-Tabellen, Severity-Engine): AeroACARS
  hat das in Form der Sub-Scores bereits. Ein Parallel-System würde kollidieren.
- **Side-Load** (`gforce_side`): aktuell nicht erfasst. Neuer DataRef + neuer
  Sub-Score = eigener fokussierter Folge-Task, falls Pilot-Logs unsaubere
  Crosswind-Touchdowns zeigen.
- **Plausibilisierung** `fnrml_total / (m_total · 9.80665)`: nette Cross-Check-
  Diagnose, nicht nötig — beide G-DataRefs sind bereits verifiziert konsistent.
