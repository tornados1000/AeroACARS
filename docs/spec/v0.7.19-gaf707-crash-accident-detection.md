## Leitentscheidungen

### 1. Accident ist kein "noch schlechterer Landing-Score"

Ein Accident darf nicht als normale Landung mit Score 0 erscheinen. Score 0 bleibt korrekt, aber die Kategorie muss anders heissen:

- `severe_landing`: harte/strukturell relevante Landung, aber noch als Landung interpretierbar.
- `accident`: Absturz, Kontrollverlust, off-runway impact oder Sim-Crash-Event.

UI-Text: **"Absturz erkannt"**, nicht nur **"Harte Landung"**.

Der numerische Score aus der `landing-scoring`-Crate wird durch Accident-Erkennung **nicht** neu berechnet. GAF 707 bekommt heute schon Score 0, weil `vs_at_edge_fpm ~= -2250` und `peak_g_load = 4.41` unter jeder Landing-Schwelle liegen. Accident ist orthogonal zur Score-Berechnung:

- Score bleibt numerisch wie bisher.
- Accident ueberschreibt das Primary Label / die fachliche Klassifikation.
- Accident verhindert `clean_ifr`-Filing.
- Die Score-Kacheln bleiben als Forensik sichtbar.

### 2. Simulator-neutrale Heuristik ist Pflicht, Sim-Events sind Zusatz

MSFS/SimConnect hat laut offizieller SDK-Doku System Events:

- `Crashed`: user aircraft crashed
- `CrashReset`: crash cut-scene completed

Referenz: `SimConnect_SubscribeToSystemEvent`  
https://docs.flightsimulator.com/html/Programming_Tools/SimConnect/API_Reference/Events_And_Data/SimConnect_SubscribeToSystemEvent.htm

Wenn MSFS `Crashed` liefert, ist das ein harter Accident-Trigger. Das ist aber nur ein Bonus-Signal fuer MSFS. Die eigentliche Accident-Erkennung muss **fuer alle Simulatoren** funktionieren:

- MSFS
- X-Plane
- spaetere Sims/Adapter

Darum ist die gemeinsame Telemetrie-Heuristik kein Fallback zweiter Klasse, sondern der zentrale Pfad. Sim-spezifische Events duerfen nur zusaetzliche Confidence liefern.

### 3. X-Plane ist first-class

X-Plane darf nicht schlechter behandelt werden als MSFS. Der X-Plane-Pfad nutzt dieselbe Accident-Heuristik auf Basis von:

- `vertical_speed_fpm`
- `g_force`
- `gear_normal_force_n` fuer den Touchdown-Edge
- `velocity_body_x_fps` / `velocity_body_z_fps` fuer Sideslip
- `stall_warning`
- `on_ground`
- Runway-/Airport-Resolution aus v0.7.18

Fuer X-Plane wird in dieser Iteration kein harter Crash-DataRef vorausgesetzt. Falls spaeter ein stabiler Crash-/Damage-DataRef verifiziert wird, kann er als Zusatzsignal nachgezogen werden (`crash_source = "xplane_crash_dataref"`). v0.7.19 darf davon aber nicht abhaengen.

### 4. Filing bleibt moeglich, aber nicht als Clean IFR

Ein erkannter Accident darf nicht still als sauberer IFR-ACARS-PIREP durchgehen. Der Pilot bekommt eine klare Auswahl:

1. **Als Unfall/Review einreichen**  
   PIREP wird mit Score 0, Accident-Marker und Notes eingereicht. Danach muss der Flug sauber beendet sein: Aircraft frei, Bid geloescht/Reservation weg, lokaler Active-Flight weg.
2. **Flug verwerfen & Cleanup**  
   Bestehender v0.7.18-Orphan/Cleanup-Pfad: PIREP canceln, Aircraft freigeben, Bid loeschen, lokale Queue/State bereinigen.
3. **Zurueck**  
   Nichts passiert, aktiver Flug bleibt erhalten.

Fuer IFR ist Accident immer mindestens `manual_review`, nie `clean_ifr`. Fuer VFR bleibt Filing moeglich, aber ebenfalls sichtbar als Accident markiert.

### 5. Bei unsicheren Faellen fragt der Client den Piloten

Nicht jeder rote Marker ist automatisch ein Absturz. Es gibt harte Landungen, Off-airport-Notlandungen, Szenario-Fluege und Simulator-Eigenheiten. Deshalb:

- `Confirmed` + Sim-Event (`MSFS Crashed`) -> Accident vorausgewaehlt, Pilot kann trotzdem "Nein, als harte Landung behandeln" waehlen, falls der Sim falsch getriggert hat.
- `Confirmed` nur durch Heuristik -> Dialog fragt: **"War das ein Absturz?"**
- `Suspected` -> nur Warnung + Frage, kein automatisches Accident-Filing.

Der Pilot darf das Ergebnis korrigieren, aber die Korrektur wird in Notes/Audit sichtbar gemacht:

```text
AeroACARS suspected accident, pilot overrode to hard landing.
Reasons: V/S -1050 fpm, no runway match, sideslip 38 deg.
```

### 6. CrashReset loescht den aktiven Accident-Latch nicht

`CrashReset` ist ein Simulator-UI-/Recovery-Ereignis, kein Beweis dass der Accident fachlich erledigt ist. Wenn MSFS nach einem Crash-Dialog `CrashReset` sendet, darf der aktive Flug den Accident-Status nicht verlieren.

Regel:

- Adapter darf seinen rohen `crashed`-Flag nach `CrashReset` fuer neue Snapshots zuruecksetzen.
- `FlightStats.accident_detected` bleibt aber bis **Flight-End**, **Cleanup/Cancel** oder **neuem Flight-Start** gelatcht.
- Der Accident-Dialog muss auch dann noch erscheinen, wenn der Pilot zuerst im Simulator "Reset/Acknowledge" klickt und erst danach in AeroACARS filen/canceln will.

### 7. Multi-Touchdown-Aggregation: Worst Case gewinnt

Ein PIREP kann mehrere Touchdowns haben: Bounce, Touch-and-Go, Training, Zwischenlandung, finaler Touchdown. Fuer den PIREP-Status gilt:

```text
PIREP accident status = Max-Severity ueber alle Touchdowns dieses PIREPs
```

Severity-Reihenfolge:

```text
Confirmed Accident > Suspected Accident > Severe Landing > Hard Landing > normal/no accident
```

Wenn mehrere Touchdowns dieselbe Accident-Severity haben, gewinnt der mit der hoechsten absoluten `vs_at_edge_fpm`. `accident_kind` und `accident_reasons` im PIREP kommen vom Worst-Case-Touchdown. Die Webapp-History muss dieselbe Aggregation verwenden, sonst laufen Client und VPS bei Touch-and-Go-Fluegen auseinander.

## Detection

### Zusatz-Trigger: MSFS `Crashed`

Im MSFS-Adapter:

```rust
const CRASHED_EVENT_ID: u32 = 302;
const CRASH_RESET_EVENT_ID: u32 = 303;

SimConnect_SubscribeToSystemEvent(handle, CRASHED_EVENT_ID, "Crashed");
SimConnect_SubscribeToSystemEvent(handle, CRASH_RESET_EVENT_ID, "CrashReset");
```

Dispatch:

```rust
if event_id == CRASHED_EVENT_ID {
    shared.crashed.store(true, Ordering::Relaxed);
}
if event_id == CRASH_RESET_EVENT_ID {
    shared.crash_reset_seen.store(true, Ordering::Relaxed);
}
```

Snapshot-Mapping:

```rust
snap.crashed = shared.crashed.load(Ordering::Relaxed);
snap.crash_source = if snap.crashed {
    Some("msfs_crashed_event".into())
} else {
    None
};
```

Acceptance:

- Wenn `snap.crashed == true` waehrend eines aktiven Fluges: `accident_detected=true`.
- `CrashReset` darf die Anzeige nicht sofort verlieren. Der aktive Flug muss den Accident-Zustand behalten.

### Klassifikator-Trigger-Punkte

Es gibt zwei Pfade, die `accident_detected=true` setzen koennen. Sie laufen zu unterschiedlichen Zeitpunkten und duerfen nicht vermischt werden:

| Pfad | Trigger | Latenz | Datenbasis |
|---|---|---|---|
| Sim-Event-Pfad | `snap.crashed` flippt `false -> true` waehrend ein Flug aktiv ist | sofort, ggf. vor Bodenkontakt | Simulator-Event |
| Heuristik-Pfad | Touchdown-Edge im 50-Hz-Sampler | beim Aufsetzen / LandingRecord-Bau | `vs_at_edge_fpm`, `peak_g_load`, Sideslip, Runway usw. |

#### Sim-Event-Pfad

Sobald `snap.crashed` von `false` auf `true` wechselt und `active_flight` existiert:

```rust
stats.accident_detected = true;
stats.accident_at = Some(now);
stats.accident_kind = Some("sim_crash".into());
stats.accident_confidence = Some("high".into());
stats.accident_reasons.push("simulator_crashed_event".into());
```

Dieser Pfad wartet nicht auf einen Touchdown-Edge. Das ist wichtig, weil ein Simulator-Crash-Event bereits mid-air oder im Crash-Dialog kommen kann.

#### Heuristik-Pfad

`classify_accident(stats, analysis)` wird **nicht** in jedem 50-Hz-Loop-Tick ausgefuehrt. Der Aufruf passiert genau dann, wenn der Touchdown-Edge verarbeitet wird:

- im selben Tick wie `build_landing_record`, oder
- direkt nachdem die Landing-Analysis (`vs_at_edge_fpm`, `peak_g_load`, `runway_match_*`) materialisiert wurde.

Das Ergebnis wird in `FlightStats` gespeichert und danach fuer Client-Tab, Flight-End-Dialog, MQTT, LandingRecord und PIREP wiederverwendet.

#### Ueberlagerung

Beide Pfade duerfen sich ueberlagern:

- Wenn Sim-Event zuerst feuert und Heuristik spaeter `Confirmed(Impact)` liefert:
  - `accident_kind` bleibt `sim_crash` (Sim-Event ist autoritativ).
  - `accident_reasons` werden um Heuristik-Gruende ergaenzt.
  - `accident_at` bleibt der erste Crash-Zeitpunkt.
- Wenn Heuristik zuerst feuert und spaeter ein Sim-Event kommt:
  - `accident_kind` darf zu `sim_crash` hochgestuft werden.
  - `accident_at` bleibt der erste Accident-Zeitpunkt.

Rationale: Kein Flicker, keine Approach-Stall-False-Positives, kein Tick-fuer-Tick-Neuklassifizieren.

### Gemeinsamer Trigger: Impact-Heuristik

Dieser Pfad laeuft fuer **MSFS und X-Plane identisch**. Er darf keine MSFS-only-Felder voraussetzen.

Neue Pure Function:

```rust
fn classify_accident(stats: &FlightStats, analysis: Option<&serde_json::Value>) -> AccidentDecision
```

Empfohlenes Ergebnis:

```rust
enum AccidentDecision {
    None,
    Suspected { kind: AccidentKind, reasons: Vec<String> },
    Confirmed { kind: AccidentKind, reasons: Vec<String> },
}

enum AccidentKind {
    SimCrash,
    Impact,
    OffAirportImpact,
}
```

#### Confirmed Accident

Ein Treffer reicht:

1. **Simulator Crash Event**
   - `snap.crashed == true`

2. **GAF-707-Pattern: extremer Impact**
   - `abs(vs_at_edge_fpm) >= 1500`
   - UND `peak_g_load >= 3.0`

3. **Off-airport impact**
   - `abs(vs_at_edge_fpm) >= 1000`
   - UND `runway_match_icao == None`
   - UND mindestens 2 rote Zusatzmarker:
     - `peak_g_load >= 2.5`
     - `abs(sideslip_deg) >= 45`
     - `landing_wing_strike_severity_pct >= 100`
     - `approach_stall_warning_count >= 3`
     - `rollout_distance_m <= 300`

#### Suspected Accident

Kein harter Confirm, aber Review-Banner:

- `abs(vs_at_edge_fpm) >= 1000`
- UND mindestens 2 Zusatzmarker:
  - `peak_g_load >= 2.1`
  - `abs(sideslip_deg) >= 30`
  - `landing_wing_strike_severity_pct >= 75`
  - `approach_stall_warning_count >= 1`
  - `runway_match_icao == None`

Suspected Accident wird in UI/Webapp als Warnung gezeigt, blockiert aber nicht automatisch. Confirmed Accident schaltet den Filing-Pfad auf Accident/Review, aber der Pilot bestaetigt oder korrigiert die Einordnung im Dialog.

### Anti-False-Positive-Regeln

Diese Faelle duerfen **nicht** als Accident gelten:

| Fall | Beispiel | Erwartung |
|---|---|---|
| Normal hard landing on runway | `-700 fpm`, `2.5 G`, runway match vorhanden, stable attitude | `severe_landing`, nicht Accident |
| Sim strut-compression | `-116 fpm`, `2.3 G` | B-009 bleibt: acceptable/firm, nicht Accident |
| Off-airport emergency landing weich | `-300 fpm`, `1.4 G`, no runway match | Divert/off-airport, nicht Accident |
| B-013 Reset sample | lat/lon 0/0 oder Altitude jump | Sample wird verworfen, kein Accident daraus |
| MSFS pause/replay/reset | grosse Luecke `dt > 2s` | B-013 dt-Guard greift, kein Jump-Accident |

Regel:

- Sobald der Touchdown/Impact als `Confirmed Accident` erkannt wurde, zeigt der Tab Landung den Accident-Banner.
- Der Pilot muss nicht erst auf "Flug beenden" klicken, um die Einordnung zu sehen.
- Der Flight-End-Dialog nimmt denselben Zustand auf und fragt dann "War das ein Absturz?".

   - PIREP kann weiter als Severe/Hard Landing eingereicht werden.

4. Wenn "Flug verwerfen & Cleanup":
   - v0.7.18 Cleanup-Pfad verwenden.
5. Wenn "Zurueck":
   - kein State-Change.

```text
Moeglicher Absturz / Review empfohlen
```

GAF 707 ist `Confirmed`, nicht nur `Suspected`.

### Webapp History / VPS-Nachklassifikation

Die Webapp darf Accident-Status nicht nur anzeigen, wenn der neue Client `accident=true` sendet. Sie muss fuer alte und neue Records eine gemeinsame Anzeige-Entscheidung treffen:

1. Wenn Payload `accident=true` hat:
   - Client-Wert uebernehmen.
   - `accident_kind`, `accident_confidence`, `accident_reasons` anzeigen.

2. Wenn Payload `accident_classifier_version` hat:
   - Client-Entscheidung uebernehmen.
   - Fehlendes `accident` oder `accident=false` bedeutet: Client hat klassifiziert und **kein** Accident erkannt.
   - Webapp darf nicht nachklassifizieren und den Client ueberstimmen.

3. Wenn Payload weder `accident` noch `accident_classifier_version` hat:
   - Webapp/VPS rechnet mit derselben Heuristik aus den vorhandenen Touchdown-Payload-Feldern nach.
   - Ergebnis wird als **"nachklassifiziert"** markiert, nicht als originaler Client-Wert.

4. Wenn zu wenig Felder vorhanden sind:
   - Kein Accident-Banner.
   - Optional kleiner Hinweis im Debug/Details: "Accident-Klassifikation nicht moeglich, alte Payload unvollstaendig."

Damit koennen historische Fluege wie GAF 707 in der History korrekt als Accident erscheinen, ohne dass der Pilot mit einem neuen Client nochmal fliegen muss.

Anzeige-Beispiel:

```text
Accident erkannt
Impact / High confidence / Nachklassifiziert aus historischem Payload
```

History-Listen sollen nicht nur `score=0` zeigen, sondern einen eigenen Marker:

| Ansicht | Darstellung |
|---|---|
| Touchdown Detail | roter Accident-Banner mit Gruenden |
| Landing History Liste | Badge `Accident` statt nur Score `0` |
| Pilot-/VA-History | Filter/Marker fuer Accident, sofern die Ansicht Scores/Badges zeigt |

Wichtig: Nachklassifikation im Web ist **Anzeige- und Review-Logik**, kein nachtraegliches phpVMS-Mutieren. Alte PIREPs werden nicht automatisch geaendert, keine Aircraft/Bid-Aktionen fuer historische Records.

### Webapp Multi-Touchdown-Aggregation

History-/PIREP-Ansichten duerfen nicht nur den letzten oder ersten Touchdown nehmen. Fuer PIREP-Level-Badges gilt dieselbe Worst-Case-Regel wie im Client:

```text
Confirmed Accident > Suspected Accident > Severe Landing > Hard Landing > normal/no accident
```

Beispiel:

- Touchdown 1: Touch-and-Go, `-120 fpm`, kein Accident
- Touchdown 2: Crash, `-1800 fpm`, `3.2 G`
- History-Badge: `Accident`
- Detailgruende: vom zweiten Touchdown

Bei gleicher Severity gewinnt der Touchdown mit der hoechsten absoluten `vs_at_edge_fpm`.

## Filing-Regeln

### IFR / Bid

| Accident State | Clean Auto-File | Manual/Review File | Cancel/Cleanup |
|---|---:|---:|---:|
| none | erlaubt | erlaubt bei normalen Recovery-Faellen | erlaubt |
| suspected | nicht automatisch clean; Pilot muss bestaetigen | erlaubt mit Notes | erlaubt |
| confirmed | nicht clean | erlaubt als Accident/Review | erlaubt |

Confirmed Accident darf niemals als `clean_ifr` durch die neue Filing-Policy laufen.

### VFR / Free Flight

VFR bleibt locker, aber nicht unsichtbar:

- Accident darf eingereicht werden.
- Notes/Banner muessen sichtbar sein.
- Kein "Clean Landing"-Label.

## Accident Cleanup / phpVMS State

Ein Accident darf kein Flugzeug und keinen Bid auf phpVMS haengen lassen. Nach jeder finalen Pilot-Entscheidung muss klar sein, welcher Server-State entsteht.

### Zielzustand

| Pilot-Entscheidung | PIREP | Aircraft | Bid |
|---|---|---|---|
| Unfall/Review einreichen | FILED / pending review mit Accident-Notes | frei | geloescht / Reservation weg |
| Als harte Landung behandeln | FILED mit Override-Notes | frei | geloescht / Reservation weg |
| Flug verwerfen & Cleanup | CANCELLED | frei | geloescht / Reservation weg |
| Zurueck | unveraendert IN_PROGRESS | bleibt reserviert | bleibt reserviert |

Wichtig: "Aircraft frei" ist nicht optional. Der GAF-707-Pilot darf nach Accident/Cancel nicht wieder beim VA-Admin landen, nur weil phpVMS das Flugzeug noch als `in flight` sieht.

### Operationen

Der Client nutzt dieselben Bausteine wie v0.7.18 B-011:

1. **PIREP filed oder cancelled**
   - Filing-Pfad: `file_pirep` / bestehender `flight_end`-Pfad.
   - Cleanup-Pfad: `cancel_pirep(pirep_id)`.
   - phpVMS/Disposable Basic gibt das Aircraft beim Cancel bzw. Abschluss frei. Falls phpVMS nach FILED nicht automatisch freigibt, muss die Spec-Implementierung das als Blocker behandeln.

2. **Bid-Drop**
   - Nach FILED oder CANCELLED immer `delete_bid(bid_id, flight_id)` versuchen.
   - Reihenfolge wie v0.7.18:
     - wenn `bid_id` vorhanden: `DELETE /api/user/bids` mit `{ bid_id }`
     - bei `400 | 404 | 422` und `flight_id` vorhanden: Retry mit `{ flight_id }`
     - wenn nur `flight_id` vorhanden: direkt `{ flight_id }`
     - wenn beide fehlen: kein Bid-Drop moeglich, aber Activity-Log-Warnung schreiben.

3. **Lokales Aufraeumen**
   - `state.active_flight = None`
   - Persisted active flight loeschen
   - Position-Outbox leeren
   - `discard_queued_positions_for(pirep_id)`
   - `pirep_queue::remove(pirep_id)`

### Reihenfolge fuer "Unfall/Review einreichen"

```text
Pilot bestaetigt Accident
  -> file_pirep mit accident=true / score=0 / notes
  -> wenn FILED oder QUEUED:
       - bei QUEUED: KEIN cancel_pirep, kein Bid-Drop jetzt; Queue muss spaeter nach erfolgreichem File den Bid droppen
       - bei FILED: delete_bid(bid_id, flight_id)
       - wenn delete_bid transient fehlschlaegt: pending_bid_cleanup persistieren
       - local cleanup
  -> wenn HardFail:
       - active_flight bleibt
       - Dialog: "Einreichen fehlgeschlagen. Trotzdem verwerfen & Cleanup?"
```

Rationale: Bei transientem Fehler darf nicht gecancelt werden, sonst entsteht derselbe Datenverlust-Bug wie B-014. Wenn das Accident-PIREP in der Queue liegt, muss ein Queue-Success-Hook den Bid-Drop nachholen.

### Pending Bid Cleanup Queue

Wenn `file_pirep` erfolgreich war, `delete_bid` danach aber wegen Netzwerk/5xx/Timeout scheitert, darf der Pilot nicht mit einem filed PIREP und haengendem Bid zurueckbleiben. Dafuer braucht v0.7.19 eine kleine persistente Cleanup-Queue:

```rust
struct PendingBidCleanup {
    pirep_id: String,
    bid_id: Option<u64>,
    flight_id: Option<String>,
    reason: String, // "accident_filed" | "hard_landing_override" | future
    created_at: DateTime<Utc>,
    last_attempt_at: Option<DateTime<Utc>>,
    attempts: u32,
}
```

Regeln:

- Nach FILED + `delete_bid` transient-fail: Eintrag persistieren.
- Background-Worker retried `delete_bid(bid_id, flight_id)`.
- Bei Erfolg: Eintrag loeschen und Activity/Event loggen.
- Bei permanentem 4xx ausser bekannten Fallback-Codes: Eintrag sichtbar machen, damit B-011 Cleanup/Details den Zustand anzeigen kann.
- `pending_bid_cleanup` darf nie `cancel_pirep` ausloesen. Der PIREP ist bereits filed; es geht nur noch um die Reservierung.

### Reihenfolge fuer "Flug verwerfen & Cleanup"

```text
Pilot bestaetigt Cleanup
  -> cancel_pirep(pirep_id)
  -> delete_bid(bid_id, flight_id)
  -> local cleanup
  -> Activity Log: "Accident verworfen, PIREP cancelled, Aircraft/Bid freigegeben"
```

Wenn `cancel_pirep` 404 liefert:

- Das PIREP existiert serverseitig nicht mehr oder wurde bereits bereinigt.
- UI fragt wie v0.7.18: **"Remote PIREP nicht gefunden, lokal vergessen?"**
- Bei Zustimmung: lokales Aufraeumen + `delete_bid` trotzdem versuchen, falls `bid_id`/`flight_id` vorhanden.

### Queue-Success-Hook

Wenn Accident-Filing wegen Netzwerk/5xx in `pirep_queue` landet, muss nach erfolgreichem Retry dasselbe Aufraeumen passieren wie bei direktem FILED:

```text
pirep_queue worker filed accident PIREP successfully
  -> delete_bid(bid_id, flight_id)
  -> wenn delete_bid transient fehlschlaegt: pending_bid_cleanup persistieren
  -> remove queue entry
  -> ensure local active flight is gone
  -> activity/log event
```

Dafuer muss die Queue genug Metadaten speichern:

```rust
queued_pirep.bid_id: Option<u64>
queued_pirep.flight_id: Option<String>
queued_pirep.accident: bool
```

Falls die Queue diese Felder heute nicht hat: v0.7.19 muss sie ergaenzen oder den Accident-Pfad darf bei transientem File-Fehler nicht als "fertig" anzeigen.

## Acceptance Criteria

- GAF 707 wird als `accident=true`, `accident_kind=impact`, `confidence=high` klassifiziert.
- GAF 707 erscheint nicht mehr nur als normale "Severe Landing".
- MSFS `Crashed` Event wird abonniert und im aktiven Flug als Accident gelatcht.
- MSFS `CrashReset` loescht den aktiven Accident-Latch nicht; der Status bleibt bis Flight-End/Cleanup/neuem Flight-Start.
- Sim-Event-Pfad setzt Accident sofort; Heuristik-Pfad laeuft nur am Touchdown-Edge, nicht pro 50-Hz-Tick.
- `accident_at` wird in FlightStats, LandingRecord, MQTT/TouchdownPayload und PIREP-Payload persistiert.
- `landing-scoring` bleibt unveraendert; Accident aendert Primary Label/Filing-Policy, nicht die numerische Score-Berechnung.
- X-Plane erkennt harte Impact-Crashes ueber dieselbe Heuristik, auch ohne Sim-spezifischen Crash-Event.
- Der Pilot bekommt bei Heuristik-Faellen die Frage **"War das ein Absturz?"** und kann Accident bestaetigen oder als harte Landung overriden.
- v0.7.19+ Payloads enthalten `accident_classifier_version`; Webapp unterscheidet damit "Classifier lief, kein Accident" von "historischer Payload".
- Bei Multi-Touchdown-PIREPs gewinnt die hoechste Severity ueber alle Touchdowns.
- B-013 Reset-Samples erzeugen keinen Accident.
- B-009 G-Force-Entschaerfung bleibt intakt: `-116 fpm / 2.30 G` wird nicht Accident.
- IFR Accident kann nicht als `clean_ifr` file-n.
- Pilot hat immer eine bewusste Wahl: Accident/Review einreichen, Cleanup, oder zurueck.
- Nach Accident-FILED oder Accident-CANCELLED ist das Aircraft auf phpVMS frei.
- Nach Accident-FILED oder Accident-CANCELLED ist der Bid geloescht bzw. die Reservation entfernt.
- Bei queued Accident-Filing wird der Bid-Drop nach erfolgreichem Queue-Retry nachgeholt.
- Bei FILED-ok/delete_bid-transient-fail wird `pending_bid_cleanup` persistiert und spaeter retried.
- Webapp und Client zeigen denselben Accident-Status.
- Webapp/VPS kann historische Touchdown-Payloads ohne `accident` Feld nachklassifizieren.
- GAF 707 erscheint in der Webapp-History als `Accident`, nicht nur als Score-0-Landung.
- GAF 707 erscheint auf der Webapp-Seite **Landing-Analyse** als `Absturz/Accident`, nicht als normale `Hard Landing` / `Bone-Rattler`-Klassifikation.
- GAF 707 erscheint im AeroACARS-Client im Tab **Landung** als `Absturz/Accident`, nicht als normale `Hard Landing` / `Bone-Rattler`-Klassifikation.
- Control-Fixture aus demselben Export bleibt `not accident`.
- Historische Nachklassifikation mutiert keine alten phpVMS-PIREP-States und loest kein Aircraft/Bid-Cleanup aus.
- Alle neuen Felder sind backwards-compatible (`serde(default)` / optional JSON).

## Offene Entscheidungen

- Soll ein Confirmed Accident automatisch einen VA-Review-Status in phpVMS setzen, falls verfuegbar, oder nur Notes + Score 0?
- Welche deutsche UI-Bezeichnung final: **"War das ein Absturz?"**, **"Unfall erkannt"** oder **"Accident erkannt"**?
- Soll `suspected` schon den Flight-End-Dialog veraendern oder nur einen Warnbanner anzeigen? Default dieser Spec: Dialog mit Pilot-Frage.
- Soll fuer X-Plane spaeter ein eigener Crash-DataRef recherchiert werden, oder reicht dauerhaft die gemeinsame Heuristik?
- Gibt phpVMS/Disposable Basic das Aircraft nach FILED sicher frei, oder brauchen wir einen expliziten Verify-/Repair-Schritt?
- Soll die Webapp Accident-Nachklassifikation nur on-the-fly rendern oder spaeter materialisiert in der DB speichern?
