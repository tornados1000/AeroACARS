## Stream A — Phase-FSM Boarding→Pushback De-Glitch

### A.0 — Befund (verifiziert am BTX8815-Flugschrieb)

`flight-vjmnnNVozNj8MmP7.jsonl.gz`, EDVE-Vorfeld:

| Zeit (UTC) | Ereignis |
|---|---|
| 18:52:53 | `flight_started` — Phase Boarding |
| 18:53:47 → 18:53:58 | **11-Sekunden-Lücke** im Position-Stream (= Sim-Reload/Reposition durch den Piloten) |
| **18:53:58.54** | **EIN** Position-Snapshot mit `groundspeed_kt = 2.63`, `velocity_body_z_fps = -4.43`, AGL springt 9,3→10,5 ft — aber `lat`/`lon` **identisch** zum Sample davor (0,0 m Versatz), `parking_brake = true`, `engines_running = 0` |
| 18:53:58.56 | `phase_changed` Boarding→Pushback **+ `block_off_at` gesetzt** |
| 18:54:04 | nächster Snapshot: `groundspeed_kt = 0.0` wieder |
| 19:27:53 | beide Triebwerke laufen (= echter Pre-Pushback) |
| 19:28:50 | Parkbremse gelöst (= echtes Off-Block) |
| 19:29:15 | `phase_changed` Pushback→TaxiOut |

**Diagnose:** Der Sim-Reload erzeugte eine ~11 s-Lücke im Telemetrie-Stream. Der erste Snapshot danach trug einen transienten Glitch: `groundspeed_kt = 2,63` und eine Rückwärts-Body-Velocity, obwohl das Flugzeug physisch **nicht bewegt** wurde (lat/lon Bit-identisch, Parkbremse gesetzt, Triebwerke aus). Ein einziger Glitch-Tick reichte, um die FSM umzuschalten.

### A.1 — Ursache im Code

`client/src-tauri/src/lib.rs`, `FlightPhase::Boarding`-Branch (~Zeile 15558):

```rust
let on_surface = snap.on_ground
    || (snap.altitude_agl_ft < 5.0 && snap.vertical_speed_fpm.abs() < 50.0);
if on_surface && snap.groundspeed_kt > 0.5 {
    stats.block_off_at = Some(now);
    // … Loadsheet-Logging …
    next_phase = if snap.engines_running == 0 {
        FlightPhase::Pushback
    } else {
        FlightPhase::TaxiOut
    };
}
```

Der Trigger ist **ein einziger Snapshot** mit `groundspeed_kt > 0.5`. Keine Entprellung, kein Abgleich gegen echte Positionsänderung, keine Reload-Lücken-Erkennung. Der Kommentar sagt „actual movement is the only reliable trigger" — aber `groundspeed_kt` aus einem Glitch-Sample ist **keine** echte Bewegung.

**Zweitschaden:** dieselbe Zeile setzt `stats.block_off_at`. Dadurch steht im PIREP die Off-Block-Zeit auf 18:53:58 statt ~19:28:50 — die Blockzeit ist ~35 min zu lang. (`block_snapshot`-Event feuert ebenfalls hier.)

### LE1 — De-Glitch-Gate: echte Bewegung statt Geschwindigkeits-Spike

Der Boarding→Pushback/TaxiOut-Trigger braucht künftig **beide** Bedingungen:

1. **Entprellung** — `groundspeed_kt > 0.5` muss über **≥ 2 aufeinanderfolgende** Snapshots halten. Ein einzelner Spike-Sample zählt nicht.
2. **Echte Positionsverschiebung** — die Großkreis-Distanz zwischen einem beim Boarding-Eintritt (bzw. rollierend) gemerkten Referenzpunkt und der aktuellen Position muss **> `MIN_MOVEMENT_M`** sein (Vorschlag: **5 m**). Ein Glitch-Sample hat 0 m Versatz — er ist damit immun gegen `groundspeed`-Spikes.

Erst wenn beide erfüllt sind, feuert der Übergang und `block_off_at` wird gesetzt.

```rust
// Skizze — finale Form im Code-Slice.
let moved_m = haversine_m(stats.boarding_ref_pos, (snap.lat, snap.lon));
let gs_sustained = snap.groundspeed_kt > 0.5 && stats.prev_gs_over_threshold;
if on_surface && gs_sustained && moved_m > MIN_MOVEMENT_M {
    stats.block_off_at = Some(now);
    next_phase = if snap.engines_running == 0 { Pushback } else { TaxiOut };
}
stats.prev_gs_over_threshold = snap.groundspeed_kt > 0.5;
```

**Warum Positionsverschiebung als Hauptsignal:** ein Reload-Glitch kann `groundspeed_kt` und `velocity_body_*` verfälschen, aber er kann `lat`/`lon` nicht „bewegen" — das Flugzeug steht physisch am Gate. Die Positionsdifferenz ist das fälschungssichere Signal.

`MIN_MOVEMENT_M` = **5 m** (QS-R1 bestätigt). Ein echter Pushback bewegt das Flugzeug zügig zweistellig viele Meter; 5 m ist klein genug um den Pushback-START nicht zu verzögern, groß genug gegen GPS-Jitter (typisch < 1 m im Stand).

### LE2 — Reload-Lücken-Erkennung

Wenn die Zeitdifferenz zwischen zwei verarbeiteten Snapshots das erwartete Tick-Intervall deutlich übersteigt, gilt der erste Snapshot nach der Lücke als **reload-verdächtig**.

**Schwellwert (QS-R1-Fix):** `gap > 2,5 × position_interval(phase)`. In Boarding/Pushback ist `position_interval` = 4 s → Schwelle **> 10 s**. Damit wird die konkrete BTX8815-Lücke von **11 s** sicher erfasst (R0 schlug `> 12 s` bzw. `> 3×` = 12 s vor — das hätte die 11 s-Lücke **verfehlt**; QS-R1-Finding). Der Faktor 2,5 skaliert automatisch mit der Phase und liegt klar über normalem Tick-Jitter.

Der reload-verdächtige Snapshot:
- löst **keinen** geschwindigkeits-/velocity-abgeleiteten Phasenübergang aus (Boarding→Pushback, TakeoffRoll-Trigger etc.);
- setzt die **Referenz-Position** (LE1) und `prev_gs_over_threshold` auf seine eigenen Werte zurück — die Bewegungs-Messung startet sauber nach der Lücke neu.

Damit ist der konkrete BTX8815-Glitch doppelt abgesichert: er kommt direkt nach der 11 s-Lücke (→ LE2 mit `> 10 s`-Schwelle verwirft ihn) **und** hätte auch ohne LE2 mangels Positionsverschiebung (→ LE1) nicht ausgelöst.

**Offene QS-Frage A-3:** Soll LE2 nur Boarding/Pushback absichern, oder die ganze FSM (auch Climb/Descent/Approach)? Ein Reload im Reiseflug könnte ähnliche Glitches erzeugen. Vorschlag R0: **nur Boarding/Pushback** in v0.12.1 (kleiner, gezielter Fix); FSM-weite Reload-Härtung als separater späterer Task.

### LE3 — `block_off_at` / OOOI-Korrektheit

`block_off_at` wird ausschließlich im LE1-Pfad gesetzt. Mit LE1 ist damit automatisch die Off-Block-Zeit korrekt — kein separater Code, nur die Konsequenz von LE1. **Akzeptanz:** im Re-Test darf `block_off_at` nicht mehr auf einem Glitch-Sample liegen.

Kein Re-Score / keine Rückwirkung auf bereits eingereichte PIREPs (forward-only).

## Stream B — VA-Härtung: aktiver GSG-Pilot

### B.0 — Befund (GitHub-Issue #16) + Korrektur aus QS-R0

Ein VA-fremder Tester betrieb AeroACARS und plant es „für seine eigene VA" zu nutzen.

**Wie hat er das geschafft?** Das offizielle Binary kann es *nicht* gewesen sein: `phpvms_login` (`lib.rs:4645`) **verwirft** den vom UI gesendeten `url`-Parameter und erzwingt hart `https://german-sky-group.eu`. Das offizielle Binary kann physisch mit **keiner** anderen phpVMS-Instanz sprechen. Folglich hat der Tester die **Open-Source-Quellen (MIT) selbst gebaut** — `ALLOWED_PHPVMS_HOST` auf seine eigene Domain geändert, `tauri build`, gegen seine eigene phpVMS + seinen eigenen Recorder laufen lassen. GSG-Infrastruktur war dabei **nicht** beteiligt. Das ist bei Open-Source nicht verhinderbar und richtet bei GSG auch keinen Schaden an (eigene Infrastruktur). Die README stellt seit PR #22 klar: Quellcode frei, offizielle Builds GSG-only.

**Korrektur QS-R0 — das ursprüngliche Airline-Gate ist falsch:** R0 wollte `airline.icao == "GSG"` prüfen. Aber: die GSG-phpVMS-Instanz hostet **über 300 Airlines**. „German Sky Group" ist **keine Airline**, sondern die gesamte phpVMS-Installation unter `german-sky-group.eu`. Ein Airline-Gate würde 299 von 300 echten GSG-Piloten aussperren. Hinfällig.

**Was tatsächlich gilt:**
- Das offizielle Binary ist **bereits** auf `german-sky-group.eu` festgenagelt (`ALLOWED_PHPVMS_HOST`). „Ist das ein GSG-phpVMS-Pilot?" ist also für jeden, der sich im offiziellen Build überhaupt einloggen kann, **schon per Konstruktion wahr** — der Build redet mit nichts anderem. Ebenso validiert der Recorder Keys nur gegen `cfg.phpvmsBaseUrl = german-sky-group.eu`.
- **Die echte Lücke:** weder Client noch Recorder prüfen den **Piloten-Status**. Ein inaktiver, gesperrter, abgelehnter oder im Urlaub befindlicher Account kommt genauso durch wie ein aktiver.

**Vorgabe von Thomas (QS-R0):** „es muss geprüft werden ob der Pilot noch aktiv ist oder Urlaub und so → ggf. sperren bzw. Meldung anzeigen." → Das Gate ist ein **Piloten-Status-Gate**, kein Airline-Gate.

### LE4 — Piloten-Status: das `state`-Feld

phpVMS 7 führt jeden User in einem von fünf Zuständen (`App\Models\Enums\UserState`, Integer im `/api/user`-Feld `state`):

| `state` | phpVMS-Bezeichnung | Bedeutung | AeroACARS-Behandlung (Vorschlag) |
|---|---|---|---|
| 1 | ACTIVE | aktiver Pilot | ✅ **zugelassen** |
| 0 | PENDING | Registrierung wartet auf Admin-Freigabe | 🚫 blockiert — „Account noch nicht freigegeben" |
| 2 | REJECTED | Registrierung abgelehnt | 🚫 blockiert — „Account abgelehnt" |
| 3 | ON_LEAVE | im Urlaub / beurlaubt | 🚫 blockiert — freundlicher Hinweis „als im Urlaub markiert" |
| 4 | SUSPENDED | gesperrt | 🚫 blockiert — „Account gesperrt" |

Nur `ACTIVE` darf fliegen/tracken. Alle anderen Zustände werden mit einer **status-spezifischen Meldung** abgewiesen.

**Offene QS-Frage B-1 (blockierend):** Bitte die `state`-Werte gegen eure phpVMS-7-Version bestätigen — am einfachsten mit einem echten GSG-API-Key: `GET https://german-sky-group.eu/api/user` mit `X-API-Key`, und schauen welchen Wert/Typ das Feld `state` hat (Integer 0–4? oder String-Label?). Die obige Tabelle ist der phpVMS-7-Standard, aber Custom-Installs können abweichen.

**Offene QS-Frage B-2:** `ON_LEAVE` (Urlaub) — wirklich **hart blockieren**, oder nur eine **Warn-Meldung** zeigen und das Fliegen trotzdem erlauben? R1-Vorschlag: hart blockieren mit freundlicher Meldung (ein Pilot im Urlaub soll keine PIREPs einreichen) — aber das ist deine operative Entscheidung.

### LE5 — Recorder-seitiges Status-Gate (die echte Durchsetzung)

`recorder/src/provision.ts`: direkt nach `validatePhpVmsKey()`:

```ts
const user = await validatePhpVmsKey(cfg.phpvmsBaseUrl, apiKey);
if (!user) return { error: "phpVMS API key not valid", status: 401 };

// v0.12.1 — Piloten-Status-Gate: nur aktive GSG-Piloten.
if (user.state !== USER_STATE_ACTIVE) {
  return { error: "pilot not active", status: 403, pilot_state: user.state };
}
```

`PhpVmsUser` bekommt das Feld `state?: number`. Bei nicht-aktiv: **keine MQTT-Provisionierung**, kein Broker-User, kein Eintrag in `provisioned_pilots`.

**Fail-closed (QS-R1-P2-Fix):** `user.state !== USER_STATE_ACTIVE` blockt automatisch auch ein fehlendes/`undefined` `state` — das ist gewollt. Voraussetzung: B-1 bestätigt vorab, dass `/api/user` das Feld zuverlässig liefert, sonst sperrt das Gate alle aus.

**Wichtig — das Gate läuft VOR dem Cache-Lookup** (`getProvisionedPilot`): so wird auch ein Pilot, der **früher** aktiv provisioniert wurde und **inzwischen** gesperrt ist, beim nächsten `/api/provision`-Call abgewiesen. **Aber Achtung — das reicht nicht allein:** siehe LE7, der Client ruft `/api/provision` nur wenn **kein** MQTT-Cache existiert.

### LE6 — Client-seitiges Status-Gate an ALLEN `get_profile()`-Pfaden

**QS-R1-P1-Fix:** R1 nannte nur `phpvms_login`. Tatsächlich gibt es mehrere Pfade, die `get_profile()` aufrufen und eine Session aufbauen — alle brauchen das Gate:

| Pfad | Funktion (`client/src-tauri/src/lib.rs`) | Wann |
|---|---|---|
| Login | `phpvms_login` (~4645) | Pilot gibt API-Key ein |
| Session-Restore | `phpvms_load_session` (~4870) | App-Start mit gespeicherter Session |
| Profil-Refresh | `phpvms_refresh_profile` | periodische Aktualisierung |

Gemeinsame Helper-Funktion, in **allen** Pfaden direkt nach `client.get_profile()`:

```rust
let profile = client.get_profile().await?;
if let Some(reason) = pilot_state_block_reason(&profile) {
    // v0.12.1 — gesperrter/inaktiver Pilot: MQTT-Cache wegwerfen (LE7)
    clear_mqtt_credentials_cache();
    // Session nicht aufbauen / abbrechen
    return Err(UiError::new(reason, /* i18n-Key, siehe LE8 */));
}
```

`api-client::Profile` bekommt ein neues Feld `state: Option<i32>` (`#[serde(default)]`). `pilot_state_block_reason()` mappt den Status auf einen `UiError`-Code — `pilot_pending` / `pilot_rejected` / `pilot_on_leave` / `pilot_suspended` / `pilot_state_unknown` — oder `None` bei ACTIVE. **`None`/unbekannter `state`-Wert ⇒ `pilot_state_unknown` ⇒ blockiert (fail-closed, QS-R1-P2)**, konsistent zum Recorder-Gate.

Bei `phpvms_load_session` heißt „blockiert": die gespeicherte Session wird **nicht** wiederhergestellt — gleicher Pfad wie ein abgelehnter API-Key (`Err(Unauthenticated)`-Zweig, ~4936): Key/Site-Config löschen, Activity-Log-Eintrag, Pilot landet auf der Login-Form mit der Status-Meldung.

### LE7 — MQTT-Credential-Cache: Bypass schließen

**QS-R1-P1-Fix.** `init_mqtt_publisher_via_provisioning` (`lib.rs:4759`) prüft **zuerst den Keyring-Cache** (`MQTT_KEYRING_*`). Sind dort Credentials → werden sie direkt benutzt und **`provision()` wird nie aufgerufen**. Folge: ein einmal aktiv provisionierter Pilot, der später gesperrt wird, startet das Live-Tracking weiter mit gecachten Broker-Credentials — das Recorder-Gate (LE5) sieht ihn nie.

**v0.12.1-Lösung — Cache-Invalidierung am Status-Gate:**
- Das Client-Status-Gate (LE6) ruft bei Nicht-ACTIVE `clear_mqtt_credentials_cache()` (existiert bereits, `lib.rs:4840`). Damit ist der Cache weg; beim nächsten App-Start findet `init_mqtt_publisher_via_provisioning` keinen Cache → ruft `provision()` → Recorder-Gate (LE5) greift.
- Ergebnis: ein zwischen zwei Sessions gesperrter Pilot wird beim nächsten Login/Restore sauber abgefangen — Client-Meldung **und** kein Live-Tracking mehr.

**Dokumentierte Grenze — Live-Revocation mitten in der Session:** wird ein Pilot gesperrt **während die App läuft**, behält der bereits gestartete MQTT-Publisher seine gültigen Broker-Credentials bis zum nächsten App-Neustart. Eine echte Sofort-Trennung bräuchte server-seitige Revocation (Recorder validiert aktive Publisher periodisch neu gegen phpVMS und entfernt den Mosquitto-User, oder Broker-Auth-Plugin).

**Offene QS-Frage B-5:** Reicht für v0.12.1 die Login/Restore-Abfang-Lösung (Mid-Session-Sperre wirkt erst beim nächsten Start), oder soll die server-seitige Live-Revocation gleich mit rein? Vorschlag R2: Login/Restore-Lösung für v0.12.1, Live-Revocation als separater Recorder-Task danach — hält v0.12.1 klein und liefert 95 % des Nutzens (Sperren zwischen Sessions ist der Normalfall).

| Key | DE |

### LE9 — Nicht im Scope (bewusst dokumentiert)

- **Open-Source-Forks gegen fremde phpVMS-Instanzen** sind nicht verhinderbar (MIT-Lizenz) und **kein Ziel** — wer AeroACARS gegen seine *eigene* Infrastruktur baut, berührt GSG nicht (so kam Issue #16 zustande). Geschützt wird GSG-Infrastruktur (offizieller Build, Recorder, MQTT-Broker), nicht der Quellcode.
- **phpVMS-Registrierungs-Policy (wichtige Empfehlung):** das Status-Gate wirkt nur, wenn selbst-registrierte Accounts auch wirklich erst `PENDING` sind. Wenn `german-sky-group.eu` neue Registrierungen **automatisch auf ACTIVE** setzt, kann sich jeder einen aktiven GSG-Account anlegen und das Gate läuft ins Leere. **Empfehlung:** in den phpVMS-Einstellungen „Registrierung → Admin-Freigabe erforderlich" aktivieren. Das ist eine phpVMS-Admin-Einstellung, kein AeroACARS-Code — aber der notwendige Partner zu LE4–LE7. Thomas entscheidet/prüft das separat.
- Der bestehende **Domain-Lock** (`ALLOWED_PHPVMS_HOST`) bleibt unverändert — er ist die Schicht, die das offizielle Binary an GSG bindet.

## Datenmodell / Wire-Format

Stream A: **keine** Wire-Änderung. `block_off_at` und die `phase_changed`-Events behalten ihr Format — sie feuern nur zum korrekten Zeitpunkt.

Stream B:
- **`api-client::Profile`** bekommt ein neues Feld `state: Option<i32>` (`#[serde(default)]`) — phpVMS `/api/user` liefert es bereits, es wird aktuell nur nicht deserialisiert. Kein Wire-Format-Bruch.
- **Recorder `PhpVmsUser`** bekommt `state?: number`.
- Provisioning-Response-Format (`ProvisionResult`) unverändert; im Nicht-aktiv-Fall der bestehende `{ error, status }`-Pfad mit `status: 403` (+ optional `pilot_state` zur Diagnose).
- Keine Änderung am Touchdown-/PIREP-Wire-Format.

Neue interne Felder in `FlightStats` (Stream A, nicht persistiert / nicht im Wire):
- `boarding_ref_pos: Option<(f64, f64)>` — Referenz-Position für die Bewegungs-Messung.
- `prev_gs_over_threshold: bool` — Entprellungs-Merker.
- `last_snapshot_at: Option<Instant/DateTime>` — für die Lücken-Erkennung (LE2; evtl. existiert schon ein Äquivalent — im Code-Slice prüfen).
