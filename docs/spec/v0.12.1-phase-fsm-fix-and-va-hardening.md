## Stream A — Phase-FSM Boarding→Pushback De-Glitch

### A.0 — Befund (verifiziert am BTX8815-Flugschrieb)

`flight-vjmnnNVozNj8MmP7.jsonl.gz`, EDVE-Vorfeld:

| Zeit (UTC) | Ereignis |
|---|---|
| 18:52:53 | `flight_started` — Phase Boarding |
| 18:53:47 → 18:53:58 | **11-Sekunden-Lücke** im Position-Stream (= Sim-Reload/Reposition durch den Piloten) |
| **18:53:58.54** | **EIN** Position-Snapshot mit `groundspeed_kt = 2.63`, `velocity_body_z_fps = -4.43`, AGL springt 9,3→10,5 ft — aber `lat`/`lon` **identisch** zum Sample davor (0,0 m Versatz), `parking_brake = true`, `engines_running = 0` |
| 18:53:58.56 | `phase_changed` Boarding→Pushback **+ `block_off_at` gesetzt** |
| 18:54:04 | nächster Snapshot: `groundspeed_kt = 0.0` wieder |
| 19:27:53 | beide Triebwerke laufen (= echter Pre-Pushback) |
| 19:28:50 | Parkbremse gelöst (= echtes Off-Block) |
| 19:29:15 | `phase_changed` Pushback→TaxiOut |

**Diagnose:** Der Sim-Reload erzeugte eine ~11 s-Lücke im Telemetrie-Stream. Der erste Snapshot danach trug einen transienten Glitch: `groundspeed_kt = 2,63` und eine Rückwärts-Body-Velocity, obwohl das Flugzeug physisch **nicht bewegt** wurde (lat/lon Bit-identisch, Parkbremse gesetzt, Triebwerke aus). Ein einziger Glitch-Tick reichte, um die FSM umzuschalten.

### A.1 — Ursache im Code

`client/src-tauri/src/lib.rs`, `FlightPhase::Boarding`-Branch (~Zeile 15558):

```rust
let on_surface = snap.on_ground
    || (snap.altitude_agl_ft < 5.0 && snap.vertical_speed_fpm.abs() < 50.0);
if on_surface && snap.groundspeed_kt > 0.5 {
    stats.block_off_at = Some(now);
    // … Loadsheet-Logging …
    next_phase = if snap.engines_running == 0 {
        FlightPhase::Pushback
    } else {
        FlightPhase::TaxiOut
    };
}
```

Der Trigger ist **ein einziger Snapshot** mit `groundspeed_kt > 0.5`. Keine Entprellung, kein Abgleich gegen echte Positionsänderung, keine Reload-Lücken-Erkennung. Der Kommentar sagt „actual movement is the only reliable trigger" — aber `groundspeed_kt` aus einem Glitch-Sample ist **keine** echte Bewegung.

**Zweitschaden:** dieselbe Zeile setzt `stats.block_off_at`. Dadurch steht im PIREP die Off-Block-Zeit auf 18:53:58 statt ~19:28:50 — die Blockzeit ist ~35 min zu lang. (`block_snapshot`-Event feuert ebenfalls hier.)

### LE1 — De-Glitch-Gate: echte Bewegung statt Geschwindigkeits-Spike

Der Boarding→Pushback/TaxiOut-Trigger braucht künftig **beide** Bedingungen:

1. **Entprellung** — `groundspeed_kt > 0.5` muss über **≥ 2 aufeinanderfolgende** Snapshots halten. Ein einzelner Spike-Sample zählt nicht.
2. **Echte Positionsverschiebung** — die Großkreis-Distanz zwischen einem beim Boarding-Eintritt (bzw. rollierend) gemerkten Referenzpunkt und der aktuellen Position muss **> `MIN_MOVEMENT_M`** sein (Vorschlag: **5 m**). Ein Glitch-Sample hat 0 m Versatz — er ist damit immun gegen `groundspeed`-Spikes.

Erst wenn beide erfüllt sind, feuert der Übergang und `block_off_at` wird gesetzt.

```rust
// Skizze — finale Form im Code-Slice.
let moved_m = haversine_m(stats.boarding_ref_pos, (snap.lat, snap.lon));
let gs_sustained = snap.groundspeed_kt > 0.5 && stats.prev_gs_over_threshold;
if on_surface && gs_sustained && moved_m > MIN_MOVEMENT_M {
    stats.block_off_at = Some(now);
    next_phase = if snap.engines_running == 0 { Pushback } else { TaxiOut };
}
stats.prev_gs_over_threshold = snap.groundspeed_kt > 0.5;
```

**Warum Positionsverschiebung als Hauptsignal:** ein Reload-Glitch kann `groundspeed_kt` und `velocity_body_*` verfälschen, aber er kann `lat`/`lon` nicht „bewegen" — das Flugzeug steht physisch am Gate. Die Positionsdifferenz ist das fälschungssichere Signal.

`MIN_MOVEMENT_M` = **5 m** (QS-R1 bestätigt). Ein echter Pushback bewegt das Flugzeug zügig zweistellig viele Meter; 5 m ist klein genug um den Pushback-START nicht zu verzögern, groß genug gegen GPS-Jitter (typisch < 1 m im Stand).

### LE2 — Reload-Lücken-Erkennung

Wenn die Zeitdifferenz zwischen zwei verarbeiteten Snapshots das erwartete Tick-Intervall deutlich übersteigt, gilt der erste Snapshot nach der Lücke als **reload-verdächtig**.

**Schwellwert (QS-R1-Fix):** `gap > 2,5 × position_interval(phase)`. In Boarding/Pushback ist `position_interval` = 4 s → Schwelle **> 10 s**. Damit wird die konkrete BTX8815-Lücke von **11 s** sicher erfasst (R0 schlug `> 12 s` bzw. `> 3×` = 12 s vor — das hätte die 11 s-Lücke **verfehlt**; QS-R1-Finding). Der Faktor 2,5 skaliert automatisch mit der Phase und liegt klar über normalem Tick-Jitter.

Der reload-verdächtige Snapshot:
- löst **keinen** geschwindigkeits-/velocity-abgeleiteten Phasenübergang aus (Boarding→Pushback, TakeoffRoll-Trigger etc.);
- setzt die **Referenz-Position** (LE1) und `prev_gs_over_threshold` auf seine eigenen Werte zurück — die Bewegungs-Messung startet sauber nach der Lücke neu.

Damit ist der konkrete BTX8815-Glitch doppelt abgesichert: er kommt direkt nach der 11 s-Lücke (→ LE2 mit `> 10 s`-Schwelle verwirft ihn) **und** hätte auch ohne LE2 mangels Positionsverschiebung (→ LE1) nicht ausgelöst.

**Scope (QS-R1 bestätigt, A-3):** LE2 sichert in v0.12.1 **nur Boarding/Pushback** ab — kleiner, gezielter Fix. FSM-weite Reload-Härtung (Climb/Descent/Approach) ist ein separater späterer Task.

### LE3 — `block_off_at` / OOOI-Korrektheit

`block_off_at` wird ausschließlich im LE1-Pfad gesetzt. Mit LE1 ist damit automatisch die Off-Block-Zeit korrekt — kein separater Code, nur die Konsequenz von LE1. **Akzeptanz:** im Re-Test darf `block_off_at` nicht mehr auf einem Glitch-Sample liegen.

Kein Re-Score / keine Rückwirkung auf bereits eingereichte PIREPs (forward-only).

## Stream B — VA-Härtung: aktiver GSG-Pilot

### B.0 — Befund (GitHub-Issue #16) + Korrektur aus QS-R0

Ein VA-fremder Tester betrieb AeroACARS und plant es „für seine eigene VA" zu nutzen.

**Wie hat er das geschafft?** Das offizielle Binary kann es *nicht* gewesen sein: `phpvms_login` (`lib.rs:4645`) **verwirft** den vom UI gesendeten `url`-Parameter und erzwingt hart `https://german-sky-group.eu`. Das offizielle Binary kann physisch mit **keiner** anderen phpVMS-Instanz sprechen. Folglich hat der Tester die **Open-Source-Quellen (MIT) selbst gebaut** — `ALLOWED_PHPVMS_HOST` auf seine eigene Domain geändert, `tauri build`, gegen seine eigene phpVMS + seinen eigenen Recorder laufen lassen. GSG-Infrastruktur war dabei **nicht** beteiligt. Das ist bei Open-Source nicht verhinderbar und richtet bei GSG auch keinen Schaden an (eigene Infrastruktur). Die README stellt seit PR #22 klar: Quellcode frei, offizielle Builds GSG-only.

**Korrektur QS-R0 — das ursprüngliche Airline-Gate ist falsch:** R0 wollte `airline.icao == "GSG"` prüfen. Aber: die GSG-phpVMS-Instanz hostet **über 300 Airlines**. „German Sky Group" ist **keine Airline**, sondern die gesamte phpVMS-Installation unter `german-sky-group.eu`. Ein Airline-Gate würde 299 von 300 echten GSG-Piloten aussperren. Hinfällig.

**Was tatsächlich gilt:**
- Das offizielle Binary ist **bereits** auf `german-sky-group.eu` festgenagelt (`ALLOWED_PHPVMS_HOST`). „Ist das ein GSG-phpVMS-Pilot?" ist also für jeden, der sich im offiziellen Build überhaupt einloggen kann, **schon per Konstruktion wahr** — der Build redet mit nichts anderem. Ebenso validiert der Recorder Keys nur gegen `cfg.phpvmsBaseUrl = german-sky-group.eu`.
- **Die echte Lücke:** weder Client noch Recorder prüfen den **Piloten-Status**. Ein inaktiver, gesperrter, abgelehnter oder im Urlaub befindlicher Account kommt genauso durch wie ein aktiver.

**Vorgabe von Thomas (QS-R0):** „es muss geprüft werden ob der Pilot noch aktiv ist oder Urlaub und so → ggf. sperren bzw. Meldung anzeigen." → Das Gate ist ein **Piloten-Status-Gate**, kein Airline-Gate.

### LE4 — Piloten-Status: das `state`-Feld

phpVMS 7 führt jeden User in einem von **sechs** Zuständen (`App\Enums\UserState`, Integer im `/api/user`-Feld `state`):

| `state` | phpVMS-Bezeichnung | Bedeutung | AeroACARS-Behandlung |
|---|---|---|---|
| 1 | ACTIVE | aktiver Pilot | ✅ **zugelassen** |
| 0 | PENDING | Registrierung wartet auf Admin-Freigabe | 🚫 blockiert — „Account noch nicht freigegeben" |
| 2 | REJECTED | Registrierung abgelehnt | 🚫 blockiert — „Account abgelehnt" |
| 3 | ON_LEAVE | im Urlaub / beurlaubt | 🚫 blockiert — freundlicher Hinweis „als im Urlaub markiert" |
| 4 | SUSPENDED | gesperrt | 🚫 blockiert — „Account gesperrt" |
| 5 | DELETED | gelöscht | 🚫 blockiert — nutzt die „gesperrt"-Meldung (harter, definitiver Block) |

Nur `ACTIVE` darf fliegen/tracken. Alle anderen Zustände werden mit einer **status-spezifischen Meldung** abgewiesen.

**B-1 — verifiziert (Code-QS).** Das Mapping wurde gegen die offizielle phpVMS-Quelle (`app/Enums/UserState.php`) bestätigt: `PENDING=0, ACTIVE=1, REJECTED=2, ON_LEAVE=3, SUSPENDED=4, DELETED=5` — ein Integer-Enum. Der R0/R1-Verdacht „fünf Zustände" war unvollständig: phpVMS 7 hat den zusätzlichen Zustand **DELETED=5**, der jetzt explizit mitbehandelt wird (harter Block, „gesperrt"-Meldung). Damit ist B-1 nicht mehr offen.

**`ON_LEAVE` (QS-R1 bestätigt, B-2):** wird **hart blockiert** — mit freundlicher Meldung (ein Pilot im Urlaub soll keine PIREPs einreichen).

### LE5 — Recorder-seitiges Status-Gate (die echte Durchsetzung)

`recorder/src/provision.ts`: direkt nach `validatePhpVmsKey()`:

```ts
const user = await validatePhpVmsKey(cfg.phpvmsBaseUrl, apiKey);
if (!user) return { error: "phpVMS API key not valid", status: 401 };

// v0.12.1 — Piloten-Status-Gate: nur aktive GSG-Piloten.
if (user.state !== USER_STATE_ACTIVE) {
  return { error: "pilot not active", status: 403, pilot_state: user.state };
}
```

`PhpVmsUser` bekommt das Feld `state?: number`. Bei nicht-aktiv: **keine MQTT-Provisionierung**, kein Broker-User, kein Eintrag in `provisioned_pilots`.

**Fail-closed (QS-R1-P2-Fix):** `user.state !== USER_STATE_ACTIVE` blockt automatisch auch ein fehlendes/`undefined` `state` — das ist gewollt. Voraussetzung: B-1 bestätigt vorab, dass `/api/user` das Feld zuverlässig liefert, sonst sperrt das Gate alle aus.

**Wichtig — das Gate läuft VOR dem Cache-Lookup** (`getProvisionedPilot`): so wird auch ein Pilot, der **früher** aktiv provisioniert wurde und **inzwischen** gesperrt ist, beim nächsten `/api/provision`-Call abgewiesen. **Aber Achtung — das reicht nicht allein:** siehe LE7, der Client ruft `/api/provision` nur wenn **kein** MQTT-Cache existiert.

### LE6 — Client-seitiges Status-Gate an den Login-/Restore-Pfaden

**QS-R1-P1-Fix:** R1 nannte nur `phpvms_login`. Eine Session wird über **zwei** `get_profile()`-Pfade aufgebaut — beide brauchen das Gate:

| Pfad | Funktion (`client/src-tauri/src/lib.rs`) | Wann | v0.12.1 gegated? |
|---|---|---|---|
| Login | `phpvms_login` (~4645) | Pilot gibt API-Key ein | ✅ ja |
| Session-Restore | `phpvms_load_session` (~4870) | App-Start mit gespeicherter Session | ✅ ja |
| Profil-Refresh | `phpvms_refresh_profile` | periodisch / nach Flug-Ende, **bei laufender App** | ❌ **nein** — siehe unten |

**`phpvms_refresh_profile` wird in v0.12.1 bewusst NICHT gegated (QS-R2-P2-Fix).** Der Refresh läuft mitten in einer aktiven Session (Bid-Refresh, nach Flug-Ende). Würde das Gate hier zuschlagen, wäre das eine **Mid-Session-Live-Sperre** — die bräuchte sauberes Herunterfahren des laufenden MQTT-Publishers, Session-Teardown und eine sichtbare Logout-Meldung im Frontend. Das ist exakt der Live-Revocation-Umfang, der laut B-5 **nicht** in v0.12.1 gehört. v0.12.1 gated daher nur Login + Session-Restore (= jeder *neue* Session-Aufbau); eine Sperre, die während einer laufenden Session in phpVMS gesetzt wird, greift beim nächsten App-Start. Konsistent mit LE7.

Gemeinsame Helper-Funktion, in **beiden gegateten** Pfaden direkt nach `client.get_profile()`:

```rust
let profile = client.get_profile().await?;
if let Some(reason) = pilot_state_block_reason(&profile) {
    // v0.12.1 — gesperrter/inaktiver Pilot: MQTT-Cache wegwerfen (LE7)
    clear_mqtt_credentials_cache();
    // Session nicht aufbauen / abbrechen
    return Err(UiError::new(reason, /* i18n-Key, siehe LE8 */));
}
```

`api-client::Profile` bekommt ein neues Feld `state: Option<i32>` (`#[serde(default)]`). `pilot_state_block_reason()` mappt den Status auf einen `UiError`-Code — `pilot_pending` / `pilot_rejected` / `pilot_on_leave` / `pilot_suspended` / `pilot_state_unknown` — oder `None` bei ACTIVE. **`None`/unbekannter `state`-Wert ⇒ `pilot_state_unknown` ⇒ blockiert (fail-closed, QS-R1-P2)**, konsistent zum Recorder-Gate.

Bei `phpvms_load_session` heißt „blockiert": die gespeicherte Session wird **nicht** wiederhergestellt — gleicher Pfad wie ein abgelehnter API-Key (`Err(Unauthenticated)`-Zweig, ~4936): Key/Site-Config löschen, Activity-Log-Eintrag, Pilot landet auf der Login-Form mit der Status-Meldung.

### LE7 — MQTT-Credential-Cache: Bypass schließen

**QS-R1-P1-Fix.** `init_mqtt_publisher_via_provisioning` (`lib.rs:4759`) prüft **zuerst den Keyring-Cache** (`MQTT_KEYRING_*`). Sind dort Credentials → werden sie direkt benutzt und **`provision()` wird nie aufgerufen**. Folge: ein einmal aktiv provisionierter Pilot, der später gesperrt wird, startet das Live-Tracking weiter mit gecachten Broker-Credentials — das Recorder-Gate (LE5) sieht ihn nie.

**v0.12.1-Lösung — Cache-Invalidierung am Status-Gate:**
- Das Client-Status-Gate (LE6) ruft bei Nicht-ACTIVE `clear_mqtt_credentials_cache()` (existiert bereits, `lib.rs:4840`). Damit ist der Cache weg; beim nächsten App-Start findet `init_mqtt_publisher_via_provisioning` keinen Cache → ruft `provision()` → Recorder-Gate (LE5) greift.
- Ergebnis: ein zwischen zwei Sessions gesperrter Pilot wird beim nächsten Login/Restore sauber abgefangen — Client-Meldung **und** kein Live-Tracking mehr.

**Dokumentierte Grenze — Live-Revocation mitten in der Session:** wird ein Pilot gesperrt **während die App läuft**, behält der bereits gestartete MQTT-Publisher seine gültigen Broker-Credentials bis zum nächsten App-Neustart. Eine echte Sofort-Trennung bräuchte server-seitige Revocation (Recorder validiert aktive Publisher periodisch neu gegen phpVMS und entfernt den Mosquitto-User, oder Broker-Auth-Plugin).

**Scope (QS-R2 bestätigt, B-5):** v0.12.1 setzt **nur die Login/Restore-Abfang-Lösung** um — eine Mid-Session-Sperre (Pilot wird gesperrt während die App läuft) wirkt erst beim nächsten App-Start. Die server-seitige Live-Revocation (laufenden Publisher sofort trennen) ist ein **separater Recorder-Task nach v0.12.1**. Begründung: Sperren zwischen Sessions ist der Normalfall — die Login/Restore-Lösung liefert den Großteil des Nutzens, ohne den Mid-Session-Teardown-Umfang in v0.12.1 zu ziehen. Konsistenz-Konsequenz: `phpvms_refresh_profile` bleibt ungegated (LE6).

### LE9 — Nicht im Scope (bewusst dokumentiert)

- **Open-Source-Forks gegen fremde phpVMS-Instanzen** sind nicht verhinderbar (MIT-Lizenz) und **kein Ziel** — wer AeroACARS gegen seine *eigene* Infrastruktur baut, berührt GSG nicht (so kam Issue #16 zustande). Geschützt wird GSG-Infrastruktur (offizieller Build, Recorder, MQTT-Broker), nicht der Quellcode.

Wenn `german-sky-group.eu` neue Registrierungen **automatisch auf ACTIVE** setzt, kann sich jeder einen aktiven GSG-Account anlegen und das Gate läuft ins Leere. **Empfehlung:** in den phpVMS-Einstellungen „Registrierung → Admin-Freigabe erforderlich" aktivieren. Das ist eine phpVMS-Admin-Einstellung, kein AeroACARS-Code — aber der notwendige Partner zu LE4–LE7. Thomas entscheidet/prüft das separat.

## Stream C — Approach-Stability: Study-Sim-Flaps + Flare-Artefakt

### C.0 — Befund (Flug GSG225, Michel, CL650, X-Plane 12)

Pilot-Beschwerde: die Approach-Stability-Card zeigt `LANDING CONFIG: INCOMPLETE` (rot) und `MAX V/S DEV < 500 ft: 826` — obwohl Gear + Flaps bei 1000 ft gesetzt waren und der Anflug ruhig war. Thomas' Erst-Verdacht „wegen GA": **am Flugschrieb widerlegt** — es gibt keinen Go-Around, die Phasenkette ist sauber (Descent→Approach→Final→Landing, kein Climb-Back, kein zweiter Approach).

Stattdessen zwei unabhängige Ursachen, beide im Log verifiziert (`flight-o0Dv3exNy1PMeRpE.jsonl.gz`):

**Befund C-1 — `flaps_position` ist über den GANZEN Flug `0.0`.** Aircraft: „Challenger 650 published by X-Aviation" (= Hot-Start CL650, ein Study-Level-Add-on). Der X-Plane-Adapter liest Flaps über `sim/flightmodel2/controls/flap_handle_deploy_ratio` (`sim-xplane/src/dataref.rs:261`). Der Hot-Start CL650 fährt diese Standard-DataRef **nicht** — er nutzt eigene `CL650/...`-DataRefs. Ergebnis: AeroACARS „sieht" die Flaps nie → `flaps_position = 0` → die LANDING-CONFIG-Prüfung im Stability-Gate meldet korrekt-aus-falschem-Input „INCOMPLETE". Gear (`gear/deploy_ratio[0]`) liest dagegen korrekt `1.0` — der hatte 2026-05-04 schon mal denselben Bug und wurde mit dem `[0]`-Bracket gefixt; Flaps haben diesen Schutz nie bekommen.

**Befund C-2 — `MAX V/S DEV < 500 ft = 826` ist ein Flare-Artefakt.** Der Anflug-V/S lag bei -550…-700 fpm (3°-Soll bei ~122 kt GS ≈ -648 fpm → Abweichung meist < 150 fpm; ein kurzer Dip auf -907 fpm bei 240 ft → ~260 fpm). Der 826er-Wert stammt von **einem Sample bei 9 ft AGL**: beim Abfangen ballonte das Flugzeug kurz auf **+198 fpm** — `|+198 − (−648)| ≈ 840`. Das ist der Flare, keine Anflug-Instabilität. Die Flare-Cutoff-Zone (letzte 3 s vor TD) reicht nicht weit genug zurück: der Ballon liegt ~5 s vor dem Aufsetzen und fällt damit noch ins „Approach"-Fenster. Michels „es waren 200–280" trifft seinen echten Anflug — die 826 ist das Artefakt.

### LE10 — CL650-/Study-Sim-Flaps: DataRef-Befund

`sim/flightmodel2/controls/flap_handle_deploy_ratio` ist für Standard-X-Plane-Flugzeuge korrekt, aber Study-Level-Add-ons (Hot-Start CL650, ToLiss, FlightFactor …) bedienen sie oft nicht. Eine vollständige Lösung — eine **CL650-Aircraft-Profil-DataRef** (analog zum bestehenden `aircraft_profile`-System, das MSFS-PMDG/Fenix schon kennt) — braucht die exakten Hot-Start-`CL650/...`-DataRef-Namen.

**Entscheidung (QS-R4 bestätigt, C-1):** v0.12.1 macht **nur die fail-soft-Behandlung** (LE11) — kein CL650-spezifischer DataRef. Die echte CL650-Flaps-DataRef ist ein **Folge-Task** (braucht die Hot-Start-`CL650/...`-DataRef-Namen). Begründung: fail-soft hilft sofort *jedem* Study-Sim, die CL650-DataRef ist additiv und ohne Zeitdruck nachrüstbar.

### LE11 — LANDING-CONFIG fail-soft wenn Flaps nicht lesbar

Wenn `flaps_position` über den **gesamten Approach/Final-Gate** konstant `0.0` ist, der Anflug aber sonst klar ein normaler bemannter Anflug ist (Gear unten **und** Anfluggeschwindigkeit im Landebereich — eine echte 0-Flaps-Landung wäre deutlich schneller), ist die Flaps-Lesung **untrauwürdig**, nicht „Pilot hat Flaps vergessen".

In dem Fall meldet die LANDING-CONFIG-Kachel **nicht rot „INCOMPLETE"**, sondern einen neutralen Zustand „nicht bewertbar / Flaps n/v" — und der Faktor fließt **nicht als Penalty** in den Stability-Score ein (analog zu den Skip-Outcomes anderswo: lieber kein Urteil als ein falsches).

**Offene QS-Frage C-2:** Diskriminator-Schwelle „Anfluggeschwindigkeit im Landebereich" — Vorschlag: IAS am Gate-Ende < 160 kt. Bei einer echten flaplosen Landung läge man darüber. Thomas/Code-QS bestätigt die Zahl.

### LE12 — MAX V/S DEV: Flare-/Ground-Effect-Samples ausschließen

Die Kennzahl `approach_max_vs_deviation_below_500_fpm` bekommt eine **untere AGL-Grenze**: Samples unter einer Flare-/Ground-Effect-Höhe (Vorschlag: **< 50 ft AGL**) zählen nicht mehr in die Max-Abweichung. Ein V/S-Wert bei 9 ft AGL beschreibt das Abfangen, nicht die Anflug-Stabilität. Damit verschwindet das 826er-Artefakt; die Kennzahl bewertet nur noch den echten Anflug (50–500 ft).

(Die bestehende 3-s-Flare-Cutoff-Zone bleibt — LE12 ist eine zusätzliche, höhen-basierte Absicherung, weil die Zeit-basierte Zone bei frühem/langem Flare nicht weit genug zurückreicht.)

## Stream D — Autostart bricht nach Update (macOS)

### D.0 — Befund (Michel, macOS, v0.11.2 → v0.12.0)

Pilot-Beschwerde: nach dem Update greift der Auto-Start nicht mehr — erst nach manuellem Aus-/Einschalten des Toggles. Nach einer kompletten Neuinstallation funktioniert er dagegen.

**Ursache — Frontend-Mount-Race, der die Backend-Source-of-Truth überschreibt.** In `client/src/App.tsx` laufen beim Mount **zwei** `useEffect`s:

- **Effect A** (`~221`): liest async `auto_start_get_enabled` aus dem Backend und zieht den Frontend-State nach, falls abweichend.
- **Effect B** (`~247`): pusht `auto_start_set_enabled(autoStart)` ins Backend — **läuft auch beim ersten Mount sofort**, mit dem Initial-State `autoStart = loadAutoStart()` (= localStorage).

Effect B feuert synchron beim Mount und ist **schneller** als der async-Round-Trip von Effect A. Wenn das macOS-WebView-`localStorage` nach dem Update zurückgesetzt ist (passiert bei unsignierten Builds / Bundle-Wechsel / Gatekeeper-Re-Quarantäne), liefert `loadAutoStart()` den Default `false` → Effect B ruft sofort `auto_start_set_enabled(false)` → **das Backend überschreibt sein korrekt aus `auto_start.json` geladenes `true` mit `false`** (und persistiert das `false` zurück in die Datei). Effect A liest danach `false` — passt scheinbar, keine Korrektur. Der Auto-Start ist tot, bis der Pilot den Toggle Aus→An schaltet.

Bei einer **Neuinstallation** tritt das nicht auf, weil der Pilot ohnehin frisch konfiguriert / der Backend-Default `false` korrekt ist.

Der Code-Kommentar bei Effect A behauptet „Backend-Wert ist Source of truth" — Effect B verletzt das, weil es den Frontend-Wert beim Mount nach unten pusht, **bevor** Effect A den Backend-Wert gelesen hat.

### LE13 — Effect B nicht beim ersten Mount feuern

Der Backend-Push (`auto_start_set_enabled`) darf **nur bei einer echten Nutzer-Änderung** des Toggles laufen, nicht beim initialen Mount. Umsetzung: ein `didMount`-Ref (oder gleichwertig), der Effect B den **ersten** Lauf überspringen lässt. Der erste Mount liest dann ausschließlich (Effect A) — die Backend-Datei `auto_start.json` bleibt unangetastet Source-of-Truth.

Damit ist der Fix robust **unabhängig davon, warum** `localStorage` verloren ging — der Frontend-Cache kann jederzeit leer sein, ohne den Backend-Zustand zu beschädigen.

**Akzeptanz:** Simuliert man „localStorage leer + Backend `auto_start.json` = `{\"enabled\":true}`", muss nach dem Mount `auto_start_get_enabled` weiterhin `true` liefern und die Datei `true` bleiben.

## Stream E — Resume nach Crash: kein Auto-Confirm bei verdächtiger Position

### E.0 — Befund (Pilot-Wunsch Michel, X-Plane 12)

Michel: nach einem (Sim-)Crash soll der Resume **nicht automatisch** durchlaufen — der Pilot soll aktiv „Flug fortsetzen" drücken müssen, weil das Reinladen nach einem Crash „hakelig" ist und man „irgendwo auf dem Ground spawnt". Frage: findet eine Positions-Prüfung statt (lat/lon + Höhe)?

**Heutiger Stand — geprüft:**
- Es gibt **zwei** Resume-Pfade. (1) *Sim-Disconnect/Pause-Resume* (App lief durch, Sim kurz weg): misst die Reposition-Drift (`RESUME_DRIFT_TOAST/WARN/EXTREME_NM`) und **loggt** sie ins Activity-Log — blockiert aber nie (Spec-Prinzip „informieren statt blockieren"). (2) *App-Restart-Resume* (`try_resume_flight` + `ResumeFlightBanner`): prüft Flug-**Alter** + PIREP-**Server-Status**, aber **NICHT die Position**.
- Der `ResumeFlightBanner` zeigt einen 10-Sekunden-Countdown. Drückt der Pilot nichts, feuert nach Ablauf **automatisch** `flight_resume_confirm` (`ResumeFlightBanner.tsx:119`). Genau das ist der „direkte Autostart", den Michel meint.

→ **Antwort auf Thomas' Frage „ist das schon so?":** teils. Es gibt einen Banner *und* (im anderen Pfad) eine Drift-Messung — aber der App-Restart-Resume macht **keine** Positions-Prüfung und der Banner **bestätigt sich nach 10 s selbst**. Nach einem Sim-Crash + Reload (Flugzeug spawnt an einer Default-Boden-Position) adoptiert der Auto-Confirm diese Glitch-Position in den laufenden PIREP.

### LE14 — Auto-Confirm nur bei plausibler Position

Beim App-Restart-Resume vergleicht der Client den **ersten Post-Resume-Sim-Snapshot** mit dem persistierten Flugzustand (`PersistedFlight.stats.phase` + letzte bekannte Position). Die Position gilt als **verdächtig**, wenn mindestens eine Bedingung zutrifft:

1. **Air/Ground-Mismatch:** die persistierte Phase war airborne (Climb / Cruise / Descent / Approach / Final) **und** der aktuelle Snapshot meldet `on_ground` (bzw. AGL nahe 0). Das ist exakt Michels Fall — Crash im Reiseflug, Reload spawnt am Boden.
2. **Große laterale Drift:** Großkreis-Distanz persistierte letzte Position ↔ aktuelle Position über einer Schwelle.

**QS-R5-Fix (Thomas):** eine separate **Höhen-Sprung-Schwelle wurde gestrichen** — sie wäre zu restriktiv (ein Pilot, der die App im Sink-/Steigflug legitim neu startet, hätte fast immer ein Höhen-Delta und würde unnötig zum Pflicht-Klick gezwungen). Der Air/Ground-Mismatch (Bedingung 1) deckt den Crash-Fall sauber ab; ein reiner Höhen-Unterschied bei *gleichbleibend airborne* ist kein Glitch-Signal.

Verhalten:
- **Position plausibel** → 10-s-Auto-Confirm bleibt wie bisher (Pilot der nur die App neu startet, Sim lief durch, verliert nichts).
- **Position verdächtig** → der Banner **schaltet den 10-s-Auto-Confirm ab** und bleibt stehen, bis der Pilot **aktiv** „Flug fortsetzen" oder „Abbrechen" drückt. Zusätzlich ein erklärender Hinweis im Banner („Position weicht stark vom gespeicherten Stand ab — bitte prüfen, ob der Sim sauber geladen ist").

Umsetzung: `try_resume_flight` (bzw. der Snapshot-Pfad danach) setzt ein neues Flag, das `ActiveFlight`/der `was_just_resumed`-Payload ans Frontend durchreicht, z. B. `resume_position_suspect: bool`. Der `ResumeFlightBanner` deaktiviert seinen Countdown wenn das Flag gesetzt ist.

Kein Blockieren des Resume selbst — der Pilot kann weiterhin fortsetzen; es wird nur der **automatische** Confirm bei verdächtiger Position entfernt. Konsistent mit „informieren statt blockieren", nur eben mit Pflicht-Klick statt stillem Timer.

**Offene QS-Frage E-2:** laterale Drift-Schwelle (Bedingung 2) — den bestehenden `RESUME_DRIFT_WARN_NM` (50 NM) wiederverwenden, oder enger? Vorschlag: bestehende 50 NM wiederverwenden (eine Glitch-Reposition über 50 NM ist eindeutig). Der Air/Ground-Mismatch (Bedingung 1) ist ohnehin das schärfere, häufigere Signal für genau Michels Fall.
