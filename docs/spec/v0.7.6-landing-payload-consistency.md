**Status:** Draft for QS Review
**Stand:** 2026-05-10
**Scope-Owner:** Thomas K. (VA-Owner GSG)
**Code-Owner:** AeroACARS

> **Ziel:** Score, Payload, Web-Anzeige und Forensik sollen dieselbe Wahrheit anzeigen. Ein Datenvertrag, eine Web-Anzeige, eine Forensik-Wahrheit.

## 1. Problemraum & Motivation

QS-Sichtung von zwei realen v0.7.5-Pilot-Logs (SAS9987 EDDH→ENSB, GSG303 2OR3→OR66) hat drei Datenkonsistenz-Bugs aufgedeckt, bei denen Score, PIREP-Payload und Web-Dashboard sich gegenseitig widersprechen:

| Symptom | Was Pilot/Admin sieht | Was tatsächlich gilt |
|---|---|---|
| OFP-Treue 95 vs Burn +117 % | "Spritverbrauch katastrophal" | Trip-Burn war −5 %, Score korrekt |
| Bounce 1 (analysis) vs Bounce 0 (sub_score) | "war's ein Bounce oder nicht?" | Pipeline-interne Inkonsistenz |
| `runway_match_icao=K5S9` für `arr=OR66` | "Excellent stop, TD-Zone 1" | Match war 3.5 km daneben — Geometrie-Werte sind Müll |

Keiner dieser Bugs ist im Score-Algorithmus selbst — alle sitzen entweder im **Payload-Contract** zwischen Tauri-Client und Web-Dashboard, im **Web-Frontend**, oder in einer **fehlenden Sicherheitsnetz-Logik**. Das macht v0.7.6 zu einem reinen Konsistenz-Schnitt — kein Score-Umbau, kein neuer Sub-Score, kein FSM-Refactor.

## 3. Scope P1 (Hard Requirements)

### P1-1 — Fuel-Contract sauberziehen

**Tauri-Seite (CloudeAcars):**

PIREP-Payload-Contract erweitern um:

```rust
// build_pirep_fields in lib.rs
pub actual_trip_burn_kg: Option<f32>, // takeoff_fuel_kg - landing_fuel_kg
```

`takeoff_fuel_kg` ist bereits im PIREP-Payload (verifiziert in beiden v0.7.5-Logs) — kein Add nötig.

**Web-Seite (aeroacars-live):**

`PirepFeed.tsx:233-236` umstellen auf:

```ts
const actualBurn =
  pl.actual_trip_burn_kg
  ?? (pl.takeoff_fuel_kg != null && pl.landing_fuel_kg != null
       ? pl.takeoff_fuel_kg - pl.landing_fuel_kg
       : null)
  ?? null;  // Wenn nichts davon: Fuel-Vergleich AUSBLENDEN, nicht raw zeigen

const burnDelta = actualBurn != null && pl.planned_burn_kg != null
  ? actualBurn - pl.planned_burn_kg : null;
const burnPct = actualBurn != null && pl.planned_burn_kg
  ? Math.round((actualBurn / pl.planned_burn_kg - 1) * 100) : null;
```

`pl.fuel_used_kg` darf für den OFP-Burn-Vergleich **niemals direkt** genutzt werden. Wenn `actualBurn` `null` ist, wird die "Fuel verbraucht"-Zeile ausgeblendet, NICHT mit Raw-Wert gefüllt.

### P1-2 — Bounce-Quelle synchronisieren

**Status quo:**
- `landing_analysis.bounce_count` (forensische Detection)
- `touchdown_complete.payload.bounce_count` (final, in PIREP)
- `sub_scores[bounces]` (Score)

Bei SAS9987: forensik = 1, payload = 0, score = 100. Drei Quellen, zwei Wahrheiten.

**Layer-Verantwortung (wichtig — v1.1-Korrektur):**

Die Bounce-Threshold gehoert in die **Touchdown-/Forensik-Schicht** (`touchdown_v2`), nicht in die `landing-scoring`-Crate. Begruendung: nur die Forensik kennt den AGL-Verlauf, die Hopser-Hoehen und die Sample-Timestamps. Die Score-Crate sieht nur einen aggregierten `bounce_count` und `bounce_max_agl_ft` und darf darauf vertrauen koennen, dass dieser Wert bereits Score-relevant gefiltert ist.

**Entscheidung:** Zwei Konstanten in `touchdown_v2.rs`:

```rust
// touchdown_v2.rs (oder dedizierte forensik-konstanten datei)
pub const BOUNCE_FORENSIC_MIN_AGL_FT: f32 = 5.0;  // sichtbar im Forensik-Window
pub const BOUNCE_SCORED_MIN_AGL_FT:   f32 = 15.0; // bestraft im Sub-Score
```

Die Forensik-Schicht produziert dann pro Touchdown-Window:

| Feld | Definition | Wer liest es |
|---|---|---|
| `forensic_bounce_count` | Wiederabhebungen ≥ 5 ft (kleine Hopser sichtbar im Replay) | Web-Forensik-Tab, optional Pilot-Hint |
| `scored_bounce_count` | Wiederabhebungen ≥ 15 ft | `landing-scoring::sub_bounces`, PIREP-Payload, Sub-Score |
| `bounce_max_agl_ft` | hoechster Wiederabhebungs-Peak | Beide Layer (informativ) |

Die `landing-scoring::sub_bounces` bekommt **nur** den `scored_bounce_count` als Input, nicht den forensischen Count. Eine evtl. interne `bounce_count`-Konstante in der Score-Crate wird entfernt — die Crate vertraut dem Wert vom Caller.

**Threshold-Begruendung:**
- **5 ft** (forensic): tief genug, dass auch sanfte Federwerk-Reaktionen (gear oleo bounce) sichtbar sind ohne dass jeder Federwerk-Tick als Bounce gilt. AGL-Sensor-Rauschen liegt typisch bei 1-2 ft.
- **15 ft** (scored): hoch genug, dass ein "echter" Bounce (Pitch-Up nach harter Landung) erfasst wird, aber nicht jeder Federwerk-Hopser den Pilot bestraft. SAS9987 mit max AGL 13.6 ft faellt klar darunter → `scored_bounce_count = 0`, was zum heutigen Sub-Score `bounces=100 (clean)` passt.
- **Alternative falls Real-Daten zeigen dass 13.6 ft schon scoring-relevant sein soll:** scored auf 10 ft setzen. Empfehlung: erst v0.7.6 mit 15 ft ausliefern, dann 1-2 Wochen Pilot-Daten sammeln, falls zu viele "echte" Bounces durchrutschen → Patch auf 10 ft in v0.7.7.

**Akzeptanz SAS9987 (nach Fix mit 5/15-ft-Schema):**
- `forensic_bounce_count: 1` (13.6 ft Hopser sichtbar im Forensik-Tab/Replay)
- `scored_bounce_count: 0` (unter Penalty-Schwelle)
- `bounce_max_agl_ft: 13.57` (informativ)
- Sub-Score `bounces: 100 (clean)` — unveraendert
- Web-Card: kein Widerspruch zwischen forensischem Hint und Score

### P1-3 — Runway-Match Safety-Net

**Tauri-Seite (CloudeAcars), Pipeline in `lib.rs`:**

Konstanten in `lib.rs` (oder dedizierter `runway_match.rs`):

```rust
pub const RUNWAY_TRUST_MAX_CENTERLINE_OFFSET_M: f32 = 200.0; // fix
pub const RUNWAY_TRUST_MIN_FLOAT_DISTANCE_M:    f32 = -100.0; // fix
```

Nach dem Runway-Match einen Trust-Check:

```rust
let icao_ok = match runway_match_icao.as_deref() {
    Some(matched) => matched == arr_airport
        || Some(matched) == divert_to.as_deref(),
    None => false,
};
let offset_ok = centerline_offset_m
    .map(|o| o.abs() <= RUNWAY_TRUST_MAX_CENTERLINE_OFFSET_M)
    .unwrap_or(false);
let float_ok = landing_float_distance_m
    .map(|f| f >= RUNWAY_TRUST_MIN_FLOAT_DISTANCE_M)
    .unwrap_or(true); // missing → kein Negativ-Symptom

let runway_geometry_trusted = icao_ok && offset_ok && float_ok;

let runway_geometry_reason = if !runway_geometry_trusted {
    Some(if !icao_ok { "icao_mismatch" }
         else if !offset_ok { "centerline_offset_too_large" }
         else { "negative_float_distance" }.to_string())
} else { None };
```

**Begründung der Schwellen (v1.2):**
- **200 m Centerline-Offset:** Einfach, nachvollziehbar, robust. Runway-width-relativ wäre theoretisch sauberer, wird aber bei kleinen/alten DB-Einträgen schnell zu streng oder unklar.
- **−100 m Float-Distance:** Alles davor ist kein "Float" mehr sondern ein Geometrie-Problem. Runway-länge-relativ würde zu viel Spielraum geben (z.B. bei 4000-m-Pisten wäre −400 m noch trusted, was nicht plausibel ist).

PIREP-Payload bekommt zwei neue Felder:

```rust
pub runway_geometry_trusted: bool,
pub runway_geometry_reason: Option<String>,
```

**Welche Felder sind Runway-Geometrie-abhaengig, welche nicht?**

| Feld | Quelle | Trust-abhaengig? |
|---|---|---|
| `landing_touchdown_zone` | Threshold-Distanz aus Runway-DB | **JA** — wenn untrusted, auf `None` setzen |
| `landing_float_distance_m` | TD-Position - Threshold-Position | **JA** — wenn untrusted, ausblenden / als raw `threshold_offset_m` umlabeln |
| `runway_match_centerline_offset_m` | TD-Position - Runway-Centerline | **JA** — informativ, aber nicht im Score verwendet |
| `runway_length_m` | aus Runway-DB | **JA** — nur informativ |
| `rollout_distance_m` | **GPS-Rollstrecke** vom TD bis Stillstand | **NEIN** — kommt aus GPS-Track, unabhaengig von Runway-DB |
| `landing_yaw_rate_deg_per_sec` | aus Heading-Verlauf nach TD | **NEIN** — Sensor-basiert |

**Wenn `runway_geometry_trusted == false`:**

- `landing_touchdown_zone` wird im finalen Payload auf `None` gesetzt (Schema-Stabilitaet — Feld existiert weiterhin, ist nur leer)
- `landing_float_distance_m` bleibt im Payload (raw-Wert nicht hart geloescht — interne Diagnostik), wird aber im **Web komplett ausgeblendet**. Kein "Threshold-Offset (raw)"-Disclaimer-Display, weil das den Pilot mit einem kaputten Geometrie-Ergebnis konfrontiert das wir nicht erklaeren koennen.
- **`rollout_distance_m` bleibt unveraendert** und wird weiter vom `rollout`-Sub-Score genutzt — der Wert kommt aus dem GPS-Track (TD-Position bis Stillstand), nicht aus der Runway-Datenbank. **Rollout-Sub-Score bleibt also valide auch bei untrusted geometry.**

**Web-Seite:** Wenn `runway_geometry_trusted == false`, blendet die "Touchdown Zone"- und "Float Distance"-Anzeige aus, zeigt stattdessen einen kleinen Hinweis-Pill "⚠️ Runway-Match unsicher (`{reason}`)". Rollout-Tile bleibt sichtbar.

**Akzeptanz GSG303 (nach Fix):** `runway_geometry_trusted = false`, `reason = "icao_mismatch"`. Kein "TD Zone 1, 947 m"-Display für Geometrie die auf K5S9 statt OR66 basiert.

### P2-3 — Legacy `fuel_efficiency_pct` deprecaten

`touchdown_complete.payload.fuel_efficiency_pct` (−2.28 % bei SAS9987) ist ein **Altwert** der vom alten Score-Algorithmus übrig ist. Heute liefert die neue `sub_fuel_v0_7_1` einen anderen Wert (−5.0 %). Zwei Werte für dasselbe Konzept im selben Payload-Block — Quelle weiterer Verwirrung.

**Fix (sanfter Pfad — v1.1-Korrektur):**
- Feld bleibt im Payload (Backward-Compat fuer externe Consumer wie Discord-Embed-Scripts, Custom-Dashboards, evtl. phpVMS-Module die das Feld lesen).
- Web-Frontend rendert es **nicht mehr**.
- Spec-Dokumentation markiert das Feld als `@deprecated since v0.7.6 — use sub_scores[].fuel.value instead`.
- Eintrag in CHANGELOG unter "Deprecated" — Hinweis dass das Feld in einer **spaeteren Major-Version** entfernt wird (kein Termin in v0.7.6 spezifiziert).

**Single Source of Truth fuer Pilot/Admin:** `sub_scores[].fuel.value`.

## 9. Roll-out-Reihenfolge

**Entscheidung v1.2:** Alle drei Phasen werden als **ein gemeinsames v0.7.6-Paket** ausgeliefert. Kein Mini-Web-Release davor.

Wichtige Bedingung: **Web-Fallback-Kette muss backward-compatible** sein, damit alle bestehenden v0.7.5-PIREPs in der DB nach dem Web-Deploy korrekt angezeigt werden. Konkret:

```ts
// Web rendert KORREKT fuer:
//   - alte v0.7.5-PIREPs (ohne actual_trip_burn_kg)  → Fallback auf takeoff-landing
//   - neue v0.7.6-PIREPs (mit actual_trip_burn_kg)   → primaer-Feld
//   - Edge-Case ohne takeoff_fuel_kg (sehr alt?)     → Fuel-Vergleich ausgeblendet
const actualBurn =
  pl.actual_trip_burn_kg
  ?? (pl.takeoff_fuel_kg != null && pl.landing_fuel_kg != null
       ? pl.takeoff_fuel_kg - pl.landing_fuel_kg
       : null);
```

Genauso fuer `runway_geometry_trusted`:

```ts
// Wenn Feld fehlt (alte PIREPs) → wie heute behandeln (trust-implicit)
// Wenn Feld da (neue PIREPs)    → echten Trust-Status anwenden
const trusted = pl.runway_geometry_trusted ?? true;
```

Damit kann das Web vor, gleichzeitig oder nach dem Tauri-Release deployed werden — die Reihenfolge ist egal solange beide gleichzeitig in v0.7.6-Paket bezeichnet werden.

**Tag/Release-Schnitt fuer v0.7.6 enthaelt:**
- CloudeAcars: `actual_trip_burn_kg`, Bounce-Threshold-Konstanten, Runway-Trust-Check, Banner-Master-Score (P3, falls klein), Tests
- aeroacars-live: PirepFeed.tsx Fuel-Fallback, Render-Artefakt-Fix, Runway-Trust-Display, Legacy-Fuel-Render entfernt

## 10. Entscheidungs-Log

**In v1.1 entschieden:**
- ✓ **Bounce-Threshold-Layer:** Forensik-Schicht (touchdown_v2), nicht landing-scoring-Crate
- ✓ **Bounce-Threshold-Werte:** 5 ft forensic, 15 ft scored. SAS9987 (13.6 ft) → forensic visible, scored 0
- ✓ **`landing_touchdown_zone` bei untrusted geometry:** auf `None` setzen (Schema-Stabilitaet)
- ✓ **Rollout bei untrusted geometry:** bleibt bewertet (kommt aus GPS-Track, nicht aus Runway-DB)
- ✓ **Legacy `fuel_efficiency_pct`:** deprecaten, nicht hart loeschen

**In v1.2 entschieden (final-vor-Code):**
- ✓ **Centerline-Offset-Threshold:** **fix 200 m**. Einfach, nachvollziehbar, robust. Runway-width-relativ klingt sauberer wird aber bei kleinen/alten DB-Eintraegen schnell zu streng oder unklar.
- ✓ **Negative Float Distance Threshold:** **fix −100 m**. Alles davor ist kein "Float Distance" mehr sondern Threshold-/Geometry-Problem. Relativ zur Runway-Laenge wuerde wieder zu viel Spielraum geben.
- ✓ **`landing_float_distance_m` bei untrusted im Web:** **komplett ausblenden** statt als Raw-Wert zeigen. Sonst erklaeren wir Piloten ein kaputtes Geometrie-Ergebnis. Intern/logisch bleibt der Raw-Wert im Payload.
- ✓ **Phase A/B/C:** **Alles als ein v0.7.6-Paket**, kein Mini-Web-Release davor. Web-Fallback muss backward-compatible sein (siehe §9).
- ✓ **P3 Banner aus Master-Score:** **mit rein, wenn klein und klar**, weil es direkt zur Score-Konsistenz gehoert. Falls es mehr als ein kleiner UI-Patch wird (z.B. State-Management-Refactor): nach v0.7.7 schieben. Entscheidungs-Trigger: wenn nach 1 Stunde Code-Sichtung absehbar ist dass es > 1 Datei beruehrt → splitten.

**Damit ist die Spec final — ready for code.**
