## 1. Problemraum & Motivation

QS-Sichtung von zwei realen v0.7.5-Pilot-Logs (SAS9987 EDDH→ENSB, GSG303 2OR3→OR66) hat drei Datenkonsistenz-Bugs aufgedeckt, bei denen Score, PIREP-Payload und Web-Dashboard sich gegenseitig widersprechen:

| Symptom | Was Pilot/Admin sieht | Was tatsächlich gilt |
|---|---|---|
| OFP-Treue 95 vs Burn +117 % | "Spritverbrauch katastrophal" | Trip-Burn war −5 %, Score korrekt |
| Bounce 1 (analysis) vs Bounce 0 (sub_score) | "war's ein Bounce oder nicht?" | Pipeline-interne Inkonsistenz |
| `runway_match_icao=K5S9` für `arr=OR66` | "Excellent stop, TD-Zone 1" | Match war 3.5 km daneben — Geometrie-Werte sind Müll |

Keiner dieser Bugs ist im Score-Algorithmus selbst — alle sitzen entweder im **Payload-Contract** zwischen Tauri-Client und Web-Dashboard, im **Web-Frontend**, oder in einer **fehlenden Sicherheitsnetz-Logik**. Das macht v0.7.6 zu einem reinen Konsistenz-Schnitt — kein Score-Umbau, kein neuer Sub-Score, kein FSM-Refactor.

## 3. Scope P1 (Hard Requirements)

### P1-1 — Fuel-Contract sauberziehen

**Tauri-Seite (CloudeAcars):**

PIREP-Payload-Contract erweitern um:

```rust
// build_pirep_fields in lib.rs
pub actual_trip_burn_kg: Option<f32>, // takeoff_fuel_kg - landing_fuel_kg
```

`fuel_used_kg` bleibt im Payload (Backward-Compat), wird aber explizit als **raw/sim-cumulative** in der Spec markiert — nicht für OFP-Vergleich.

`takeoff_fuel_kg` ist bereits im PIREP-Payload (verifiziert in beiden v0.7.5-Logs) — kein Add nötig.

**Web-Seite (aeroacars-live):**

`PirepFeed.tsx:233-236` umstellen auf:

```ts
const actualBurn =
  pl.actual_trip_burn_kg
  ?? (pl.takeoff_fuel_kg != null && pl.landing_fuel_kg != null
       ? pl.takeoff_fuel_kg - pl.landing_fuel_kg
       : null)
  ?? null;  // Wenn nichts davon: Fuel-Vergleich AUSBLENDEN, nicht raw zeigen

const burnDelta = actualBurn != null && pl.planned_burn_kg != null
  ? actualBurn - pl.planned_burn_kg : null;
const burnPct = actualBurn != null && pl.planned_burn_kg
  ? Math.round((actualBurn / pl.planned_burn_kg - 1) * 100) : null;
```

`pl.fuel_used_kg` darf für den OFP-Burn-Vergleich **niemals direkt** genutzt werden. Wenn `actualBurn` `null` ist, wird die "Fuel verbraucht"-Zeile ausgeblendet, NICHT mit Raw-Wert gefüllt.

**Akzeptanz SAS9987 (nach Fix):**

| Wert | Vorher | Nachher |
|---|---|---|
| Fuel verbraucht | **19.98 t / 9.23 t (+117 %)** | **8.76 t / 9.23 t (−5 %)** |
| OFP-Treue Sub-Score | 95 | 95 (unverändert) |
| Konsistenz-Check | ❌ Widerspruch | ✓ |

### P1-2 — Bounce-Quelle synchronisieren

**Status quo:**
- `landing_analysis.bounce_count` (forensische Detection)
- `touchdown_complete.payload.bounce_count` (final, in PIREP)
- `sub_scores[bounces]` (Score)

Bei SAS9987: forensik = 1, payload = 0, score = 100. Drei Quellen, zwei Wahrheiten.

**Entscheidung:** Eine **gemeinsame Threshold-Konstante** `BOUNCE_SCORED_MIN_AGL_FT` einführen, in der `landing-scoring`-Crate definiert. Alle drei Quellen leiten ihren Wert davon ab:

```rust
pub const BOUNCE_SCORED_MIN_AGL_FT: f32 = 5.0; // Vorschlag — final TBD durch QS
```

- `landing_analysis.bounce_count` = ALLE detektierten Wiederabhebungen (incl. < 5 ft) — als "forensic_bounce_count" umbenannt damit die Trennung klar ist
- `landing_analysis.scored_bounce_count` (NEU) = Wiederabhebungen ≥ Threshold
- `payload.bounce_count` und `sub_scores[bounces].value` lesen `scored_bounce_count`

**Alternative falls QS gegen Umbenennung:** Eine einzige Threshold-Logik in der Forensik anwenden, dann zählt nur was scoring-relevant ist.

**Akzeptanz SAS9987 (nach Fix):** Entweder
- `forensic_bounce_count: 1, scored_bounce_count: 0` (transparent getrennt), oder
- `bounce_count: 0` überall (wenn Threshold ≥ 13.6 ft).

Kein Widerspruch zwischen den drei Stellen, gleiche Wahrheit gegenüber dem Pilot.

### P1-3 — Runway-Match Safety-Net

**Tauri-Seite (CloudeAcars), Pipeline in `lib.rs`:**

Nach dem Runway-Match einen Trust-Check:

```rust
let runway_geometry_trusted = match (runway_match_icao, arr_airport, divert_to) {
    (Some(matched), arr, div) => {
        let icao_ok = matched == arr || Some(matched.as_str()) == div.as_deref();
        let offset_ok = centerline_offset_m.abs() <= 200.0;
        let float_ok = landing_float_distance_m.is_none()
                        || landing_float_distance_m.unwrap() >= -100.0;
        icao_ok && offset_ok && float_ok
    }
    _ => false,
};

let runway_geometry_reason = if !runway_geometry_trusted {
    Some(if !icao_ok { "icao_mismatch" }
         else if !offset_ok { "centerline_offset_too_large" }
         else { "negative_float_distance" })
} else { None };
```

PIREP-Payload bekommt zwei neue Felder:

```rust
pub runway_geometry_trusted: bool,
pub runway_geometry_reason: Option<String>,
```

**Wenn `runway_geometry_trusted == false`:**

- `landing_touchdown_zone` wird auf `None` gesetzt im finalen Payload (oder bleibt aber wird vom Score ignoriert — TBD welche Variante sauberer ist)
- `landing_float_distance_m` wird umbenannt zu `landing_threshold_offset_m` ODER bleibt aber Web zeigt es nicht als "Float Distance"
- `rollout_distance_m` bleibt (wird aus GS+Bremsweg berechnet, nicht aus Runway-Geometrie) → `rollout`-Sub-Score bleibt valide

**Web-Seite:** Wenn `runway_geometry_trusted == false`, blendet die "Touchdown Zone"- und "Float Distance"-Anzeige aus, zeigt stattdessen einen kleinen Hinweis-Pill "⚠️ Runway-Match unsicher (`{reason}`)".

**Akzeptanz GSG303 (nach Fix):** `runway_geometry_trusted = false`, `reason = "icao_mismatch"`. Kein "TD Zone 1, 947 m"-Display für Geometrie die auf K5S9 statt OR66 basiert.

## 4. Scope P2 (Polish)

### P2-1 — Web Render-Artefakte fixen

`PirepFeed.tsx:398, 401`:

```tsx
// VORHER (rendert "0" wenn count === 0)
{pl.go_around_count && pl.go_around_count > 0 && (...)}
{pl.touchdown_count && pl.touchdown_count > 1 && (...)}

// NACHHER
{(pl.go_around_count ?? 0) > 0 && (...)}
{(pl.touchdown_count ?? 0) > 1 && (...)}
```

Komplette Datei nach demselben Anti-Pattern durchsuchen — vermutlich gibt's noch mehr Stellen mit `{value && <Tile>}`-Pattern.

**Akzeptanz:** Keine streunenden `0`/`00` in den PIREP-Cards.

### P2-2 — Web-Label "Fuel verbraucht" konsistent

Wenn `actualBurn` aus `actual_trip_burn_kg` oder Diff-Fallback kommt → Label "Fuel verbraucht" passt.

Wenn nur `pl.fuel_used_kg` (Raw) verfügbar wäre → Vergleich-Zeile ausblenden statt Label "Fuel raw" einführen. Pilot sieht im Zweifel keine Fuel-Bilanz, statt eine falsche.

### P2-3 — Legacy `fuel_efficiency_pct` aus Payload entfernen

`touchdown_complete.payload.fuel_efficiency_pct` (−2.28 % bei SAS9987) ist ein **Altwert** der vom alten Score-Algorithmus übrig ist. Heute liefert die neue `sub_fuel_v0_7_1` einen anderen Wert (−5.0 %). Zwei Werte für dasselbe Konzept im selben Payload-Block — Quelle weiterer Verwirrung.

**Fix:** `fuel_efficiency_pct` aus dem Payload entfernen. Wenn jemand den Wert braucht: `sub_scores[].fuel.value` ist die Single Source.

## 5. Scope P3 (Optional, kann auch v0.7.7 werden)

### P3-1 — Pilot-Client Banner aus Master-Score

Im Tauri-Client zeigt das LandingPanel-Banner heute "Severe" / "Hard" / "Acceptable" / "Smooth" basierend auf der Legacy `LandingScore::numeric()` (nur VS+G).

Bei SAS9987 ist Master-Score 67 → Banner sollte z.B. "Borderline" oder "Hard but acceptable" sein, nicht "Severe".

**Vorschlag:**
- Banner-Tone aus Master-Score-Bands ableiten (≥ 85 = ok, ≥ 65 = good, ≥ 45 = neutral, ≥ 25 = warn, sonst = err — gleiche Logik wie `PirepFeed.scoreTone`)
- Touchdown-VS-Klasse separat klein darunter ("Touchdown −680 fpm" + Color-Coding), aber nicht als Headline-Banner

**Web-Dashboard ist davon NICHT betroffen** — zeigt nur Score-Pill, kein Banner-Wording.

## 9. Roll-out-Reihenfolge

Da die Web-Fixes (P1-1 Web-Teil + P2-1 + P2-2) **sofort** ohne Tauri-Release deploy-bar sind, schlage ich vor:

1. **Phase A (sofort, kein Pilot-Update nötig):** aeroacars-live deployen mit Web-Fallback-Logik (`takeoff_fuel_kg − landing_fuel_kg`) und Render-Artefakt-Fix. Pilot sieht sofort korrekte Fuel-Anzeige für ALLE bestehenden v0.7.5-PIREPs.
2. **Phase B (v0.7.6 Tauri-Release):** Payload erweitert um `actual_trip_burn_kg` + Bounce-Sync + Runway-Trust-Check. Web kennt das neue Feld und nutzt es bevorzugt. Backward-kompatibel zu Phase-A-Fallback.
3. **Phase C (v0.7.6 Web-Update nach Tauri-Release):** Web zeigt Runway-Trust-Pill, blendet untrusted Geometry aus.

Phase A und B können auch zusammenfallen, wenn der User das gemeinsam will — keine harte Abhängigkeit.
