## Ausgangslage (Code-Stand)

`client/src-tauri/crates/sim-xplane/src/dataref.rs` hat eine **statische**
`DATAREFS`-Tabelle: feste Liste `(dataref-name, FieldId)`, die der Adapter per
UDP-RREF abonniert. Es gibt **keinen** per-Flugzeug-Mechanismus. Der Adapter
liefert bereits `aircraft_title` im Snapshot (GSG225: `"Challenger 650 published
by X-Aviation"`) und `aircraft_icao` (`"CL60"`).

## Leitentscheidungen

### LE1 — Aircraft-Erkennung: Titel-Match **+ Probe** (QS-R0-P1-Fix)

R0 wollte nur über `aircraft_title` matchen. **Problem (QS):** `aircraft_title`
ist kein RREF-Wert — er kommt nur als Overlay aus der X-Plane-Web-API
(`adapter.rs`, `snapshot()`). Ist die Web-API aus (Pilot nicht aktiviert) oder
läuft X-Plane 11, bleibt der Titel `None` — das CL650-Profil würde **nie**
greifen, obwohl die CL650-DataRefs existieren.

R1 — **zweistufige Erkennung**, ein Profil gilt als aktiv wenn **eine** Stufe
zieht:

1. **Titel-Match** (bevorzugt, wenn `aircraft_title` vorhanden): der Titel
   enthält case-insensitiv **alle** `title_match`-Substrings. Für die CL650:
   `"Challenger 650"` **und** `"X-Aviation"` (so matcht der Default-X-Plane-
   Challenger nicht mit). `aircraft_icao` (`"CL60"`) ist **kein** Match-Schlüssel
   — Default + Hot-Start teilen den ICAO.
2. **Probe** (robust, funktioniert ohne Web-API / unter XP11): der Adapter
   abonniert zusätzlich den **Signatur-DataRef** des Profils (`probe_dataref`).
   Liefert X-Plane dafür innerhalb weniger Ticks einen Wert, **existiert** der
   DataRef → das Flugzeug ist das Profil-Flugzeug → Profil aktiv. Ein nicht
   existierender DataRef bleibt im RREF-Stream einfach stumm.

Damit greift das CL650-Profil auch dann, wenn der Titel fehlt.

**RREF-Index-Regel für die Probe (QS-R2-P1-Fix).** Der Adapter routet
Subscriptions heute strikt index-basiert: `send_subscriptions()` schickt den
`CATALOG`, `XPlaneState::apply(index, value)` mappt `index → CATALOG[index]`.
Damit die Probe da nicht hineinpfuscht, gilt verbindlich:

- **Reservierter Discovery-Index-Bereich.** Probe-DataRefs bekommen Indizes
  **oberhalb** des regulären Katalogs (Vorschlag: ab `DISCOVERY_INDEX_BASE`,
  ein Index pro bekanntem Profil). Diese Range überschneidet sich nie mit
  Katalog-Indizes.
- **Probe-Pakete werden VOR `XPlaneState::apply()` abgefangen.** Ein RREF-Paket
  mit einem Index aus dem Discovery-Bereich geht an die Profil-Erkennung, **nicht**
  an `apply()`. Es **mutiert keinen Snapshot-Wert** — die Probe stellt nur
  Existenz fest („Profil P hat geantwortet").
- **Nach Profil-Aktivierung: sauberer Neuaufbau.** Sobald ein Profil aktiv ist
  (Titel **oder** Probe), baut der Adapter den **aktiven Katalog** (LE6) und
  **abonniert komplett neu** — frische, lückenlose Indizes über den aktiven
  Katalog. Die Discovery-Subscriptions werden dabei verworfen.
- Anfangs (vor jeder Erkennung) läuft der Adapter mit dem statischen `CATALOG`
  plus den Discovery-Probes — der Snapshot ist sofort voll funktionsfähig.

### LE2 — Profil-Datenmodell (statisch im Code)

Ein Aircraft-Profil ist eine statische Struktur (analog zur `DATAREFS`-Tabelle —
Pre-1.0 kein config-driven Profil-File):

```rust
struct XplaneAircraftProfile {
    /// Substrings, die ALLE im aircraft_title vorkommen müssen (case-insensitiv).
    /// Stufe 1 der Erkennung.
    title_match: &'static [&'static str],
    /// Signatur-DataRef für die Probe-Erkennung (Stufe 2). Existiert er im
    /// laufenden Flugzeug, gilt das Profil als zutreffend. Üblich: der
    /// score-relevanteste Override-DataRef des Profils.
    probe_dataref: &'static str,
    /// DataRef-Overrides: ersetzen den Standard-Eintrag des jeweiligen FieldId.
    overrides: &'static [DatarefOverride],
}

struct DatarefOverride {
    field: FieldId,
    dataref: &'static str,
    /// Wie der Rohwert auf die interne Repräsentation abgebildet wird.
    mapping: ValueMapping,
}

enum ValueMapping {
    /// 1:1 übernehmen (z.B. int 0/1 Lights).
    Passthrough,
    /// Detent-Index → interner f32-Wert über eine Lookup-Tabelle.
    /// Out-of-range-Verhalten: siehe LE7.
    DetentTable(&'static [f32]),
}
```

Eine `PROFILES`-Liste hält alle bekannten Profile. Beim Aircraft-Wechsel sucht
der Adapter das erste passende Profil (LE1).

### LE4 — Flaps-Detent → interner Wert (A-2 doku-verifiziert)

`FLAPS_LVR` ist ein **Hebel-Detent `int 0–3`**, **kein** Deploy-Ratio. AeroACARS'
internes `flaps_position` ist `f32 0.0–1.0`; die LANDING-CONFIG-Prüfung
(`compute_approach_stability_v2`) nutzt `flaps_position >= 0.70` als „Landing-
Config gesetzt".

**A-2 aufgelöst (R2) — Quelle: Hot-Start-CL650-Dokumentation.** Die CL650 hat
**vier Flap-Stufen: 0 / 20 / 30 / 45 Grad** — belegt durch die Boolean-Wires
`FLAP_IND/0/wires/{20_DEG,30_DEG,45_DEG}` und den FP32-Wire
`FLAP_IND/0/wires/ANGLE` (echter Flap-Winkel in Grad). R0-Annahme `0/1/20/30`
ist damit endgültig verworfen; eine manuelle Detent-Erfassung ist nicht mehr
nötig.

> **Quellen-Nachweis (QS-R2-P3).** Verifiziert im lokalen Hot-Start-Doku-Bundle
> `Documentation.zip` (von Thomas, 2026-05-20): Datei `Documentation/Integration/
> Wires.txt`, Modul-Abschnitt `FLAP_IND:` mit den Wires `20_DEG` / `30_DEG` /
> `45_DEG` (`INT32 boolean`) und `ANGLE` (`FP32`). Hot Start liefert die
> CL650-Dokumentation im Aircraft-Verzeichnis aus (öffentlich bestätigt:
> X-Pilot-Forum „CL650 Included Documentation"); eine frei verlinkbare Online-
> Kopie der `Wires.txt` gibt es nicht — das Bundle liegt der Code-QS bei.

| `FLAPS_LVR` | CL650-Flaps | `flaps_position` | Landing-Config? |
|---|---|---|---|
| 0 | 0° | 0.00 | nein |
| 1 | 20° | 0.45 | nein |
| 2 | 30° | 0.80 | **ja** (≥ 0.70) |
| 3 | 45° | 1.00 | **ja** |

**Finale `DetentTable`: `[0.0, 0.45, 0.80, 1.0]`.** Hebel 0/1 (Flaps 0/20)
bleiben < 0.70, Hebel 2/3 (Flaps 30/45) ≥ 0.70. So bleibt die `>= 0.70`-Logik
in `compute_approach_stability_v2` **unverändert** — das Profil liefert nur
einen passend skalierten Wert.

**Landing-Config-Regel (A-2b, VA-Policy):** Default — als Lande-Konfiguration
zählt **Flaps ≥ 30** (Hebel ≥ 2). Flaps 20 zählt **nicht** (im CL650 eine
Takeoff-/Approach-Stufe; normale Landungen mit 30 oder 45). Will GSG Flaps 20
explizit als zulässige Landing-Config akzeptieren, wird der Tabellenwert für
Hebel 1 auf ≥ 0.70 angehoben — bewusste, hier zu dokumentierende Entscheidung.
Die `CL650 Operations Reference.pdf` aus dem Doku-Bundle ist die fachliche
Referenz dafür.

**Alternative DataRef (notiert, nicht für v0.12.2 gewählt):**
`abus/CL650/modules/FLAP_IND/0/wires/ANGLE` liefert den **echten Flap-Winkel
in Grad** (FP32, kontinuierlich). Sauberer als der Hebel, bräuchte aber ein
Grad-Mapping statt der `DetentTable`. v0.12.2 nutzt `FLAPS_LVR` — von Michel
bereits live im DataRefTool erfasst, und mit der doku-verifizierten
Detent-Tabelle eindeutig. `ANGLE` bleibt als spätere Verfeinerung offen.

### LE5 — Fallback ohne Profil unverändert

Flugzeuge ohne passendes Profil laufen exakt wie heute: Standard-`DATAREFS`-
Tabelle, und falls die Flaps-DataRef dort `0` bleibt, greift die v0.12.1-LANDING-
CONFIG-fail-soft-Behandlung. Kein Verhaltensbruch für Nicht-Profil-Flugzeuge.

### LE6 — Aktiver Katalog + Re-Subscription (QS-R0-P2-Fix)

**QS-Befund:** der X-Plane-Katalog ist heute statisch und **index-basiert** —
`apply(index, value)` schlägt `CATALOG[index].field` nach (`dataref.rs`), und
`seen`/`last_values` sind auf die `CATALOG`-Länge dimensioniert. R0 ließ offen,
wie ein Override da hineinkommt.

R1 — präzise Regel: **dynamischer aktiver Katalog**.

1. Der **statische `DATAREFS`** bleibt die Basis (Index-stabil).
2. Bei Aircraft-Erkennung (LE1) wird ein **aktiver Katalog** gebaut: Kopie der
   Basis, in der jeder Profil-Override den Basiseintrag mit **demselben
   `FieldId`** **ersetzt** (gleicher `FieldId`, neuer DataRef-Name + `mapping`).
   Ein Override fügt **keinen** neuen `FieldId` hinzu — er ersetzt nur die
   DataRef-Quelle eines bestehenden Feldes. (Falls ein Profil künftig ein Feld
   abdeckt, das die Basis nicht hat, wird es ans Ende angehängt — saubere
   Index-Vergabe deckt das ab.)
3. Die RREF-Indizes werden über den **aktiven Katalog** vergeben; `seen` und
   `last_values` werden auf dessen Länge (re)dimensioniert.
4. `apply(index, value)` schlägt im **aktiven** Katalog nach und wendet dessen
   `ValueMapping` an.
5. **Aircraft-Wechsel zur Laufzeit:** Adapter wertet die Erkennung neu aus,
   baut den aktiven Katalog neu, dimensioniert `seen`/`last_values` neu und
   abonniert mit **frischen Indizes** neu. Beim Sim-(Re)connect ohnehin Teil
   des Subscription-Setups.

Damit gibt es keine doppelten oder fehl-gemappten RREF-Indizes; jeder Index
zeigt eindeutig auf genau einen aktiven Katalogeintrag.

### LE7 — `DetentTable`: Out-of-range-Verhalten (QS-R0-P2-Fix)

Der RREF-Rohwert ist ein `f32`. Beim `DetentTable`-Mapping:

- Ist der Wert **nicht finite** (`NaN`/`Inf`) → Override liefert **keinen** Wert;
  das Feld behält seinen Standard-/`None`-Zustand. `tracing::warn!` einmalig.
- **Sauberer-Detent-Check (QS-R2-P2-Fix):** `FLAPS_LVR` ist ein dokumentierter
  Integer-Detent. Der Rohwert muss **nahe an einer Ganzzahl** liegen:
  `|raw − raw.round()| ≤ DETENT_TOLERANCE` (Vorschlag: `0.05`). Sonst — z.B.
  `1.51` — liefert der Override **keinen** Wert (`tracing::debug!`); ein
  unsauberer Zwischenwert wird **nicht** auf einen Detent gesnappt.
- Erst dann `idx = raw.round() as i32`. Liegt `idx` **außerhalb** `0..table.len()`
  (z.B. `-1`, `4`) → ebenfalls **kein** Wert, `tracing::debug!`.
- Nur ein finiter, *ganzzahl-naher*, im Tabellenbereich liegender Index wird
  gemappt.

**Kein Panic, kein stilles `1.0`-Default.** Liefert der Override keinen Wert,
greift für die Landing-Config die v0.12.1-fail-soft-Behandlung (LE5).
