# v0.9.0 — Telemetry Contract (Daten- und Eventvertrag)

**Status:** ✅ **FROZEN** — Spec accepted für Step 0 in QS-Round-6 vom 2026-05-18. Nachfolgende Änderungen erfordern QS-Re-Review.
**Freeze-Datum:** 2026-05-18 (QS-Round-6)
**Datum:** 2026-05-18 (Initial PROPOSAL), 2026-05-18 (FROZEN)
**Quelle:** QS-Round-2-Befund („P0: Gemeinsamer Telemetry Contract fehlt als zentrales Kapitel"), 5 Iterationen bis QS-Round-6-Acceptance
**Ziel:** Bevor Delta-Sampling, Per-Bounce, Expanded-Phases, Discord-RPC und GlitchTip nebeneinander her bauen, definiert dieses Dokument die EINE Wahrheit für Telemetrie-Felder, Events, Ownership, Persistenz und Backward-Compat. Alle 5 Detail-Specs sind Konsumenten dieses Contracts — Konflikte zwischen ihnen werden hier zuerst aufgelöst.

## Kurzfassung

Die fünf v0.9.0-Features greifen alle auf die gleiche Datenbasis:

- **Position-Samples** (= jeder Tick im Sim) → von Delta-Sampling reduziert, von Expanded-Phases gelesen, von Discord-RPC für State-Updates konsumiert
- **Phase-Transitions** → von Expanded-Phases neu erweitert, von Discord-RPC angezeigt, von GlitchTip als Tag, von Per-Bounce als Timing-Anker
- **Touchdown-Window** → von Per-Bounce mit Detail-Records angereichert, von Expanded-Phases als Trigger (LANDED → DEBOARDING), von GlitchTip als Forensik-Event
- **PIREP-Ausgabe** → von allen 5 Features beeinflusst (Notes, Stats, neue Felder)

Ohne gemeinsamen Contract entstehen Bugs der Klasse „Feature A erwartet Phase=APPROACH bei 8000ft, Feature B emittet erst bei 5000ft" oder „Per-Bounce zählt 3, Discord-RPC zeigt 1". Dieses Dokument ist die Schiedsrichter-Instanz.

Out of Scope: konkrete UI-Mockups (gehören in die Detail-Specs), Performance-Optimierungen, Wire-Format-Schema-Versionierung (= eigener Pfad bei Major-Releases).

## Verbindlichkeit

Dieses Spec ist **vorrangig** vor den 5 Detail-Specs. Bei Konflikt zwischen Detail-Spec und Telemetry-Contract gilt der Contract.

Änderungen am Contract erfordern Update aller 5 betroffenen Detail-Specs (im selben PR). Detail-Specs dürfen NICHT eigenständig Felder umbenennen, Ownership verschieben oder Persistenz-Pfade ändern.

## Glossar

| Begriff | Bedeutung |
|---|---|
| **Telemetrie-Feld** | Ein benannter Wert in einem Position-/Phase-/Touchdown-/PIREP-Payload (z.B. `vs_fpm`, `phase`, `bounce_max_agl_ft`) |
| **Event** | Eine MQTT-Message auf einem `aeroacars/{va}/{pilot}/{channel}`-Topic ODER ein interner Aktivity-Log-Eintrag |
| **Owner** | Die Code-Komponente die einen Feld-Wert berechnet und schreibt (= „Single Writer Principle") |
| **Konsument** | Code-Komponente die einen Feld-Wert liest aber nicht schreibt |
| **Kanonisch** | Der EINE autoritative Wert für ein Konzept (z.B. canonical vs_at_impact statt fünf verschiedene smoothed-Werte) |
| **Live-only** | Nur im aktuellen Flug relevant, nicht persistiert |
| **Persistiert** | In SQLite/PIREP-JSON gespeichert, nach Flug-Ende noch verfügbar |
| **Backward-Compat** | Alte PIREPs / alte Clients laufen weiter ohne neue Felder |

## 3. Ownership-Matrix (Single-Writer-Principle)

Pro Telemetrie-Feld GENAU ein Schreiber. Mehrere Konsumenten erlaubt. Verstöße sind Bugs.

```
Sim-Adapter (snap.*) ─────────► position.{ts, lat, lon, alt_ft, agl_ft, pitch_deg, bank_deg,
                                          hdg_*, ias_kt, gs_kt, tas_kt, vs_fpm, g_force,
                                          on_ground, parking_brake, gear_position,
                                          flaps_position, engines_running, fuel_*, wind_*,
                                          oat_c, qnh_hpa, ap_*, simulator}

FlightMeta (= AppState) ──────► position.{callsign, aircraft_icao, aircraft_registration,
                                          aircraft_title, dep, arr, pirep_id, client_version}

StateMachine (FlightStats) ───► position.phase
                                phase.{ts, phase}
                                pirep.{rto_event, go_around_events}      [NEU v0.9.0]
                                Activity-Log

Touchdown-Detector ───────────► touchdown.{ts, vs_fpm, ias_kt, pitch_deg, bank_deg, g_load,
                                            peak_g_load, score, bounce, bounce_count,
                                            forensic_bounce_count, scored_bounce_count,
                                            bounce_max_agl_ft, bounces,                [NEU]
                                            airport, runway, forensics_version}
                                pirep.bounces                                          [NEU]

Streamer-Tick ────────────────► (sendet nur — schreibt keine eigenen Felder)
                                Implementierung dieser Spec sagt: kein eigener Wert,
                                nur Forward der Sim/Meta/State-Werte mit Filter.

Discord-RPC-Manager ──────────► NICHTS außer Discord-API. Liest Position/Phase/Meta, schreibt
                                keine AeroACARS-Felder.

Sentry-SDK ───────────────────► NICHTS außer Sentry-API. Liest Activity-Log/Exceptions, schreibt
                                keine AeroACARS-Felder.
```

**Konflikt-Auflösung wenn zwei Specs den gleichen Owner beanspruchen:** Sieht so aus dass Per-Bounce „Touchdown-Detector ist Owner von `bounces`" und Expanded-Phases „nichts an Touchdown ändern". Konflikt-frei. Wenn ein Detail-Spec versucht ein bereits-vergebenes Feld zu schreiben (z.B. Discord-RPC will `phase` selbst setzen), wird das im Telemetry-Contract abgelehnt und der Detail-Spec muss angepasst werden.

## 4. Persistenz-Pfade

| Daten | Live-only | persistiert in | Retention |
|---|---|---|---|
| Position-Stream | nein | Recorder `positions`-Tabelle + Pilot-Client JSONL | 30 Tage (Recorder) / Flug-Ende (Client) |
| Phase-Transitions | nein | Recorder `flight_events`-Tabelle (kind='phase') + JSONL | unbegrenzt (DB) / Flug-Ende (Client) |
| Touchdown-Records | nein | Recorder `touchdowns`-Tabelle (payload_json voll) + Pilot-Client `LandingRecord` (landing_history.json) + phpVMS-PIREP | unbegrenzt |
| `bounces` (NEU) | nein | innerhalb `touchdowns.payload_json` (= JSON-Blob im Recorder, KEIN eigenes Schema-Feld); im `LandingRecord` als top-level Feld; im phpVMS-PIREP-`fields` | unbegrenzt |
| `rto_event`, `go_around_events` (NEU) | nein | innerhalb `pireps.payload_json` (= JSON-Blob im Recorder); im phpVMS-PIREP-Notes UND `fields` | unbegrenzt |
| Discord-Activity-State | **ja** | nirgendwo | nur RAM während Flug |
| Sentry-Events | nein | GlitchTip-Postgres | 30 Tage (siehe GlitchTip-Spec) |

**Regel:** Discord-RPC ist live-only — wenn Pilot-Client crasht, ist Discord-Status weg. Kein State-File. Vereinfacht Recovery.

## 5. Backward-Compat-Matrix

Für jeden Konsumenten-Pfad muss klar sein: was passiert wenn ein Feld fehlt oder ein alter String reinkommt?

| Konsument | Feld | v0.8.x-Verhalten | v0.9.0-Verhalten | Compat-Strategie |
|---|---|---|---|---|
| Pilot-Client LandingPanel | `bounces` | n/a | zeigt Detail-Modal | `record.bounces ?? []` — leerer Array bei alten PIREPs |
| Webapp Live-Map | `phase=REJECTED_TAKE_OFF` | unbekannt → showed-as-string | rendert als gelbe Pill mit i18n-Label | Fallback i18n: `t(\`phase.${value}\`, { defaultValue: value })` |
| Webapp Live-Map | `phase=GO_AROUND` | dito | dito | dito |
| Webapp Live-Map | `phase=DEBOARDING` | dito | dito | dito |
| Recorder mqttSubscriber | `phase=REJECTED_TAKE_OFF` | wird in `flight_events.payload_json` gespeichert | unverändert (TEXT-Spalte) | kein Schema-Change nötig |
| phpVMS PIREP-Konsument | `bounces`-Array im Body | unbekannt → ignoriert | wird als custom_field gespeichert wenn phpVMS-Custom-Field konfiguriert ist | bei alten phpVMS: Feld wird stillschweigend verworfen, kein Error |
| Discord-RPC | `phase=$NEW_UNKNOWN_STRING` | n/a | zeigt Phase-Name 1:1 als State | unbekannte Phasen werden als String angezeigt, kein Crash |
| Old Pilot-Client v0.8.x (vom Recorder gesehen) | published `bounces`=undefined | normal | normal | Recorder akzeptiert weiter alte Clients ohne `bounces` |

**Pflicht-Regel — „Kein Konsument darf bei fehlendem Feld crashen":**

- Alle TypeScript-Konsumenten: `?? []`, `?? null`, `?? defaultValue` für jede neue Spalte
- Alle Rust-Konsumenten: `Option<T>` mit `#[serde(default)]` oder `#[serde(skip_serializing_if = "Option::is_none")]`
- Alle SQL-Queries: `COALESCE(field, default)` wenn Spalte hinzukommt
- Alle Tests: ein Legacy-Fixture ohne neue Felder pro Detail-Spec

## 6. Verbindliche Bau-Reihenfolge (Dependency-DAG)

Wenn alles in v0.9.0 muss, ist die technische Reihenfolge erzwungen:

```
                      ┌─────────────────────┐
                      │  Telemetry Contract │  ← DIESES Dokument zuerst freezed
                      │  (= dieses Dokument)│
                      └──────────┬──────────┘
                                 │ definiert Felder/Owner/Events
                                 ▼
              ┌──────────────────┴──────────────────┐
              │                                     │
              ▼                                     ▼
  ┌────────────────────────┐         ┌─────────────────────────┐
  │  Step 1:               │         │  Step 0 (parallel):     │
  │  Delta-Position-       │         │  GlitchTip VPS-Setup    │
  │  Sampling              │         │  (Infra, keine Code-Dep)│
  │                        │         └────────────┬────────────┘
  │  → published events    │                      │
  │    bleiben kompatibel  │                      │ DSN-Env-Var ready
  │    aber seltener       │                      │
  └───────────┬────────────┘                      │
              │ Phase-Transition zwingt force-publish │
              ▼                                       │
  ┌────────────────────────┐                          │
  │  Step 2:               │                          │
  │  Expanded Flight       │                          │
  │  Phases (RTO, GO_AR,   │                          │
  │  DEBOARDING)           │                          │
  │                        │                          │
  │  → 3 neue phase strs   │                          │
  │  → PIREP rto/go_around │                          │
  └───────────┬────────────┘                          │
              │ Touchdown bleibt eigener Pfad         │
              ▼                                       │
  ┌────────────────────────┐                          │
  │  Step 3:               │                          │
  │  Per-Bounce JSON-      │                          │
  │  Detail-Records        │                          │
  │                        │                          │
  │  → bounces[] in pirep  │                          │
  └───────────┬────────────┘                          │
              │                                       │
              ▼                                       ▼
  ┌──────────────────────────────────────────────────────────┐
  │  Step 4: GlitchTip SDK-Integration (Client + Recorder    │
  │  + Webapp) — bekommt jetzt Tags aus den oben definierten │
  │  kanonischen Feldern (callsign, aircraft, route, sim,    │
  │  pilot_hash). Setzt voraus dass Step 2 + 3 Feld-Owner   │
  │  geklärt haben.                                          │

  │  Meta. Setzt voraus dass alle drei vom Contract konstant │
  │  sind. KEIN Schreib-Recht auf Telemetrie.                │
  └───────────────┬──────────────────────────────────────────┘

                  ▼
  ┌──────────────────────────────────────────────────────────┐
  │  Step 6 (zwingend): Gesamt-QS mit E2E-Golden-Flights     │
  │  (siehe Sektion 8 unten).                                │
  └──────────────────────────────────────────────────────────┘
```

**Begründung jeder Kante:**

- **Contract → Step 1 (Delta-Sampling):** Delta-Logik muss den Phase-Trigger kennen → Phase ist Owner StateMachine → Contract sagt wie Force-Publish funktioniert
- **Step 1 → Step 2 (Phases):** Neue Phase-Werte (RTO, GO_AROUND, DEBOARDING) brauchen Delta-Force-Publish bei Phase-Transition → Step 1 muss stehen damit Step 2 darauf bauen kann
- **Step 2 → Step 3 (Per-Bounce):** Per-Bounce schreibt `bounces` ins Touchdown-Payload. Expanded-Phases setzt LANDED + DEBOARDING basierend auf Touchdown-Edge. Step 3 hängt nicht direkt von Step 2 ab, aber wir bauen sequentiell damit das Touchdown-Window getestet werden kann (1 Bug pro Sprint statt mehrere zusammen)
- **Steps 1-3 → Step 4 (GlitchTip):** Sentry-Tags brauchen die kanonischen Felder. Wenn Step 2 + 3 noch shippen, könnten Tags inkonsistent werden
- **Step 4 → Step 5 (Discord-RPC):** Discord-RPC braucht funktionierende Phase-Strings (Step 2) und ggf. Bounce-Counter (Step 3) für State-Text. Step 4 davor weil Bug-Tracking funktionieren muss bevor Discord öffentlich shippt
- **Step 6 (Integrationstest):** kann erst NACH Step 5 weil sonst die Test-Matrix unvollständig ist

**Was passieren kann wenn die Reihenfolge missachtet wird:**

- Wenn Delta-Sampling NACH Expanded-Phases gebaut wird: könnte sein dass neue Phase-Transitions nicht sofort published werden → Webapp/Discord-RPC verzögert
- Wenn Per-Bounce VOR Touchdown-Stabilität (= bestehende v0.7.6/v0.8.3-Fixes) eingreift: könnte `bounces`-Array nicht mit `bounce_count` konsistent sein
- Wenn GlitchTip VOR Phase-Strings festgezogen: Sentry-Tags zeigen alte Phase-Werte → Trend-Analyse verfälscht

## 7. Feature-Flags / Rollout-Gates

Jedes der 5 Features braucht einen Flag damit es bei Bedarf einzeln aus-/eingeschaltet werden kann ohne Re-Release.

| Feature | Flag-Name | Default v0.9.0 | Ort | Aus-Schalter-Wirkung |
|---|---|---|---|---|
| **Delta-Position-Sampling** | `position_sampling.delta_enabled` | `true` | `config.json` (Pilot-Client) | Fallback auf v0.8.x Fixed-Rate (5-8s Tick) |
| **Expanded Phases (RTO)** | `phases.rejected_take_off_enabled` | `true` | `config.json` (Pilot-Client) | Pilot sieht weiterhin „TAKEOFF" auch bei RTO |
| **Expanded Phases (GO_AROUND)** | `phases.go_around_enabled` | `true` | `config.json` | Pilot sieht weiterhin „FINAL"/"APPROACH" |
| **Expanded Phases (DEBOARDING)** | `phases.deboarding_enabled` | `true` | `config.json` | Pilot sieht weiterhin „ARRIVED" |
| **Per-Bounce JSON-Detail** | `forensics.per_bounce_enabled` | `true` | `config.json` | `bounces`-Array wird nicht emittiert, nur Counter (= v0.8.x-Verhalten) |
| **GlitchTip Error-Reporting** | `error_reporting.enabled` | **`false`** bis Pilot Opt-In | `config.json` | KEIN Sentry-Init, keine Events gesendet |
| **Discord Rich Presence** | `integrations.discord_rpc.enabled` | **`false`** | `config.json` | Discord-Manager nicht initialisiert |
| **Per-Pilot Hash für GlitchTip** | `error_reporting.anonymize_pilot_id` | `true` | `config.json` | Wenn false: pilot_id ungehasht (NIE empfohlen, nur für lokale Tests) |

**Regeln:**
- Defaults sind verbindlich (= Opt-out für Features die der Pilot „kostenlos" bekommt, Opt-in für Features mit Privacy-Implikation)
- Settings-UI im Pilot-Client zeigt ALLE Flags (inkl. Default-Erklärung)
- Recorder hat KEINEN Flag für diese Features (= reagiert immer auf das was Client schickt)
- VPS-Webapp hat KEINEN Flag (= zeigt alle Daten die der Recorder hat)

**Settings-UI-Sektion:** Alle Flags landen in der bestehenden Settings-Panel-Sektion „Erweitert" mit Pflicht-Erklärung pro Toggle.

### 7.1 Disable-Verhalten (Kill-Switch-Semantik pro Flag)

Pro Flag definiert die folgende Tabelle GENAU was beim Wechsel von `true` → `false` passiert. Pflicht für G10 (E2E-Golden-Flight „Alle neuen Features deaktiviert"). Details siehe [`v0.9.0-feature-flags-semantics.md`](v0.9.0-feature-flags-semantics.md), hier nur kompakter Vertrag pro Disable-Aspekt:

| Flag | Laufende Tasks | Bereits erzeugte Daten | UI sofort | Persistierte Daten | Latenz Wirksamkeit |
|---|---|---|---|---|---|
| **`position_sampling.delta_enabled`** | Streamer-Tick läuft weiter (keine Tasks zu stoppen) | bleiben in `positions`-Tabelle | nein (nur neue Position-Records betroffen) | bleiben | nächster Tick (~5-8s) |
| **`phases.rejected_take_off_enabled`** | Phase-Detector-Task läuft weiter | bestehende PIREPs mit RTO-Event bleiben | bei aktivem Flug: Phase „rollt zurück" zu TAKEOFF wenn RTO gerade aktiv; sonst kein UI-Wechsel | bleiben (alte PIREPs zeigen weiter RTO-Event) | nächster Phase-Check-Tick (~1s) |
| **`phases.go_around_enabled`** | dito | dito | dito (rollt zu APPROACH/FINAL zurück) | dito | dito |
| **`phases.deboarding_enabled`** | dito | dito | dito (rollt zu ARRIVED zurück) | dito | dito |
| **`forensics.per_bounce_enabled`** | Touchdown-Detector läuft weiter, baut bounces[] aber publisht es nicht | `bounces`-Arrays in alten PIREPs bleiben | LandingPanel zeigt für NEUE PIREPs nur Counter, für ALTE weiterhin Detail-Modal | bleiben (alte PIREPs unverändert) | nächster Touchdown |
| **`error_reporting.enabled`** | Sentry-SDK wird **graceful shutdown** (= laufende HTTP-POSTs an GlitchTip werden gecancelt, NICHT geflusht) | KEINE bereits gesendeten Events werden gelöscht (= existieren weiter auf GlitchTip-Server, normale Retention 30d gilt) | sofort: kein neues Event mehr | bleiben auf GlitchTip; lokaler Sentry-Buffer wird VERWORFEN (NICHT bei Re-Enable nachgesendet) | sofort (binnen 1s, siehe DS7) |
| **`integrations.discord_rpc.enabled`** | DiscordPresenceManager `clear_activity()` + `disconnect()` werden aufgerufen | n/a (= live-only) | sofort: Discord-Profil zeigt KEIN AeroACARS mehr (Pilot-Sichtprüfung: Activity verschwindet binnen 5s, siehe DS8) | n/a | binnen 5s (siehe DS8) |
| **`error_reporting.anonymize_pilot_id`** | Nur Hash-Funktion-State wechselt; SDK läuft weiter | bereits gesendete Events bleiben mit ihrem alten Hash/Klartext-Wert auf GlitchTip-Server | nächster Event nutzt neuen Modus | bleiben (kein nachträgliches Re-Hashing möglich, weil Events bereits gesendet) | nächster Event |

**Kritische Notes:**
- **`error_reporting.enabled = false` löscht NICHT bereits gesendete Events.** Pilot kann existierende Events nur via Server-side Admin-Tool aus GlitchTip löschen lassen (= „Recht auf Vergessenwerden" via Support-Request an den VA-Admin).
- **`forensics.per_bounce_enabled = false` versteckt NICHT die UI für alte PIREPs.** Wenn Pilot später bewusst alte PIREPs ansieht, sieht er die `bounces[]`-Details. Wenn er das auch nicht will, muss er die PIREPs serverseitig löschen.
- **`integrations.discord_rpc.enabled = false` ist Discord-clear-pflicht.** Toggle-Aus ohne Discord-Clear ist ein Privacy-Bug (Activity bleibt sichtbar) und wird via DS8 hart getestet.
- **Settings-UI muss bei jedem Disable-Toggle eine kurze Bestätigung zeigen** wenn die Aktion bereits-erzeugte Daten betrifft: „Diese Einstellung wird auf zukünftige Flüge angewendet. Bereits erfasste Daten bleiben erhalten."

**Anti-Pattern (= darf NICHT passieren):**
- Sentry-Buffer wird bei Re-Enable nicht „nachgeflusht" (= verworfene Events sind verloren, nicht heimlich versendet)
- Discord-RPC clearet nicht (= bleibt 60s sichtbar bis nächster Heartbeat das Cleared-Stage anwendet)
- Per-Bounce-Disable löscht bestehende `bounces`-Arrays (= retention-Risiko, kein Pilot-Wunsch)

### 9.1 Technische Snapshots (Redaction-Tests)

QS-Round-3-Befund P2: DS1-DS8 ist eine richtige Checkliste, aber QS-tauglich erst mit konkreten Negativ-Beispielen. Für jeden Datenkanal definieren wir hier (a) was im Payload PFLICHT-VERBOTEN ist (= Negativ-Liste), (b) ein realer Snapshot wie ein gesäubertes Event aussieht, (c) ein realer Snapshot wie ein SCHMUTZIGES Event aussehen würde (= was wir gerade NICHT senden wollen — zur Negativ-Validierung).

#### DS1.1 — Sentry-Event-Negativliste (gilt für DS1)

In KEINEM Sentry/GlitchTip-Event (= weder `exception` noch `message` noch `breadcrumb`) darf folgendes vorkommen — egal ob als Top-Level-Feld, als Tag, als Extra, als Body-String, als Header, als Stack-Frame-Argument:

| Verbotenes Datum | Wie es im AeroACARS-Code vorkommt | Sanitization-Strategie |
|---|---|---|
| Email-Adresse | `email: String` in Login-Response | `before_send`: `event.user.email = None` |
| phpVMS-API-Token | `Authorization: Bearer <token>` Header in ApiClient | `before_send`: regex-replace `Bearer\s+\S+` → `Bearer [REDACTED]` in allen String-Feldern |
| phpVMS-Cookie | `Cookie: phpvms_session=...` | `before_send`: strip `Cookie`-Header |
| Plain-Text-Passwort | nirgendwo gespeichert (Tauri-Keyring), aber wenn doch | regex `password["']?\s*[:=]\s*\S+` → `password=[REDACTED]` |
| Pilot-ID-im-Klartext (`pilot_id: "23"`) | Übertragen in Position-Payload, könnte in Tags landen | Wenn `error_reporting.anonymize_pilot_id=true`: SHA256(salt+id) → ersten 8 Hex-Zeichen |
| Pilot-Email | Aus phpVMS-API-Profile | NIE in Tags/User-Object (`user.email = None`) |
| Pilot-Name | dito | NIE in Tags/User-Object (`user.username = None`) |
| PIREP-Notes-Inhalt | Pilot-Schreibtext, kann persönliches enthalten | NIE als Tag/Extra. Stack-Trace zeigt nur Field-Name, nie Wert. |
| Lokale Datei-Pfade mit User-Verzeichnis | `C:\Users\Thomas\AppData\...` in Stack-Frames | Sentry-SDK strippt automatisch ab `AppData/Roaming/` (Standard-Verhalten) |
| Lokale IP-Adressen | `192.168.1.5`, `10.0.0.1` etc. | `before_send`: regex-replace IP-Pattern in allen Strings → `[LOCAL_IP]` |
| Externe IP-Adressen | Pilot-Public-IP | Sentry-Standard: `send_default_pii: False` (= IP wird beim Empfang verworfen) |
| Lat/Lon im Klartext | Position-Payload | NIE als Tag — nur `airport` (ICAO-Code) + `runway` als Geo-Tag. Lat/Lon können in Stack-Frame-Args landen wenn ein Helper-Function-Aufruf failet — `before_send` strippt Float-Tupel-Patterns aus Frame-Variablen |
| MQTT-Broker-URL | `wss://live.kant.ovh/mqtt` | OK (= public, nicht sensibel) |
| Discord-User-ID | NICHT gespeichert (Discord-RPC zeigt nur Local-State) | n/a |

## 11. Versionierungs-Strategie für neue Felder

Für `bounces`, `rto_event`, `go_around_events` und alle weiteren v0.9.0-Felder:

| Aspekt | Regel |
|---|---|
| **Serialization** | Rust: `#[serde(skip_serializing_if = "Option::is_none")]`. TypeScript: `field?: T` mit `??`-Fallbacks |
| **Schema-Marker** | Jede Payload hat `forensics_version` (bestehend) ODER `schema_version` Feld. v0.9.0 bumpt nicht — die neuen Felder sind additiv, kein Bruch |
| **Defensive Parsing** | Webapp/Recorder/phpVMS-Custom-Fields: unbekannte Felder werden GENULLED, nicht Crash |
| **Spätere Migration** | Wenn v1.0.0 doch ein Schema-Bruch braucht, kann `forensics_version=3` als Diskriminator dienen. v0.9.0-Felder bleiben deserialisierbar via `#[serde(default)]` |
| **Export-Kompat** | PIREP-Notes-Format bleibt Plain-Text. Custom-Fields nur in phpVMS-`fields` (= optional, ignorable) |

**Pflicht:** Jede neue Feld-Einführung muss in der Detail-Spec einen „Versionierung"-Abschnitt haben mit konkreten Beispielen wie alte und neue Payloads koexistieren.
