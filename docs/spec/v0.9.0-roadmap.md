# v0.9.0 — Roadmap: Skalierung, Observability, Forensik-Tiefe

**Status:** PROPOSAL — bereit für Implementation nach v0.8.4
**Datum:** 2026-05-18
**Quelle:** Stratos-Analyse (Skyvex Software 2.14.0) + GSG-Pilot-Feedback Q2-2026 (Adrian Bounce-Report, Michel Wetlease, Svenny PIREP-Spam)
**Ziel:** AeroACARS für den 10× Pilot-Wachstum vorbereiten — Bandwidth/Storage halbiert, Bug-Sichtbarkeit automatisiert, Forensik so reich wie kommerzielle Konkurrenz (Stratos), Pilot-UX-Polish (Discord-Presence, präzise Phasen). 5 unabhängige Features, parallel implementierbar, keine Schema-Brüche.

## Kurzfassung

v0.9.0 ist **kein Bug-Fix-Release** wie v0.8.3/v0.8.4 sondern ein **Skalierungs- und Observability-Release**. Fünf Features kommen aus zwei Quellen:

1. **Code-Vergleich mit Stratos** (kommerziellen ACARS-Konkurrent, 110 MB Electron-App, ~24 Mo am Markt) hat drei Architektur-Ideen aufgedeckt die wir 1:1 adoptieren können (Delta-Sampling, Pro-Bounce-Details, granularere Phasen).
2. **Eigene Operations-Pain-Points** der letzten Wochen (Adrian/Svenny/Michel-Bug-Triage via Discord-Screenshots) zeigen dass wir eine zentrale Bug-Triage-Lösung brauchen statt der ad-hoc-Discord-Forensik. Plus eine Quality-of-Life-Verbesserung (Discord Rich Presence) die Pilots-Community-Sichtbarkeit steigert.

## Problem (Gesamt-Sicht)

### Operations-Realität nach 18.05.2026

Wir haben heute (Stand v0.8.4):
- **12 Pilots** auf GSG (German Sky Group)
- ~**20-30 Flüge/Tag** im Durchschnitt
- ~**70k Position-Events/Tag** in `positions`-Tabelle (132 MB DB nach 11 Tagen)
- Bug-Reports kommen **ausschließlich via Discord** (Screenshots, freier Text)
- Bug-Diagnose dauert **1-3h pro Report** (= Discord → Repro raten → VPS-DB-Query → Code-Lesen → Hypothese → Pilot fragen)

Wenn die Pilot-Anzahl auf 30 wächst (= GSG kontaktiert weitere VAs / wachsendes Interesse nach v0.8.x Release):
- **210k Position-Events/Tag** → DB wächst auf 400+ MB/Monat
- **60-90 Bug-Reports/Monat** → Discord-Triage skaliert nicht
- **2× MQTT-Bandwidth** auf der VPS → mosquitto-Last steigt

### Forensik-Lücken aufgedeckt durch Stratos-Analyse

Stratos (Skyvex) macht zwei Sachen die wir aktuell nicht haben:

1. **Pro-Bounce-Detail-Records** (`{bounceNumber, ts, maxAltitudeAgl, vs, gForce, durationMs}` × N). Wir aggregieren nur (`bounce_count`, `bounce_max_agl_ft`). Bei Adrians #167-Bug (14ft-Hopser, nicht erkannt) hätten Detail-Records direkt geholfen — Pilot sieht „Bounce 1: 14 ft / -116 fpm / 0.3s" statt nur „0".

2. **Delta-getriggertes Position-Sampling**. Stratos speichert Position nur bei signifikanten Änderungen (>5 kt GS, >250 ft Alt, >250 fpm VS, >3° Heading, >1° Bank), plus 5s-Heartbeat. Bei 4h Cruise speichert das ~30 Samples statt unserer ~2500. = -98% Position-Daten bei langweiligen Phasen, 0% Qualitätsverlust bei interessanten.

### Pilot-Erlebnis-Lücken

- **Kein Discord-Status**: Andere Pilots sehen nicht „Thomas fliegt gerade GSG3184 EDDB→KMRH FL360". Stratos hat das, ist beliebt bei VA-Communities.
- **Keine RTO/GO_AROUND-Phase**: Pilots die abbrechen (Rejected Take-Off) oder durchstarten (Go-Around) sehen ihre Aktion nicht im PIREP. Stratos hat 17 Phasen, wir ~12.

## Leitentscheidungen

### 1. v0.9.0 hat NULL Schema-Brüche

Alle 5 Features sind additiv. Bestehende `flight_sessions`, `positions`, `touchdowns`, `flight_session_stats` werden NICHT umgebaut. Neue Felder werden mit `#[serde(default)]` / `serde(skip_serializing_if = "Option::is_none")` hinzugefügt. Alte Pilot-Clients laufen weiter; alte PIREPs bleiben lesbar.

**Begründung:** Wir hatten zwei Bug-Releases in einer Woche (v0.8.3 + v0.8.4). Pilots wollen Stabilität, nicht weitere Migrations-Risiken.

### 2. Features sind **unabhängig deploybar**

Jedes der 5 Features kann **einzeln** released werden (= eigener Tag möglich: v0.9.0 für #1, v0.9.1 für #2, etc.). Aber bevorzugt: alle 5 in einem v0.9.0-Bundle damit die bilingualen Release-Notes einmalig groß ausfallen.

**Begründung:** Reduziert Coupling-Risiko, erlaubt Rollback eines einzelnen Features ohne den Rest.

### 3. Server-Side Features ZUERST, Client-Side Features DANACH

Reihenfolge der 5 Items berücksichtigt:
- **#1 Delta-Sampling** = Client-Refactor, KEIN Server-Change nötig (server akzeptiert was kommt)
- **#2 GlitchTip** = primär VPS-Infra, kleiner Client-SDK-Call dazu
- **#3 Pro-Bounce** = Client-Forensik + Server-Schema-Add + Webapp-UI
- **#4 Discord-RPC** = pure Client-Feature
- **#5 Mehr Phasen** = Client-FSM-Refactor + Server-Recorder akzeptiert neue Strings

Ergibt Roll-Out-Plan:

```
Woche 1:  #2 GlitchTip (Infra) + #1 Delta-Sampling (Client) → v0.9.0
Woche 2:  #4 Discord-RPC (Client) + #5 Phasen (Client) → v0.9.1
Woche 3:  #3 Pro-Bounce (Client + Server) → v0.9.2
```

Oder alles auf einmal als monster-v0.9.0 (= 1 Release, weniger Overhead, mehr Risiko in einem Sprung).

### 4. GlitchTip kommt SELF-HOSTED, nicht als Cloud-Service

Wir hosten GlitchTip selber auf live.kant.ovh (= 5 docker-Container, ~500 MB RAM Footprint). DSGVO-konform (alle Daten bei uns), keine $30/Mo Cloud-Kosten, Sentry-SDK-kompatibel (= Code-Migration zu echtem Sentry später trivial).

**Begründung:** Hetzner-VPS hat 3.4 GB RAM frei. Self-hosted spart €360/Jahr. DSGVO-Pilot-Information ist einfacher wenn alle Daten in der EU bleiben.

### 5. Delta-Sampling ist NUR Pilot-Client-Side, kein Recorder-Change

Der Recorder akzeptiert weiterhin **jede** Position die reinkommt — er entscheidet nicht ob Pilot-Client „zu wenig" oder „zu viel" sendet. Damit funktionieren alte Clients (die fixed-rate senden) und neue Clients (die delta-sampled senden) gleichzeitig auf demselben Recorder. Beim allmählichen Rollout über die Auto-Updater entsteht kein Server-seitiger Migrations-Druck.

**Begründung:** Maximale Backward-Compat. Recorder wird nicht für Client-Optimierungen umgebaut.

### 6. Discord-Rich-Presence ist OPT-IN, nicht Default

Erste v0.9.x-Implementation hat einen Settings-Toggle (Default: AUS). Pilots aktivieren bewusst. Privacy-by-Default.

**Begründung:** Manche Pilots wollen nicht dass „GSG-Pilot X fliegt heimlich Cargolux 3217 GLA→HKG" auf ihrem Discord öffentlich sichtbar ist. Opt-in respektiert das.

### 7. Pro-Bounce-Details werden in `flight_session_stats.bounces TEXT` (JSON-Array) gespeichert, NICHT in separater Tabelle

Stratos macht's so, und es ist die richtige Wahl: Bounces gehören zur Landung (1:1 mit Session). Separater Tabellen-Lookup wäre Overhead ohne Mehrwert.

**Begründung:** SQLite JSON-Operatoren (`json_extract`, `json_each`) erlauben Queries auf das Array. Webapp kann direkt rendern. Migrations-frei (= add column statt new table).

### 8. Mehr Phasen — nur PFLICHT-Erweiterungen, kein FSM-Rebuild

Wir fügen **3 neue Phasen** hinzu: `REJECTED_TAKE_OFF`, `GO_AROUND`, `DEBOARDING`. Wir refaktorieren NICHT die bestehende FSM. Phase-Transitions für die neuen Werte sind klar definiert (siehe Spec #5), bestehende Phasen funktionieren unverändert.

**Begründung:** Stratos hat 17 Phasen, wir kommen auf 15. Diff ist verschmerzbar. FSM-Rebuild würde Bugs bringen ohne Mehrwert.

## Abhängigkeiten + Risiken

### Inter-Feature-Abhängigkeiten

```
#1 Delta-Sampling     → keine Abhängigkeiten (pure Client)
#2 GlitchTip          → keine Abhängigkeiten (Infra + 5min SDK-Call)
#3 Pro-Bounce         → benötigt v0.8.3 forensic_bounce_count (✓ schon da)
#4 Discord-RPC        → keine Abhängigkeiten
#5 Mehr Phasen        → keine Abhängigkeiten
```

→ **Alle 5 Features sind unabhängig.** Reihenfolge frei wählbar.

### Risiken

**Risiko R1 — Delta-Sampling: Webapp-Live-Map zeigt „Pilot-Sprünge"**
- Wenn Pilot 4h cruiset und Position nur alle 5s gepublisht wird (= Heartbeat), zeigt Webapp ihn statisch
- Aber: jede Bewegung (5 kt GS-Change reicht) triggert sofort Update
- Bei langem Cruise: Pilot bewegt sich physikalisch trotz „statischer Position-Übertragung" weiter — Webapp muss Position interpolieren oder akzeptieren dass Position alle 5-30s aktualisiert
- **Mitigation:** Heartbeat-Schwelle auf 3s statt 5s wenn Pilots „statisch" beklagen

**Risiko R2 — GlitchTip: Pilots-DSGVO-Anforderung**
- Pilots müssen informiert werden dass ihre Crash/Error-Daten gesammelt werden
- Eintrag in DSGVO-Hinweis im Login-Screen / Settings
- Self-hosted = alle Daten bei uns auf EU-VPS = einfach
- **Mitigation:** Einmalige Banner-Info beim ersten Start nach v0.9.0-Update

**Risiko R3 — Discord-RPC: Discord-API ist zickig auf Linux/Mac**
- Discord-RPC funktioniert via Unix-Socket (Linux/Mac) oder Named Pipe (Windows)
- Wenn Discord nicht läuft → no-op (kein Crash)
- Wenn Discord läuft aber Pipe-Verbindung scheitert → Stratos retried periodisch
- **Mitigation:** Settings-Status zeigt „verbunden / nicht verbunden", Pilot kann selbst sehen

**Risiko R4 — Mehr Phasen: alte Web-Pireps zeigen unbekannte Phasen-Strings**
- Webapp / Recorder kennen aktuell nur die ~12 Phasen
- Neue Phasen (`REJECTED_TAKE_OFF`, `GO_AROUND`, `DEBOARDING`) müssen Webapp-i18n bekommen
- Recorder akzeptiert jeden String — kein Schema-Issue
- **Mitigation:** i18n-Strings im gleichen v0.9.0-Release pushen für DE/EN/IT

- **OTA-Updates für Renderer/Core** (Stratos-Pattern) — sicherheits-konflikt mit Tauri-Modell
- **OAuth-Login für VA-Auth** — braucht phpVMS-Plugin
- **Multi-Sim-Erweiterung** auf P3D/FSX (Stratos hat alle 4 Sims) — Sim-Adapter-Aufwand zu hoch für jetzt
- **Webapp-Tab für Pro-Bounce-Visualisierung** (= 3D-Bounce-Trajektorie) — UX-Refactor, eigenes Ticket
- **Echte Sim-Adapter via node-simconnect** (Stratos macht's so, wir bleiben bei Rust-Crates) — Architektur-Wahl gegen Tauri
- **Eigener Cloud-Service** (Stratos-Pattern: skyvexsoftware.com) — Konflikt mit VA-Hoheit-Modell
