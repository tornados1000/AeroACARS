## Audit-Methodik

Für jede der 5 Detail-Specs werden 7 Dimensionen geprüft:

1. **Felder-Owner** (Contract Sektion 1, 3) — schreibt die Spec ein Feld dessen Owner laut Contract eine andere Komponente ist?
2. **Phase-Werte** (Contract Sektion 1.3) — referenziert die Spec Phase-Strings die nicht in der 18-Werte-Liste sind?
3. **Persistenz-Pfade** (Contract Sektion 4) — spezifiziert die Spec einen Persistenz-Pfad der vom Contract abweicht (z.B. eigene DB-Tabelle statt JSON-Blob)?
4. **Backward-Compat** (Contract Sektion 5) — fehlt der Spec eine `serde(default)` / `?? null`-Strategie für neue Felder?
5. **Feature-Flag** (Contract Sektion 7) — Flag-Name korrekt? Default korrekt? Kill-Switch erwähnt?
6. **Bau-Reihenfolge** (Contract Sektion 6) — postuliert die Spec eine Reihenfolge die mit dem DAG kollidiert?
7. **E2E-Tests** (Contract Sektion 8) — referenziert die Spec G1-G10 oder definiert sie eigene unabhängige Tests die G1-G10 nicht abdecken?

Jede Detail-Spec bekommt eine Audit-Tabelle mit:
- **Aussage** — was die Spec sagt
- **Stelle** — Datei + Sektion oder Zeile
- **Contract-Referenz** — welche Contract-Sektion ist betroffen
- **Verdict** — ✓ OK / ⚠ Klarstellung nötig / ✗ Konflikt zu korrigieren
