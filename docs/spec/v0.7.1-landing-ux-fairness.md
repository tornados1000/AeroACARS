## 0. Warum dieses Dokument

v0.7.0 hat die Landerate-Berechnung strukturell saniert (4-Layer touchdown_v2, HARD GUARDS, Multi-TD, sim-spezifische Validation). Im laufenden Pilot-Feedback gibt es jetzt **9 zusammenhaengende UX-/Fairness-Punkte** gleicher Wurzel:

1. **VFR-Piloten werden vom Startflow blockiert oder bekommen unfaire Scores**, weil ZFW/Fuel-Plan implizit als IFR-OFP-Vollstaendigkeit angenommen werden
2. **Sparsame Piloten werden bestraft**, weil Fuel-Score absolute Abweichung vom Plan misst statt Mehrverbrauch
3. **Die Forensik-v2-Verbesserungen sind im UI unsichtbar** — Pilot fragt weiter "warum diese Landerate?"
4. **Stability-Bewertung wirkt willkuerlich**, weil das bewertete Fenster im Chart nicht markiert ist und der Flare optisch in derselben Zone liegt
5. **Begriffe wie "Spritverbrauch" sind ambivalent** (Effizienz vs. Planabweichung) und Client/Web zeigen teils unterschiedliche Labels

**Befund:** alle 9 Punkte gehoeren in EIN Release-Paket — sie loesen zusammen das gleiche uebergeordnete Problem ("Piloten verstehen und akzeptieren die Bewertung"). Einzeln aufgeschnuert wuerden sie sich gegenseitig widersprechen (z.B. Stability-v2-Werte anzeigen ohne Gate-Markierung verschlimmert die Konfusion).

**Kein Touchdown-Core-Refactor.** v0.7.0 bleibt unangetastet. v0.7.1 ist:
- 70% Frontend (LandingPanel, PhaseTimeline, Tooltips, i18n)
- 20% Scoring-Logik (Fuel-Score-Asymmetrie, Stability-Sub-Score Erweiterung)
- 10% Backend-Datenfluss (forensics_version + confidence durchreichen, Flare-Felder in PIREP-Payload)

## 3. Architektur-Entscheidungen

### 3.1 Score-Algorithmus wird nach Rust portiert (Single Source of Truth)

**P1.1-B Fix gegen v1.0:** v1.0 sagte "scoring bleibt clientseitig in TS, Backend befuellt sub_scores aus computeSubScores()". Das geht NICHT — Rust kann die TS-Funktion nicht aufrufen, der PirepPayload wird in Rust gebaut.

**v1.1 Entscheidung:** Score-Algorithmus wird in einer **neuen Rust-Crate** `client/src-tauri/crates/landing-scoring/` portiert (1:1-Port von `landingScoring.ts`). Die Crate ist die einzige Stelle die Schwellen, Asymmetrie und Voting kennt.

Konsumenten:
- **Backend (`lib.rs`):** ruft `landing_scoring::compute_sub_scores(&record)` direkt — schreibt Ergebnis in PirepPayload.sub_scores
- **Tauri-Frontend (Client UI):** nutzt entweder den vom Backend berechneten Wert (PirepPayload-Roundtrip), oder ruft die Crate via `wasm-bindgen` direkt aus React (Live-Score waehrend Flug ohne MQTT-Latenz)
- **aeroacars-live (webapp + monitor):** liest `sub_scores` direkt aus PirepPayload — KEINE Re-Berechnung

**Folge:** `landingScoring.ts` wird durch eine duenne Wrapper-Datei ersetzt die nur noch den WASM-Bindings ruft. Tests bleiben in TS, plus neue Rust-Tests in der Crate (Goldenset gegen 6 Replay-Fixtures wie touchdown_v2_replay.rs).

**Migration:**
1. Phase 1 erstellt Crate + portiert subFuel/subStability/subTouchdown/subBounces/subGForce/subRollout
2. Tests in Rust: golden test-vectors aus aktuellem TS-Verhalten generieren, Rust matched bit-identisch
3. Backend benutzt Crate zum Befuellen sub_scores
4. Phase 3 ersetzt `landingScoring.ts` Body durch WASM-Wrapper

Begruendung gegen Alternative "Web/Monitor rechnet selbst":
- Drift-Risiko zu hoch (3 Implementierungen muessten synchron bleiben)
- Web hat nicht alle Roh-Felder (z.B. AGL-Buffer fuer Stability-Window)
- v0.7.0-Lehre: Single-Source-of-Truth ist nicht-verhandelbar

### 3.2 Score-Asymmetrie wird im Sub-Score-Algorithmus codiert, nicht via Override

`subFuel()` bekommt direkt eine asymmetrische Schwelle. KEIN nachgelagerter Override ("wenn negativ → 100"). Begruendung: Override-Pattern war v0.6.x bei `vs_at_edge` die Wurzel des Bugs.

### 3.3 Confidence ist Anzeige-only, nicht Score-Bestandteil

Confidence (High/Medium/Low/VeryLow) ist v0.7.1 NICHT scoring-relevant. Sie wird angezeigt damit Piloten verstehen wie sicher die Messung ist. Score-Mapping (z.B. "VeryLow → Score auf 50% reduzieren") waere v0.7.2-Diskussion.

Begruendung: Pilot-Akzeptanz waechst durch Transparenz, nicht durch versteckte Score-Modifikatoren.

### 3.4 Gate-Definition bleibt clientseitig in `landingScoring.ts`

Approach-Stability-Gate (z.B. 1000ft AGL stabilized) wird NICHT ins Backend verschoben. Der Client kennt die Daten und rendert sowohl Score als auch Chart — Single-Source-of-Truth fuer Gate-Boundary.

## 4. Feature-Pakete

### F1 — VFR/Manual-Mode: Start ohne ZFW

**Problem.** Im VFR-/Manual-Mode soll Start auch ohne ZFW funktionieren. **Realitaet (P1.1-A):** `flight_start_manual` in `lib.rs:5829-5837` rejected `None` ODER `<= 0.0` aktiv mit Error `invalid_zfw`:

```rust
let planned_zfw = match plan.planned_zfw_kg {
    Some(v) if v > 0.0 => v,
    _ => {
        return Err(UiError::new("invalid_zfw", "ZFW (Zero Fuel Weight) ist Pflicht…"));
    }
};
```

Zusaetzlich Asymmetrie: der Standard-`flight_start` (IFR-Pfad, ~lib.rs:5075+) hat KEINE ZFW-Pflichtpruefung. Pilot kann IFR ohne ZFW starten, Manual nicht.

**Loesung.**

1. **Backend Gate lockern (`lib.rs:5829-5837`):**
   ```rust
   let planned_zfw: Option<f32> = match plan.planned_zfw_kg {
       Some(v) if v > 0.0 => Some(v),
       Some(v) if v <= 0.0 => {
           // Defensiv: 0/negativ ist Eingabefehler, nicht "kein Plan"
           return Err(UiError::new("invalid_zfw_value",
               "ZFW muss > 0 sein wenn angegeben."));
       }
       _ => None, // VFR/Manual ohne Plan — erlaubt
   };
   ```
   `stats.planned_zfw_kg = planned_zfw;` (statt `Some(planned_zfw)`).

2. **Backend TOW/LDW-Berechnung (`lib.rs`, suchen nach `planned_takeoff_weight` / `planned_landing_weight`):** nur berechnen wenn `planned_zfw_kg` UND `planned_burn_kg` vorhanden:
   ```rust
   if let (Some(zfw), Some(burn)) = (stats.planned_zfw_kg, stats.planned_burn_kg) {
       stats.planned_takeoff_weight_kg = Some(zfw + initial_fuel_kg);
       stats.planned_landing_weight_kg = Some(zfw + initial_fuel_kg - burn);
   }
   // sonst: bleibt None, UI zeigt "nicht verfuegbar"
   ```

3. **Frontend (`ManualFlightModal.tsx:36-46`):** ZFW-Feld bekommt Hint "Optional. Ohne ZFW keine Loadsheet-/Mass-Wertung." Submit-Button bleibt aktiviert auch bei leerem ZFW-Feld.

4. **Scoring (Rust-Crate `landing-scoring`):** `sub_loadsheet()` skipt sauber:
   ```rust
   if record.planned_zfw_kg.is_none() {
       return SubScore::skipped("loadsheet", "no_planned_zfw");
   }
   ```

5. **UI (`LandingPanel.tsx`):** Skipped-Sub-Scores werden als "nicht bewertet" gerendert (grau, mit Tooltip warum). Master-Score gewichtet skipped raus (siehe 5.4).

6. **MassPanel (`MassPanel.tsx:32-68`):** wenn `planned_zfw_kg == null` zeigt "Loadsheet nicht verfuegbar (VFR/Manual ohne Plan)" statt leerer Tabelle.

**IFR-Asymmetrie:** v0.7.1 macht IFR-Pfad NICHT plotzlich strenger. IFR ohne ZFW war bisher erlaubt, bleibt erlaubt — Loadsheet-Sub-Score skipt dann genauso. Dokumentiert in den Phase-Transition-Asserts mit Kommentar "ZFW ist NIE Pflicht — Sub-Score-Skip statt Hard-Gate".

**Files Affected.**
- `client/src-tauri/src/lib.rs:5829-5837` — Gate von Hard-Reject auf Option<f32> umstellen
- `client/src-tauri/src/lib.rs` (TOW/LDW-Berechnung, suchen nach `planned_takeoff_weight`) — nur wenn beide vorhanden
- `client/src-tauri/crates/landing-scoring/` (NEU, P1.1-B Fix) — `sub_loadsheet` mit Skip-Logik
- `client/src/components/ManualFlightModal.tsx:36-46` — ZFW-Hint, Submit ohne ZFW erlaubt
- `client/src/components/MassPanel.tsx:32-68` — Render-Pfad fuer "kein ZFW vorhanden"
- `client/src/components/LandingPanel.tsx:39-108` — `LandingRecord` bekommt skipped-Sub-Scores Render
- `client/src/locales/{de,en,it}/common.json` — Strings

**Acceptance.**
- Backend: `flight_start_manual` mit `planned_zfw_kg: None` returnt Ok (heute: Err `invalid_zfw`)
- Backend: `flight_start_manual` mit `planned_zfw_kg: Some(0.0)` oder negativ returnt Err `invalid_zfw_value` (Eingabefehler-Schutz)
- Pilot kann Manual-Flug ohne ZFW starten — keine Fehlermeldung, kein blockierender Modal
- LandingPanel zeigt "Loadsheet: nicht bewertet" mit Tooltip "Im VFR-/Manual-Mode ohne Dispatch-Daten"

- IFR-Pfad bleibt unveraendert in seinem Verhalten

### F3 — Fuel-Score-Asymmetrie: Minderverbrauch nicht bestrafen

**Problem.** `subFuel()` benutzt `Math.abs(fuel_efficiency_pct)`. -5% = +5% im Score. Sparsame Piloten werden bestraft.

**Loesung.**

Asymmetrische Schwellen + Label-Aenderung von "Spritverbrauch" zu "OFP-Treue". Algorithmus in der Rust-Crate `landing-scoring` (P1.1-B), nicht mehr in `landingScoring.ts`:

```rust
let efficiency = record.fuel_efficiency_pct; // negative = weniger als geplant

// Mehrverbrauch (efficiency > 0) bewerten wie heute
if efficiency > 0.0 {
    return match efficiency {
        e if e < 2.0  => score(100, "within_plan"),
        e if e < 5.0  => score(80,  "slight_over"),
        e if e < 10.0 => score(55,  "over"),
        e if e < 20.0 => score(25,  "significant_over"),
        _             => score(5,   "gross_over"),
    };
}

// Minderverbrauch (efficiency <= 0) — leicht bonus, nie Strafe
let under = efficiency.abs();
if under < 5.0  { return score(100, "efficient"); }
if under < 15.0 { return score(95,  "very_efficient"); }
// Jenseits 15% Minderverbrauch: Hinweis auf moeglichen OFP-/Mess-/Routenfehler
score_with_warning(85, "suspiciously_efficient", "planned_burn_may_be_off")
```

**Label-Aenderung in i18n:**
- `landing.sub.fuel`: "Spritverbrauch" → "OFP-Treue" (DE), "Fuel" → "OFP Compliance" (EN)
- `landing.info.fuel`: neu schreiben — "Bewertet wird die Treue zum OFP-Plan. Mehrverbrauch zaehlt, Minderverbrauch wird nicht bestraft."

**Files Affected.**
- `client/src-tauri/crates/landing-scoring/src/sub_fuel.rs` (NEU) — Asymmetrie-Logik
- `client/src-tauri/crates/landing-scoring/tests/sub_fuel_replay.rs` (NEU) — Tabellen-Test
- `client/src/locales/{de,en,it}/common.json` — Labels

**Acceptance.**
- Flug mit -10% Verbrauch: Score 95, Label "Sehr effizient" — KEIN Penalty
- Flug mit +10% Verbrauch: Score 55, Label "Mehrverbrauch" wie bisher

## 11. Glossar

- **Gate (Stability-Gate):** Hoehenfenster ueber dem Anflug bei dem Stability bewertet wird. v0.7.1 = 1000 ft → 200 ft AGL.
- **OFP-Treue:** Treue zum SimBrief-OFP-Plan, nicht Sparsamkeit. Mehrverbrauch zaehlt, Minderverbrauch nicht.
- **Confidence:** Wie sicher die Touchdown-Forensik ist. High/Medium/Low/VeryLow. Anzeige-only in v0.7.1.
- **forensics_version:** Marker im PirepPayload. 2 = touchdown_v2 (v0.7.0+).
- **ux_version:** Marker im PirepPayload. 1 = v0.7.1 Score-Logik. 0/fehlt = pre-v0.7.1.
- **Skipped Sub-Score:** Sub-Score der nicht bewertet wurde (z.B. Loadsheet ohne ZFW). Master-Score gewichtet skipped raus.
- **Late-Premium:** Premium-X-Plane-Plugin-TD-Event das nach RREF/gear-force-Edge kommt (v0.7.0 Round-4 Fix, hier nur erwaehnt zum Verstaendnis).
