## 0. Warum dieses Dokument

v0.7.0 hat die Landerate-Berechnung strukturell saniert (4-Layer touchdown_v2, HARD GUARDS, Multi-TD, sim-spezifische Validation). Im laufenden Pilot-Feedback gibt es jetzt **9 zusammenhaengende UX-/Fairness-Punkte** gleicher Wurzel:

1. **VFR-Piloten werden vom Startflow blockiert oder bekommen unfaire Scores**, weil ZFW/Fuel-Plan implizit als IFR-OFP-Vollstaendigkeit angenommen werden
2. **Sparsame Piloten werden bestraft**, weil Fuel-Score absolute Abweichung vom Plan misst statt Mehrverbrauch
3. **Die Forensik-v2-Verbesserungen sind im UI unsichtbar** ‚Äî Pilot fragt weiter "warum diese Landerate?"
4. **Stability-Bewertung wirkt willkuerlich**, weil das bewertete Fenster im Chart nicht markiert ist und der Flare optisch in derselben Zone liegt
5. **Begriffe wie "Spritverbrauch" sind ambivalent** (Effizienz vs. Planabweichung) und Client/Web zeigen teils unterschiedliche Labels

**Befund:** alle 9 Punkte gehoeren in EIN Release-Paket ‚Äî sie loesen zusammen das gleiche uebergeordnete Problem ("Piloten verstehen und akzeptieren die Bewertung"). Einzeln aufgeschnuert wuerden sie sich gegenseitig widersprechen (z.B. Stability-v2-Werte anzeigen ohne Gate-Markierung verschlimmert die Konfusion).

**Kein Touchdown-Core-Refactor.** v0.7.0 bleibt unangetastet. v0.7.1 ist:
- 70% Frontend (LandingPanel, PhaseTimeline, Tooltips, i18n)
- 20% Scoring-Logik (Fuel-Score-Asymmetrie, Stability-Sub-Score Erweiterung)
- 10% Backend-Datenfluss (forensics_version + confidence durchreichen, Flare-Felder in PIREP-Payload)

## 3. Architektur-Entscheidungen

### 3.1 Score-Algorithmus wird nach Rust portiert (Single Source of Truth)

**P1.1-B Fix gegen v1.0:** v1.0 sagte "scoring bleibt clientseitig in TS, Backend befuellt sub_scores aus computeSubScores()". Das geht NICHT ‚Äî Rust kann die TS-Funktion nicht aufrufen, der PirepPayload wird in Rust gebaut.

**v1.1 Entscheidung:** Score-Algorithmus wird in einer **neuen Rust-Crate** `client/src-tauri/crates/landing-scoring/` portiert (1:1-Port von `landingScoring.ts`). Die Crate ist die einzige Stelle die Schwellen, Asymmetrie und Voting kennt.

Konsumenten:
- **Backend (`lib.rs`):** ruft `landing_scoring::compute_sub_scores(&record)` direkt ‚Äî schreibt Ergebnis in `PirepPayload.sub_scores` und in `landing_history.json`
- **Tauri-Frontend (Client UI):** liest **AUSSCHLIESSLICH** gespeicherte `sub_scores` aus dem `LandingRecord` (PIREP-Payload + landing_history). KEIN WASM-Bindings-Pfad in v0.7.1, KEINE Live-Berechnung. Damit ist garantiert dass alte und neue PIREPs konsistent bleiben (siehe ¬ß3.5)
- **aeroacars-live (webapp + monitor):** liest `sub_scores` direkt aus PirepPayload ‚Äî KEINE Re-Berechnung

**Folge:** `landingScoring.ts` wird durch eine duenne Re-Export-Datei ersetzt die nur noch fuer Test-Kompatibilitaet existiert (vitest greift weiter auf TS, ruft aber im Test-Setup die kompilierten Werte aus dem Goldenset ab). Tests in Rust gehen ueber `cargo test` in der Crate (Goldenset gegen 6 Replay-Fixtures wie touchdown_v2_replay.rs). KEIN zweiter Produktionspfad.

**WASM-Bindings explizit out-of-scope** (siehe ¬ß1 Was BEWUSST nicht in v0.7.1). Live-Score-Vorschau im Cockpit waere ein eigenes Feature fuer v0.7.2/v0.8.x.

**Migration (P2.4-A Korrektur ‚Äî kein WASM):**
1. Phase 0 erstellt Crate + portiert subFuel/subStability/subTouchdown/subBounces/subGForce/subRollout
2. Tests in Rust: golden test-vectors aus aktuellem TS-Verhalten generieren, Rust matched bit-identisch
3. Phase 1: Backend benutzt Crate zum Befuellen `sub_scores` in PirepPayload + landing_history.json
4. Phase 3: TS-Modul `landingScoring.ts` wird **entfernt** (wenn keine UI-Stelle es noch importiert) ODER zum reinen Test-Kompatibilitaets-Shim ohne Produktionspfad reduziert. **KEIN WASM-Wrapper** ‚Äî UI liest ausschliesslich gespeicherte `sub_scores` aus dem Record (siehe ¬ß3.5 Legacy-Schutz)

Begruendung gegen Alternative "Web/Monitor rechnet selbst":
- Drift-Risiko zu hoch (3 Implementierungen muessten synchron bleiben)
- Web hat nicht alle Roh-Felder (z.B. AGL-Buffer fuer Stability-Window)
- v0.7.0-Lehre: Single-Source-of-Truth ist nicht-verhandelbar

### 3.2 Score-Asymmetrie wird im Sub-Score-Algorithmus codiert, nicht via Override

`subFuel()` bekommt direkt eine asymmetrische Schwelle. KEIN nachgelagerter Override ("wenn negativ ‚Üí 100"). Begruendung: Override-Pattern war v0.6.x bei `vs_at_edge` die Wurzel des Bugs.

### 3.3 Confidence ist Anzeige-only, nicht Score-Bestandteil

**Korrektur gegen v1.4.** v1.1-v1.4 sagten "Gate = 1000 ft ‚Üí 200 ft AGL". Realitaet (verifiziert `lib.rs:3042-3049` + `APPROACH_FLARE_CUTOFF_MS = 3000` in `lib.rs:2883`): das Backend-Stability-Gate ist:

**Pseudocode (Crate):**
```rust
pub fn sub_fuel(record: &LandingRecord) -> SubScoreEntry {
    let Some(planned) = record.planned_burn_kg else {
        return SubScoreEntry::skipped("fuel", "landing.sub.fuel", "no_planned_burn");
    };
    let Some(actual) = record.actual_trip_burn_kg else {
        return SubScoreEntry::skipped("fuel", "landing.sub.fuel", "no_actual_burn");
    };
    let efficiency = (actual - planned) / planned * 100.0;
    let (score, band, rationale, tip, warning) = classify_fuel(efficiency);
    SubScoreEntry {
        key: "fuel".into(),
        score,
        points: score,
        band: band.into(),
        label_key: "landing.sub.fuel".into(),
        value: Some(format!("{:+.1}%", efficiency)),
        rationale_key: Some(rationale.into()),
        tip_key: Some(tip.into()),
        skipped: false,
        reason: None,
        warning,
    }
}
```

| Sub-Score | Gewicht |
|---|---|
| `landing_rate` | 3 |
| `g_force` | 3 |
| `bounces` | 2 |
| `stability` | 2 |
| `rollout` | 1 |
| `fuel` | 1 |

**Formel (Rust-Crate `landing_scoring::aggregate_master_score`):**
```rust
pub fn aggregate_master_score(subs: &[SubScore]) -> Option<u8> {
    let weights = sub_score_weights(); // siehe Tabelle oben
    let mut sum = 0.0;
    let mut wsum = 0.0;
    for s in subs {
        if s.skipped { continue; } // Skip aus Summe UND Gewichtssumme
        let w = weights.get(&s.key).copied().unwrap_or(1) as f32;
        sum += (s.score as f32) * w;
        wsum += w;
    }
    if wsum > 0.0 { Some((sum / wsum).round() as u8) } else { None }
}
```

**Konsequenzen:**
- VFR-Flug ohne Loadsheet/Fuel: nur 6 von 8 Sub-Scores tragen bei, gewichtet aus 3+3+2+2+1+1=12 statt 14 ‚Üí Master-Score wird aus den vorhandenen fair gemittelt, kein 0-Penalty fuer fehlende
- IFR-Flug mit allen Sub-Scores: identisches Ergebnis wie v0.7.0 (gleiche Gewichte, alle vorhanden)
- Neue `loadsheet`/`flare` Sub-Scores haben Gewicht 1 ‚Äî bewusste Entscheidung damit sie nicht v0.7.0-Master-Score-Werte stark verschieben fuer bestehende IFR-Fluege

## 11. Glossar

- **Gate (Stability-Gate):** Hoehenfenster ueber dem Anflug bei dem Stability bewertet wird. v0.7.1 = 1000 ft ‚Üí 200 ft AGL.
- **OFP-Treue:** Treue zum SimBrief-OFP-Plan, nicht Sparsamkeit. Mehrverbrauch zaehlt, Minderverbrauch nicht.
- **Confidence:** Wie sicher die Touchdown-Forensik ist. High/Medium/Low/VeryLow. Anzeige-only in v0.7.1.
- **forensics_version:** Marker im PirepPayload. 2 = touchdown_v2 (v0.7.0+).
- **ux_version:** Marker im PirepPayload. 1 = v0.7.1 Score-Logik. 0/fehlt = pre-v0.7.1.
- **Skipped Sub-Score:** Sub-Score der nicht bewertet wurde (z.B. Loadsheet ohne ZFW). Master-Score gewichtet skipped raus.
- **Late-Premium:** Premium-X-Plane-Plugin-TD-Event das nach RREF/gear-force-Edge kommt (v0.7.0 Round-4 Fix, hier nur erwaehnt zum Verstaendnis).
