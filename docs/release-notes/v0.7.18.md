## 🇩🇪 Deutsch

**v0.7.18 — Orphan-Flight-Cleanup-Cluster. Vier zusammenhängende Bugs adressiert: Verwaiste PIREPs lassen sich endlich aus AeroACARS heraus aufräumen, Off-Airport-Landungen werden korrekt erkannt, Sim-Position-Resets werfen keine Phantom-Sinkraten mehr, und der „Flug verwerfen"-Button verliert keine geflogene Strecke mehr.**

### Für alle Piloten

#### Verwaiste Flüge aufräumen (B-011)
- **Problem:** Wenn ein Flug nicht sauber beendet wurde (Sim-Crash, AeroACARS-Restart, Stromausfall etc.), blieb der PIREP auf phpVMS im Status `IN_PROGRESS` hängen. Beim nächsten Start verweigerte phpVMS einen neuen Flug — und es gab keinen UI-Weg den alten loszuwerden.
- **Fix:** Neuer Bereich **Einstellungen → Verwaiste Flüge**. Zeigt alle deine IN_PROGRESS-PIREPs mit Aircraft, Route und Alter. Pro Eintrag ein **Cancel-Button** mit destruktiver Bestätigung. Falls der PIREP serverseitig schon weg ist (404), gibt es einen **Fallback „Lokal vergessen"** der nur die lokale Spur löscht ohne phpVMS zu kontaktieren.

#### Off-Airport-Landungen werden jetzt erkannt (B-012)
- **Problem:** Wenn du an einem anderen Flughafen als dem geplanten Ziel aufgesetzt hast (Divert, Engine-Out, Sightseeing-Pause), zeigte AeroACARS trotzdem das geplante Destination-ICAO. Live-Map und PIREP-Aktivitätslog waren irreführend.
- **Fix:** Drei-Pfad-Auflösung mit Quelle-Markierung:
  - **Divert (gelb):** Pilot ist nah genug an einer anderen ICAO-Stelle → die wird übernommen.
  - **Nearest (orange):** Kein anderer Flughafen passend, aber nächster Airport > 10 NM von Destination → als Off-Airport markiert.
  - **No-Nearest (rot):** Mitten im Nirgendwo aufgesetzt (Off-Field / Strip ohne ICAO).
- Pilot-Client und Live-Map zeigen den Off-Airport-Banner farbig je nach Quelle, mit Distanz zur Destination und zum nächsten Flughafen.

#### Sim-Position-Resets erzeugen keine Phantom-Sinkraten mehr (B-013)
- **Problem:** Der 50-Hz-Touchdown-Sampler nahm auch Garbage-Samples auf — z. B. wenn MSFS für eine Sekunde `lat/lon = 0/0` lieferte (Mid-Atlantic Null-Island), du per F8 zu einer Aircraft-Variante teleportiert hast, oder das Game beim Loading kurz die Altitude auf 50.000 ft schickte. Folge: falsche Sinkrate, falscher Score, falscher Touchdown-Frame im Replay.
- **Fix:** dt-Guard mit drei Reset-Detektoren — Null-Island (`|lat|+|lon| < 0.1°`), Altitude-Sprung (> 10.000 ft pro Tick) oder Distanz-Sprung (> 10 NM pro Tick). Wenn die Bedingung **innerhalb von 2 Sekunden** auftaucht → Sample wird verworfen. Bei längeren Pausen (legitimes Game-Loop-Stall) wird die Heuristik bewusst nicht aktiv.

#### „Flug verwerfen" verliert keine fast-fertige Landung mehr (B-014)
- **Problem:** Ein einziger Klick auf „Flug verwerfen" am Ende einer kompletten Landung warf den PIREP auf CANCELLED — auch wenn der Pilot eigentlich nur den Cleanup-Klick gemacht hat. Realer Datenverlust für komplette Sektoren.
- **Fix:** Wenn der Flug in einer TD-Phase (Landing / TaxiIn / BlocksOn / Arrived) ist und gültigen Touchdown-Marker hat, kommt jetzt ein **3-Optionen-Dialog**:
  1. **Lieber filen versuchen** → Backend ruft `flight_end`, drei mögliche Ergebnisse:
     - **Filed:** PIREP direkt eingereicht. ✓
     - **Queued:** Transient-Netzfehler → PIREP wartet in der Background-Queue, wird automatisch nach Verbindungs-Recovery eingereicht. ⏳
     - **Cancelled:** Nur wenn du im zweiten Dialog explizit „Trotzdem verwerfen" wählst.
  2. **Andere Aktion** → führt zum zweiten Dialog „Trotzdem verwerfen?" mit destruktiver Bestätigung.
  3. **Trotzdem verwerfen** (nur via zweitem Dialog) → klassischer Cancel-Pfad.
- **UX-Sicherheit (QS-R2):** Wenn der „Filen versuchen"-Pfad mit einem harten Fehler scheitert (Validierung, Server-Hard-Fail), wird **nicht mehr automatisch gecancelt**. Stattdessen zeigt das UI einen zweiten Dialog mit der Begründung: „Filen ist fehlgeschlagen: <Grund>. Trotzdem verwerfen?". Du musst explizit zustimmen.

### Unter der Haube

#### phpVMS-Kompatibilität (QS-R1)
- `cancel_pirep` nutzt jetzt **DELETE mit PUT-405-Fallback** statt POST → kompatibel mit canonical phpVMS-Core (vorher hätte der Cancel-Flow gegen Standard-phpVMS still gefailed).
- HTTP-Status-Check akzeptiert jetzt **alle 2xx-Codes** (vorher nur 200). DELETE/PUT-Endpoints liefern oft 204 No Content — das wurde fälschlich als Fehler interpretiert.
- `delete_bid` Fallback-Cascade: bid_id-Versuch → bei `400 / 404 / 422` (mit 404 als Orphan-Cleanup-Fall) → flight_id-Body. Kompatibel mit beiden phpVMS-Bid-Schemas.
- `PirepSummary.flight_id` akzeptiert jetzt sowohl String- als auch Integer-IDs (manche VAs liefern numerische IDs ohne Quotes).

#### Webapp Off-Airport-Banner (B-012-Sync)
- Live-Map zeigt den selben Off-Airport-Banner wie der Pilot-Client (gelb/orange/rot je nach Quelle).
- Neue MQTT-Payload-Felder: `airport_source`, `airport_distance_to_destination_nm`, `airport_nearest_distance_nm`, `planned_arr_airport`.

## 🇬🇧 English

**v0.7.18 — Orphan-flight cleanup cluster. Four related bugs fixed: orphaned PIREPs can finally be cleaned up from inside AeroACARS, off-airport landings are properly recognized, sim position resets no longer cause phantom sink rates, and the "Discard flight" button no longer loses a flown sector.**

### For all pilots

#### Clean up orphan flights (B-011)
- **Problem:** If a flight didn't end cleanly (sim crash, AeroACARS restart, power outage, etc.), the PIREP on phpVMS stayed stuck in `IN_PROGRESS` status. The next flight start was refused by phpVMS — and there was no UI path to get rid of the old one.
- **Fix:** New section **Settings → Orphan flights**. Shows all your IN_PROGRESS PIREPs with aircraft, route and age. Each row has a **Cancel button** with destructive confirmation. If the PIREP is already gone server-side (404), there's a **fallback "Forget locally"** that just deletes the local trace without contacting phpVMS.

#### Off-airport landings now recognized (B-012)
- **Problem:** If you touched down at a different airport than the planned destination (divert, engine out, sightseeing stop), AeroACARS still showed the planned destination ICAO. Live map and PIREP activity log were misleading.
- **Fix:** Three-path resolution with source tagging:
  - **Divert (yellow):** Pilot is close enough to another ICAO location → it gets used.
  - **Nearest (orange):** No other matching airport, but nearest airport > 10 NM from destination → marked as off-airport.
  - **No-Nearest (red):** Touched down in the middle of nowhere (off-field / strip without ICAO).
- Pilot client and live map show the off-airport banner color-coded by source, with distance to destination and to nearest airport.

#### Sim position resets no longer cause phantom sink rates (B-013)
- **Problem:** The 50 Hz touchdown sampler also picked up garbage samples — e.g., when MSFS briefly returned `lat/lon = 0/0` for a second (mid-Atlantic null island), when you teleported to an aircraft variant via F8, or when the game briefly sent altitude 50,000 ft during loading. Result: wrong sink rate, wrong score, wrong touchdown frame in the replay.
- **Fix:** dt-guard with three reset detectors — null island (`|lat|+|lon| < 0.1°`), altitude jump (> 10,000 ft per tick), or distance jump (> 10 NM per tick). If the condition fires **within 2 seconds** → sample discarded. For longer pauses (legitimate game-loop stall) the heuristic deliberately stays off.

#### "Discard flight" no longer loses an almost-finished landing (B-014)
- **Problem:** A single click on "Discard flight" at the end of a complete landing put the PIREP on CANCELLED — even when the pilot just intended a cleanup click. Real data loss for entire sectors.
- **Fix:** If the flight is in a TD phase (Landing / TaxiIn / BlocksOn / Arrived) and has a valid touchdown marker, you now get a **3-option dialog**:
  1. **Try filing instead** → backend calls `flight_end`, three possible outcomes:
     - **Filed:** PIREP filed directly. ✓
     - **Queued:** Transient network error → PIREP waits in the background queue, will be filed automatically after connection recovery. ⏳
     - **Cancelled:** Only if you explicitly pick "Discard anyway" in the second dialog.
  2. **Other action** → leads to the second "Discard anyway?" dialog with destructive confirmation.
  3. **Discard anyway** (only via second dialog) → classic cancel path.
- **UX safety (QS-R2):** If the "try filing" path fails with a hard error (validation, server hard-fail), the system **no longer cancels automatically**. Instead, the UI shows a second dialog with the reason: "Filing failed: <reason>. Discard anyway?". You have to explicitly agree.

### Under the hood

#### phpVMS compatibility (QS-R1)
- `cancel_pirep` now uses **DELETE with PUT-405 fallback** instead of POST → compatible with canonical phpVMS core (previously the cancel flow would have silently failed against standard phpVMS).
- HTTP status check now accepts **all 2xx codes** (previously only 200). DELETE/PUT endpoints often return 204 No Content — this was wrongly interpreted as an error.
- `delete_bid` fallback cascade: bid_id attempt → on `400 / 404 / 422` (with 404 being the orphan cleanup case) → flight_id body. Compatible with both phpVMS bid schemas.
- `PirepSummary.flight_id` now accepts both string and integer IDs (some VAs return numeric IDs without quotes).

#### Webapp off-airport banner (B-012 sync)
- Live map shows the same off-airport banner as the pilot client (yellow/orange/red by source).
- New MQTT payload fields: `airport_source`, `airport_distance_to_destination_nm`, `airport_nearest_distance_nm`, `planned_arr_airport`.

### What pilots need to do

Nothing — the auto-updater rolls v0.7.18 out as usual. Orphan PIREPs in your phpVMS list? Just start with the new version, open **Settings → Orphan flights**, click Cancel.
