## 🇩🇪 Deutsch

**Sim-Recovery Release — laufende Flüge überleben jetzt Simulator-Crash, Pause, Neustart oder kurze Rechner-Unterbrechung sauber.**

### Warum

Real-Pilot-Incident am 2026-05-11: bei einem AUA-323-Flug LOWW→ESGG ist MSFS im Descent eingefroren. AeroACARS hat nach 30 s pausiert — der Pilot hat es aber erst nach der manuellen Landung am Boden in ESGG bemerkt. Resultat: zwei Sessions im History-Tab, falsche Flugzeit, kein durchgehender Position-Trace.

v0.7.15 packt diese Wurzel an drei Stellen an:

1. **Auto-Resume** — wenn der Simulator wieder Daten liefert, läuft die Aufzeichnung automatisch weiter. Kein „Flug fortsetzen"-Klick mehr nötig
2. **Pause-Akkumulator** — Pausezeit wird gezählt und von Flight-/Block-Time im PIREP abgezogen. 20 Minuten Sim-Crash zählen nicht mehr als 20 Minuten Flugzeit
3. **Server-Session-Identität** — der Recorder erkennt denselben Flug an der `pirep_id` wieder. Lange Positions-Lücken erzeugen keine zwei Sessions mehr

### Was Piloten merken

- **Nach Sim-Crash**: Sim läuft wieder → AeroACARS resumed von alleine, Activity-Log zeigt „▶ Flug automatisch fortgesetzt"
- **Bei MSFS Esc-Pause**: AeroACARS erkennt die Pause sofort (SimConnect-Event `Pause_EX1`), wartet auf Unpause, kein Klick nötig. Flight-Time im PIREP zieht die Pausezeit ab
- **Bei X-Plane Pause / Replay**: gleich — wird über `sim/time/paused` + `sim/time/is_in_replay` erkannt
- **Falsches Flugzeug nach Restart geladen**: Activity-Log zeigt eine Warnung („⚠ Aircraft-Mismatch: Sim meldet A320, Bid erwartet B738"). Resume läuft trotzdem — der Pilot kann den PIREP canceln wenn das wirklich der falsche Flug ist
- **Server-History**: lange Positions-Lücken (z.B. 23 Minuten Sim-Freeze) erzeugen keine zwei Sessions mehr. Eine `pirep_id` = ein Flug
- **Manueller Resume-Button bleibt** als Fallback erhalten

### Was VA-Owner einmalig macht

Nichts auf der Webapp-Admin-Seite. Der Server-Recorder wird per `deploy-recorder.sh` auf `live.kant.ovh` deployed (Server-Patches sind in `aeroacars-live` bereits gemerged). Ab diesem Moment greift der `pirep_id`-Join für alle aktiven Flüge.

### Heartbeat-Verhalten

Während einer Pause (Sim-Crash, MSFS-Pause, X-Plane-Pause) sendet AeroACARS weiterhin alle 30 s einen Heartbeat an phpVMS mit der letzten bekannten Position. Damit bleibt der PIREP **unbegrenzt am Leben**, solange die App läuft — egal wie lange die Pause dauert. Erst wenn die App geschlossen wird, beginnt das phpVMS-Cron-Fenster (~6 h) zu ticken.

### Was im PIREP korrekt ist

- **Flight-Time** = Wall-Clock − akkumulierte Pause-Zeit
- **Block-Time** = ebenfalls Pause-bereinigt
- **Distance** = nur tatsächlich geflogene Strecke (Reposition zählt nicht)
- **Pause-Segmente** = Audit-Log mit Start, Ende, Dauer, Reason, Drift NM (in `active_flight.json` persistiert)

## 🇬🇧 English

**Sim-Recovery release — active flights now survive simulator crash, pause, restart, or short computer interruption cleanly.**

### Why

Real pilot incident on 2026-05-11: on an AUA 323 flight LOWW→ESGG, MSFS froze during descent. AeroACARS paused after 30 s — but the pilot only noticed after manually landing on the ground at ESGG. Result: two sessions in the history tab, wrong flight time, no continuous position trace.

v0.7.15 attacks this at three points:

1. **Auto-resume** — when the simulator returns data, recording continues automatically. No more "resume flight" click needed
2. **Pause accumulator** — pause time is counted and subtracted from flight/block time in the PIREP. 20 minutes of sim crash no longer count as 20 minutes of flight time
3. **Server session identity** — the recorder identifies the same flight by its `pirep_id`. Long position gaps no longer create two sessions

### What pilots notice

- **After sim crash**: sim back up → AeroACARS resumes by itself, activity log shows "▶ Flight automatically continued"
- **MSFS Esc pause**: AeroACARS detects the pause immediately (SimConnect event `Pause_EX1`), waits for unpause, no click needed. Flight time in PIREP subtracts the pause duration
- **X-Plane pause / replay**: same — detected via `sim/time/paused` + `sim/time/is_in_replay`
- **Wrong aircraft loaded after restart**: activity log shows a warning ("⚠ Aircraft mismatch: sim reports A320, bid expects B738"). Resume runs anyway — pilot can cancel the PIREP if it really is the wrong flight
- **Server history**: long position gaps (e.g. 23 minutes of sim freeze) no longer create two sessions. One `pirep_id` = one flight
- **Manual resume button stays** as fallback

### What the VA owner does once

Nothing in the webapp admin. The server recorder is deployed via `deploy-recorder.sh` on `live.kant.ovh` (server patches are already merged in `aeroacars-live`). From that moment on, the `pirep_id` join takes effect for all active flights.

### Heartbeat behaviour

During a pause (sim crash, MSFS pause, X-Plane pause) AeroACARS continues to send a heartbeat to phpVMS every 30 s with the last known position. This keeps the PIREP **alive indefinitely**, as long as the app is running — regardless of pause duration. Only when the app is closed does the phpVMS cron window (~6 h) start ticking.

### What's correct in the PIREP

- **Flight time** = wall clock − accumulated pause time
- **Block time** = also pause-adjusted
- **Distance** = only actually flown distance (reposition does not count)
- **Pause segments** = audit log with start, end, duration, reason, drift NM (persisted in `active_flight.json`)

### Sim coverage

| Feature | MSFS | X-Plane |
|---|---|---|
| Auto-resume after disconnect | ✅ | ✅ |
| Pause accumulator | ✅ | ✅ |
| Heartbeat during pause | ✅ | ✅ |
| Native sim pause (Esc/P) | ✅ via `Pause_EX1` | ✅ via `sim/time/paused` |
| Replay mode | n/a | ✅ via `sim/time/is_in_replay` |
| Aircraft change warning | ✅ via SimConnect | ✅ X-Plane ≥12.1 + Web API on |

X-Plane <12.1 or with disabled Web API: aircraft change warning silently skipped (no false positives).

### Security / data

- Pause segments are stored only locally in `active_flight.json` — they don't go to phpVMS or the recorder
- `pirep_id` in the position MQTT payload is the normal PIREP ID phpVMS already knows
- Forward-only: pre-v0.7.15 `active_flight.json` still loads (all new fields have `serde(default)`)
- On the recorder side: 6 h cutoff for `pirep_id` match, terminal protection for filed PIREPs
