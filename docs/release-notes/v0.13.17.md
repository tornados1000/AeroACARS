# v0.13.17 — Triebwerks-Erkennung für Addons mit „totem" COMBUSTION-Flag (A340-600) + Airborne-Rescue

🇩🇪 **Deutsch**

## Was ist neu

- **Fix: Flüge mit der iniBuilds/Aerosoft A340-600 (und ähnlichen Addons)
  blieben die ganze Zeit in „Pushback" hängen → keine Landung, keine
  Landedaten, PIREP nicht auffindbar.** Ursache: AeroACARS leitet „Triebwerk
  läuft" aus der Standard-SimVar `GENERAL ENG COMBUSTION` ab — die liefert bei
  diesem Addon (MSFS 2024) **konstant 0**, obwohl die Triebwerke laufen. Da der
  Phasen-Ablauf Pushback → Taxi → Start auf „Triebwerke an" wartet, kam er nie
  in Gang: die Phase blieb den gesamten Flug auf „Pushback", der Touchdown
  wurde nie erfasst (V/S/Score leer), und der PIREP landete ohne Landedaten in
  der Review (Live-Befund IRM1140 / IBE778, Aerosoft A346, 2026-06-03).
- **Lösung 1 — N1-Fallback:** Wenn `GENERAL ENG COMBUSTION` für alle Triebwerke
  0 meldet, wertet AeroACARS jetzt zusätzlich **N1** (`TURB ENG N1`) aus — eine
  Standard-SimVar, die bei diesen Addons gültig bleibt. Ein Triebwerk mit N1
  über Leerlauf/Windmilling gilt als laufend; alle vier werden einzeln gezählt
  (A340 = 4 / 4). Greift **nur**, wenn der Combustion-Flag komplett ausfällt →
  keine Änderung für Flieger, bei denen er funktioniert.
- **Lösung 2 — Airborne-Rescue (allgemeines Sicherheitsnetz):** Hängt die Phase
  trotzdem in einer Boden-Phase fest, während das Flugzeug nachweislich in der
  Luft ist (nicht am Boden + über 500 ft AGL), springt sie automatisch auf
  „Climb". Damit kann **kein** Addon mehr einen kompletten Flug an einer
  festhängenden Boden-Phase scheitern lassen — selbst wenn alle Triebwerks-
  Signale versagen.

## Für VA-Owner

- **Keine Wire-Format-Änderung.** `aeroacars-live` muss **nicht** mit-deployed
  werden — reines Client-Update.
- Wirkt für **neue** Flüge. Bereits betroffene PIREPs (ohne Landedaten) bleiben
  wie sie sind und müssen wie gehabt manuell bewertet werden.

Dank an **Michael K** und **Patrick** für die Befunde (A340-600, OIIE→UWKD /
IBE778) — inklusive der Live-Ablesung von N1 im SimVar-Inspektor.

🇬🇧 **English**

## What's new

- **Fix: flights with the iniBuilds/Aerosoft A340-600 (and similar add-ons)
  stayed stuck in "Pushback" the whole flight → no landing, no landing data,
  PIREP not findable.** Root cause: AeroACARS derives "engine running" from the
  standard `GENERAL ENG COMBUSTION` SimVar — which on this add-on (MSFS 2024)
  reads **constant 0** even with the engines running. Since the Pushback → Taxi
  → Takeoff phase chain waits for "engines on", it never advanced: the phase
  stayed on "Pushback" for the entire flight, the touchdown was never captured
  (V/S/score empty), and the PIREP ended up in review with no landing data
  (live report IRM1140 / IBE778, Aerosoft A346, 2026-06-03).
- **Fix 1 — N1 fallback:** when `GENERAL ENG COMBUSTION` reads 0 for all
  engines, AeroACARS now also evaluates **N1** (`TURB ENG N1`) — a standard
  SimVar that stays valid on these add-ons. An engine with N1 above
  idle/windmilling counts as running; all four are counted individually (A340 =
  4 / 4). Only kicks in when the combustion flag fails completely → no change
  for aircraft where it works.
- **Fix 2 — airborne rescue (general safety net):** if the phase still hangs in
  a ground phase while the aircraft is demonstrably airborne (not on ground +
  above 500 ft AGL), it auto-advances to "Climb". This means **no** add-on can
  ever again lose a whole flight to a stuck ground phase — even if every engine
  signal fails.

## For VA owners

- **No wire-format change.** `aeroacars-live` does **not** need to be
  re-deployed — pure client update.
- Applies to **new** flights. Already-affected PIREPs (no landing data) stay as
  they are and need manual review as before.

Thanks to **Michael K** and **Patrick** for the reports (A340-600, OIIE→UWKD /
IBE778), including the live N1 read in the SimVar inspector.
