#### phpVMS, Pilot-Client, Webapp und Discord-Bot zeigen jetzt IMMER denselben V/S-Wert

## 🇬🇧 English

**v0.7.21 — Structural fix for the recurring "three different V/S values for the same touchdown" inconsistency. Structural, not point-fix.**

### What changed

#### phpVMS, pilot client, webapp, and Discord bot now ALWAYS show the same V/S value

- **Problem:** For some flights the pilot client showed "-194 fpm" (Engineering edge), the webapp live map "-194", the Discord bot "-194", but the phpVMS PIREP "-205 ft/min". Different sources, different truths — embarrassing.
- **Anchor case:** MS713 (HECA→OLBA, B738, X-Plane 12, 2026-05-13). VPS DB evidence:
  - `vs_at_edge_fpm: -194.057` (FAR 25.473 engineering standard)
  - `vs_smoothed_500ms_fpm: -204.7` (Volanta-equivalent 0.5s mean)
  - phpVMS PIREP payload: `landing_vs_fpm: -204` — wrong.
- **Structural root cause:** `FlightStats` has three different V/S fields (`landing_rate_fpm`, `landing_peak_vs_fpm`, `landing_analysis["vs_at_edge_fpm"]`). Before v0.7.21 the "right" value only came out when the code site explicitly called the helper `score_basis_vs_fpm()`. Whoever forgot got the streamer-tick snapshot. v0.7.17 B-015a was a sweep for this — but the FileBody line for phpVMS was missed.

#### Structural solution instead of spot fix

- **SSoT method:** `FlightStats::canonical_landing_rate_fpm()` is the single source of truth for pilot/PIREP/MQTT/phpVMS from v0.7.21 onwards. Cascade: `vs_at_edge_fpm` (engineering) → `landing_peak_vs_fpm` → `landing_rate_fpm`.
- **Six display paths migrated:** `FileBody.landing_rate` (= phpVMS column, **today's broken path**), `build_landing_record`, `flight_end_manual` fallback, `ActiveFlightInfo.landing_rate_fpm` (= active-flight panel), PIREP custom field `Landing Rate`, PIREP notes "Landing rate".
- **Seven consistency tests in the `canonical_landing_rate_fpm_tests` module**, hardened against the MS713 anchor. Any future display site that reads `stats.landing_rate_fpm` directly instead of going through the SSoT method breaks the test.
- **Shim compat:** the old freestanding `score_basis_vs_fpm(&stats)` stays as a 1-line wrapper delegating to the method — existing call sites stay green with no migration needed.

### What pilots need to do

Nothing — auto-updater rolls v0.7.21 out. From the next touchdown onwards, the value lines up across all views.

### What was NOT changed

- **Webapp/recorder/VPS:** no code change, no deploy. Patch is pilot-client only.
- **Scoring logic:** unchanged (was already correct).
- **Historical phpVMS PIREPs:** stay with the old values — only new flights from v0.7.21 onward show the right V/S in the phpVMS landing-rate column.
