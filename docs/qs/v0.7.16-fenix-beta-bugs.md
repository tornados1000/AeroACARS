### B-004 · ARRIVED-Phase stoppt den Streamer nicht — Stale-Position-Stream + nicht gefilte PIREPs

- **Status:** 🟥 OFFEN
- **Datum:** 2026-05-12
- **Tester:** Thomas K (export von flight-gsg-24.json, CFG 1641 EDDM→EDDL Session 703)
- **Build:** v0.7.15 Client (Tester-Pilot ist noch nicht auf v0.7.16/v0.7.17)

**Symptom:**
- Pilot landet EDDL 13:22 UTC (Touchdown captured)
- FSM geht Phase → ARRIVED ✅
- **Aber:** weitere 1:29h kommen Position-Updates rein, letzter Heartbeat 14:51 UTC
- Letzte Position: 44.486°N, 6.570°E (Suedfrankreich/Alpen, ~750 km vom Zielairport EDDL weg)
- Aircraft-State: FL318, 472 kt GS, Engines On, on_ground=false
- PIREP `QG1mAQon8V0an1jy` fehlt komplett in der pireps-Liste → wurde nie eingereicht

**Wahrscheinlicher Hergang:**
1. Pilot landet in EDDL
2. Hat aus irgendeinem Grund nicht auf „Flug beenden" geklickt (vergessen, App-Bug, oder Validation gescheitert)
3. Startet im Sim einen anderen Flug (Sim-Restart, neues Aircraft, irgendwas weit weg von EDDL)
4. Streamer in AeroACARS laeuft weiter — schickt Positionen mit der alten `pirep_id` QG1mAQon8V0an1jy
5. Server-Recorder akzeptiert sie via `pirep_id`-Join (v0.7.15 F4 feature) → reportet eine 1:29h-Position-Reihe mit Aircraft 750 km vom Arr-Airport entfernt

**Code-Wurzel:**
- `flight.stop.store(true)` nur in `flight_end` / `flight_end_manual` / `flight_cancel`
- FSM Phase ARRIVED stoppt den Streamer **nicht** automatisch
- Pilot muss explizit auf „Flug beenden" klicken — wenn er das vergisst, laeuft alles weiter

**Fix-Optionen:**
1. **Auto-Stop bei ARRIVED + lange Stille:** Wenn Phase ARRIVED + `on_ground=true && groundspeed<1 && engines=0` fuer ≥5 Minuten → Streamer automatisch stoppen mit Activity-Log-Warning „Flug nicht manuell beendet — Position-Stream stoppt nach 5 min Standzeit"
2. **Sanity-Check „Phase ARRIVED, aber Aircraft fliegt":** Wenn `phase == ARRIVED && !on_ground && groundspeed > 100` → Activity-Log-Error + Streamer-Stopp („Aircraft fliegt nach Landung — alter Flug wird beendet, neuen Flug starten")
3. **Pilot-Reminder:** Wenn Phase ARRIVED erreicht ist und nach 2 min noch keine „Flug beenden"-Aktion kam → prominenter UI-Banner

**Wo:**
- `client/src-tauri/src/lib.rs` FSM-Tick-Loop und/oder Position-Streamer-Loop
- Auch Server-Side im Recorder: Position-Stream nach `phase_changed→ARRIVED` mit `pirep_id`-Join sollte aufhoeren zu akzeptieren

**Priorität:** **hoch** — verschmutzt PIREP-/Recorder-Daten (Aircraft springt von Zielairport nach Suedfrankreich im Position-Track), und Pilot verliert PIREP-Filing

**Ziel-Release:** v0.7.17 (Sammel-Patch)

### B-010 · FSM-Phasen nicht GA/Cessna-tauglich (Spurious Cruise→Descent/Approach)

- **Status:** 🟥 OFFEN
- **Datum:** 2026-05-12
- **Quelle:** GSG 105 LDDU→LDSP, Aircraft Total-Weight 1.000 kg (Cessna-Klasse), X-Plane 12

**Symptom (aus flight-APMVJmeQJWmP3MXk JSONL):**

```
17:26:53  Climb->Descent    AGL 2692 ft  (aircraft climbing!)
17:27:13  Descent->Approach AGL 2802 ft  (AGL STIEG zwischen 2 Phasen!)
17:32:20  Approach->Climb   AGL 1877 ft  (Go-Around)
17:49:35  Climb->Descent    AGL 3575 ft
17:49:41  Descent->Approach AGL 3540 ft  (6 Sekunden in Descent!)
```

AGL ist zwischen Climb->Descent und Descent->Approach GESTIEGEN — physisch unmoeglich fuer einen echten Descent. FSM hat falsch klassifiziert.

**Wurzel** in `client/src-tauri/src/lib.rs:13223`:

```rust
let low_altitude_descent = vs < -100 && agl_ft < 3000.0 && lost_from_peak > 500.0;
```

Schwellen sind fuer **Airline-Operations** kalibriert:
- AGL < 3000 ft: bei Cessna im Pattern normal (Pattern Altitude ~1500 ft AGL!)
- `lost_from_peak > 500`: bei kleinen VFR-Mustern (Steigflug auf 3000, kurzer Dip auf 2500 fuer Sightseeing) feuert das fuer einen Phantom-Descent

→ FSM springt zwischen Climb/Descent/Approach hin und her bei normalen GA-Mustern (Pattern-Work, Sightseeing-Tour, kurze Hopser).

**Fix-Optionen:**

1. **Aircraft-Type-aware FSM:** wenn Total-Weight < 5000 kg (Light-GA) → laengere Dwell-Times + grosszuegigere VS-Schwellen
2. **Pattern-Detection:** wenn AGL nie > 5000 ft + sustained < 3000 → spezielle Pattern-FSM die zwischen Climb/Approach nicht so oszilliert
3. **Hysterese:** Phase-Wechsel braucht 30-60 s sustained Bedingung, nicht single Tick
4. **lost_from_peak hoeher:** > 1000 ft statt 500 ft, damit kleine Dips nicht triggern

**Priorität:** **hoch** — alle GA-Piloten in der VA sind potenziell betroffen, FSM-Phasen feedern Activity-Log + PIREP-Custom-Fields + Approach-Stability-Score

**Ziel-Release:** v0.7.17 (Sammel-Patch)

**Update 2026-05-12 (12) — KEINE Sample-Spikes, ECHTE Sim-Physik:**

Analyse der `touchdown_window` Samples zeigt:
```
Frame    VS     G      OnGround   Gear-Force-N
[126]  -116   0.98    false      0          (vor Aufsetzen)
[127]  -103   1.75    TRUE       (edge)
[128]   -46   2.30    true       13.682     ← Strut-Compression-Peak
[129]   +18   1.61    true       (Rebound)
[130]   +18   1.61    true
[131]   +47   1.05    true       (entspannt)
```

Die 2.30 G sind **echte Sim-Physik** — Strut-Compression bei einer 1.000-kg-Cessna:
- F = 13.682 N
- a = F/m = 13.68 m/s² = 1.39 G zusätzlich zur Erdanziehung
- Plus 1 G = 2.39 G total — matched genau

→ **X-Plane simuliert die Cessna-Strut zu hart**: für eine -116 fpm Landung würde ein echter Strut-Sensor 1.3-1.5 G zeigen, nicht 2.3. Aber das ist Sim-Engine-Verhalten, nicht AeroACARS-Bug.

**Echte Wurzel:** unsere Klassifikations-Logik `worse_of(by_vs, by_g)` ist falsch fuer den Pilot-Wahrnehmungs-Bezug:
- Pilot fuehlt V/S (= das was er beim Aufsetzen merkt)
- Pilot fuehlt NICHT die Strut-Compression-G (zu kurz, zu hoch-frequent)
- Aktuell: VS sagt Smooth (100), G sagt Severe (0), `worse_of` → Severe
- Frustration: „Mein -116 fpm Butter wird als Severe gescort"

**Korrekte Logik fuer v0.7.17:**

V/S fuehrt, G ist Modifikator-nach-oben (kann Severe-Impact verstaerken), aber kann nie alleine zu Severe fuehren bei smoother V/S.

Vorschlag:
```rust
let cat = match (by_vs, by_g) {
    (Smooth, Smooth|Acceptable|Firm) => Smooth,
    (Smooth, Hard|Severe) => Acceptable,  // Strut-Anomalie, aber Pilot war gut
    (Acceptable, Hard|Severe) => Firm,
    (Firm, Severe) => Hard,
    // Hard/Severe VS dominiert
    (vs_cat, _) if vs_cat in (Hard, Severe) => max(by_vs, by_g),
    _ => by_vs,  // sonst V/S fuehrt
};
```

**Damit:** Butter-Landung kriegt Smooth, auch wenn die Sim-Strut spikt. Echte Hard-Landings (Vs < -600 fpm) werden weiter korrekt klassifiziert.

**Wo:** `client/src-tauri/crates/landing-scoring/src/lib.rs:332-360` (classify-Funktion)

**Priorität:** **kritisch** — direkt sichtbarer Fairness-Issue

**Update 2026-05-12 (11) — X-Plane hat denselben Bug, nicht nur MSFS-Total-G:**

Tester-Befund GSG 105: Touchdown VS -116 fpm (butterweich), aber `peak_g_post_500ms = 2.30 G`. Sim = X-Plane 12, wir lesen bereits `gforce_normal` (vertikale Komponente). **Trotzdem 2.30 G in einer der 324 post-edge-Samples** — physikalisch unmoeglich fuer Cessna mit 1.000 kg bei -116 fpm Impact.

**Damit zwei separate G-Force-Issues:**

a) **MSFS liest Total-G statt Vertical-G** (urspruengliche B-009)
b) **G-Force Sample-Spikes werden als Peak gewertet** (NEU — sim-agnostisch, X-Plane + MSFS betroffen)

**Wurzel (b):**

Sample-Stream hat numerische Spikes (Strut-Compression-Spring-Modell, Frame-Stuttering, etc.). Wir nehmen den `max()` ueber 800ms-Fenster → einzelner 2.30G-Sample bestimmt Score. Klassifikator: `worse_of(by_vs, by_g)` → VS sagt Smooth, G sagt Severe → **Severe wins, obwohl Landung butterweich war**.

**Fix (b):**

Statt `max()` einen robusteren Statistik-Wert:
- **Median** der post-edge G-Samples
- ODER **95th Percentile** (verschluckt 5% Spikes)
- ODER **rolling avg** ueber 100ms-Subfenster, davon dann max

**Priorität:** **kritisch** — direkt sichtbar fuer alle Piloten, jeder Butter-Touchdown wird unfair als Severe gescort

**Ziel-Release:** v0.7.17 (Sammel-Patch). **MUSS** in der Crit-Liste sein

- **Status:** 🟥 OFFEN
- **Datum:** 2026-05-12
- **Quelle:** Mehrere Pilots-Reports „G-Kraft beim Landen zu hoch"

**Problem:**

Inkonsistenz zwischen den Sim-Adaptern:

| Sim | Was AeroACARS liest | Was es ist |
|---|---|---|
| MSFS | `G FORCE` ([telemetry.rs:74](client/src-tauri/crates/sim-msfs/src/adapter/telemetry.rs#L74)) | **Gesamt-G** (alle Achsen) |

- ❌ Laengs-G aus dem Flare (Pitch-Up zieht)
- ❌ Quer-G aus Crosswind-Korrektur (Rudder/Aileron)
- ❌ Strut-Rebound

- Normal: 1.15-1.30 G
- Firm: 1.30-1.50 G
- Hard (Maintenance-Check): ≥ 1.50 G
- Severe: ≥ 2.00 G

Unsere Schwellen (`T_G_SMOOTH=1.20`) sind richtig fuer vertikale G, aber wir messen Total-G → systematische Ueberbewertung.

1. Im MSFS-Adapter zusaetzlich `ACCELERATION BODY Y` SimVar lesen (ft/s²)
2. Vertikale G berechnen: `g_vertical = ACCEL_BODY_Y / 32.174 + 1.0`
3. Touchdown-Sampler + Score nutzen vertikale G (analog X-Plane)
4. `G FORCE` als „total_g" beibehalten — fuer Debug-Vergleich im Pilot-Client
5. Beide im Debug-Panel nebeneinander zeigen

**Wo:**
- `client/src-tauri/crates/sim-msfs/src/adapter/telemetry.rs:74` — SimVar dazu

### B-007 · phpVMS-Account-Sperre/Inaktivierung wird stumm + ohne Hinweis verarbeitet

- **Status:** 🟥 OFFEN
- **Datum:** 2026-05-12
- **Quelle:** User-Frage „was passiert mit dem User im AeroACARS wenn ein User im phpVMS gesperrt oder inaktiv ist"

**Aktuelles Verhalten:**

phpVMS antwortet bei gesperrten/inaktiven Usern typischerweise mit HTTP 401 oder 403. AeroACARS reagiert an drei Stellen, keine davon sauber:

1. **App-Start (Token-Check), [lib.rs:4310-4313](client/src-tauri/src/lib.rs#L4310):**
   - API-Key wird stumm geloescht, Pilot landet auf Login-Form
   - Kein Hinweis warum → Pilot denkt „komisch, hat mich ausgeloggt"

2. **Aktiver Flug (Position-Posts), [lib.rs:8637-8656](client/src-tauri/src/lib.rs#L8637):**
   - Batch zurueck in Outbox, retry endlos (~30 s Takt)
   - Connection-State → `FAILING`
   - Pilot sieht nur „Failing"-Indikator, nicht differenziert von „Netzwerk weg"
   - phpVMS-Server wird mit retries gespammt — fuer immer

3. **PIREP-Filing, [lib.rs:7461](client/src-tauri/src/lib.rs#L7461):**
   - `is_transient_pirep_error` sagt 401/403 = nicht transient → kein Filing-Retry
   - **Aber:** `pending_pireps`-Background-Worker (Line 7491-...) retried trotzdem alle 60 s, max 50× → spammt Server weiter

**Fix-Optionen:**

1. **Unterscheidung im Position-Post-Worker:** bei 401/403 NICHT in Outbox requeuen, sondern Streamer pausieren + Activity-Log-Error („Account gesperrt — VA-Admin kontaktieren") + Connection-State auf neuer `BLOCKED`-Wert
2. **Login-Form-Hinweis:** bei Token-Check 401 → klares Banner „Dein Account wurde auf {{vaname}} gesperrt oder ist inaktiv. Bitte VA-Admin kontaktieren bevor du neu einloggst"
3. **pending_pireps-Worker:** bei 401/403 nicht retry-en, Datei mit Marker `"blocked_account"` versehen, Pilot sieht im UI „PIREP wartet — Account gesperrt"
4. **Discord-Hint:** wenn Recorder serverseitig 401 von einem aktiven Pilot bekommt, gezielt VA-Admin notify (separate Spec)

**Wo:**
- `client/src-tauri/src/lib.rs` Login-Path + Position-Worker + pending_pireps-Worker
- `client/src-tauri/crates/api-client/src/lib.rs` — evtl. neue `BlockedAccount` Variante (= 401 mit phpVMS-spezifischem Body) abgrenzen vs. echte expired-token-401

**Priorität:** mittel-hoch — passiert relativ selten (Sperre durch VA-Admin), aber wenn dann sehr verwirrend fuer den Piloten und verspamt den Server

**Ziel-Release:** v0.7.17 (Sammel-Patch)

### B-006 · Auto-File-Failure ist UNSICHTBAR — Pilot denkt „auto-filed", aber nichts ist passiert

- **Status:** 🟥 OFFEN
- **Datum:** 2026-05-12
- **Quelle:** User-Frage zu B-004 — „Auto-File ist doch default ON, hat das nicht gefeuert?"
- **Tester:** Thomas K (Tracker-Analyse CFG 1641)

**Code-Wurzel:** [CockpitView.tsx:88-104](client/src/components/CockpitView.tsx#L88-104)

```typescript
void (async () => {
  try {
    await invoke("flight_end");
    setActiveFlight(null);
  } catch {
    // Validation failure (e.g. distance to airport > MAX, fuel
    // missing) — leave activeFlight alone so the manual "End"
    // button still works and surfaces the file-or-cancel dialog.
    // Don't reset the ref: we don't want a retry loop, the pilot
    // can hit the button manually.
  }
})();
```

**Problem:** Der `catch {}` schluckt jeden Fehler stumm. Wenn `flight_end` mit `flight_validation_failed` o.ae. scheitert:
- KEIN Activity-Log-Eintrag
- KEIN UI-Banner
- KEIN Toast
- `autoFiledRef.current` bleibt gesetzt → kein Retry

→ Pilot weiss nicht, dass Auto-File versucht wurde, dass es scheiterte und dass er JETZT manuell „Flug beenden" druecken muss. Pilot denkt der PIREP ist eingereicht und macht weiter.

**Wahrscheinlich-Folge bei CFG 1641:**
1. Landung in EDDL, Phase → BlocksOn → ARRIVED
2. Auto-File feuerte
3. `flight_end` failed (z.B. `not_at_arrival` weil Pilot inzwischen vom Gate weggerollt war, oder Validation-Field fehlte)
4. Silent catch → Pilot dachte alles ok, fuhr weiter
5. Streamer lief weiter mit alter PIREP-ID (B-004)

| Altimeter-Setting | STD 1013 | lokales QNH |

A) **Fenix hat eigene Altimeter-Simulation** — divergiert bei extremen ISA-Abweichungen vom MSFS-Wettersystem-Pressure-Modell. **Update 2026-05-12 (10) — Webrecherche bestaetigt: bekanntes MSFS-Verhalten, nicht Fenix-spezifisch:**

- **PLANE ALTITUDE ist usually HIGHER als INDICATED ALTITUDE** in normaler Atmosphaere, und die Differenz waechst mit der Hoehe

- **swift-project hatte das gleiche Problem** und hat eine separate Korrektur dafuer gebaut (Issue #169)

**Saubere Fix-Strategie fuer v0.7.17:**

3 SimVars parallel lesen + kontextabhaengig nutzen:

| SimVar | Wofuer |
|---|---|
| `INDICATED ALTITUDE` | Pilot-/Live-Map-Anzeige (matched Cockpit-PFD) |
| `PRESSURE ALTITUDE` | PIREP-Cruise-Level-Eintrag + VATSIM-Kompatibilitaet (= Mode-C) |
| `PLANE ALTITUDE` (heutiges) | AGL-Berechnung + FSM-Phase-Triggers (geometric MSL ist da besser) |

SimSnapshot wird um zwei neue Felder erweitert (`altitude_indicated_ft`, `altitude_pressure_ft`); bestehende `altitude_msl_ft` bleibt fuer interne Berechnungen.

**Companion-Fix aeroacars-live Webapp:** Live-Map muss das neue Feld konsumieren (Tooltip „Indicated 33.000 ft / Pressure FL330").

### N-001 · SimBrief-Refresh wird beim Flug-Start verworfen — alte OFP-Daten zurueck

- **Status:** 🟥 OFFEN
- **Datum:** 2026-05-12
- **Build:** v0.7.16
- **Tester:** Thomas K (Owner)
- **Sim:** beliebig (nicht sim-spezifisch — OFP-Refresh-Flow)
- **Phase:** zwischen „Flug erstellt" und „Flug gestartet"

**Repro-Schritte:**
1. Flug in AeroACARS erstellen (aus Bid)
2. In SimBrief Werte aendern (z. B. Fuel, Route, Payload …) und neu generieren
3. In AeroACARS auf **„Aktualisieren"** druecken — UI zeigt korrekt die NEUEN SB-Werte
4. **Flug starten**
5. → AeroACARS zeigt wieder die **ALTEN** SB-Werte

**Workaround (aktuell noetig):**
- Zurueck auf „Meine Fluege"
- Bid nochmal oeffnen
- Nochmal „Aktualisieren" druecken
- *Dann* Flug starten

**Warum das nervt:**
- Der Refresh-Klick *vor* dem Start sieht erfolgreich aus (UI zeigt neue Werte) — der Pilot denkt der Refresh ist wirksam
- Beim Start wird er stillschweigend ueberschrieben — kein Hinweis, kein Warning
- Workaround ist zwei Klicks zu viel, und nicht intuitiv ableitbar aus dem UI-Verhalten

**Vermutete Ursache (zu verifizieren):**
- `flight_start` zieht den OFP nochmal frisch vom Server (`bid_id`-Lookup) statt den lokal refreshed In-Memory-State zu verwenden
- Der „Aktualisieren"-Button im Briefing-View updated nur die Anzeige, nicht den Server-side persisted OFP, der dann von `flight_start` re-fetched wird
- Alternativ: SimBrief-direct-Refresh schreibt in Bid-State zurueck (per `flight_refresh_simbrief`), aber `flight_start` liest den OFP-Pointer aus einem anderen Cache

**Mögliche Fixes:**
- `flight_start` soll den lokal refreshed OFP-State respektieren (kein Re-Fetch wenn frisch refreshed)
- Oder: „Aktualisieren"-Button ruft eine Variante auf, die den Server-side OFP wirklich aktualisiert, sodass `flight_start` immer den aktuellen Stand zieht
- Oder UI-Hinweis: „Hinweis: bitte zurueck zu Meine Fluege und nochmal aktualisieren bevor du startest"
  → letzteres ist Workaround-Cosmetic, nicht der eigentliche Fix

**Priorität:** **mittel-hoch** — passiert bei jedem Flug, der vor Start nochmal in SB geaendert wird (Fuel-Reload, Route-Change, neuer Payload). Real-Life-Workflow, kein Edge-Case.
